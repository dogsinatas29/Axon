# 🚀 DooM for AntigravitY - Final Blueprint v0.1

---

## 1. Project Identity

**Concept**
Classic Doom Resources + Quake Physics + Zero-G Mechanics

**Platform**
Linux / WSL

**Constraints**
- Python 3.8+ (Standard Library Only)
- No external dependencies
- ASCII Rendering (100x40)
- Keyboard Only

---

## 2. Core Architecture Principles

### 2.1 Core Philosophy
Stable Core + Disposable Experiments

- Core: ECS, Renderer → Immutable
- Extensions: Ghosts / AI → Disposable

---

### 2.2 Layer Model

[ Core ] ← (Event / Query) ← [ Extensions ]

- Core never depends on external
- External cannot mutate Core directly

---

### 2.3 Dependency Rules

**Core (Strict)**
- Pure Python only
- No external libraries

**Extensions (Loose)**
- Optional dependencies allowed
- Must be removable

---

## 3. Performance Model

### Core Principle
All computation must be **bounded**

---

### 3.1 Frame Constraints

- Target: 30~60 FPS
- Budget: 16ms ~ 33ms

---

### 3.2 Allowed Work

- Bounded loops
- Fixed-size iteration
- Cached computation

---

### 3.3 Forbidden Work

- Unbounded loops
- Growing data structures
- Recalculation without state change

---

### 3.4 System Budget

Each system has a time budget:

- render: 8ms
- physics: 4ms
- ai: 2ms

---

### 3.5 Degradation Strategy

When budget exceeded:
1. Skip system
2. Reduce update rate
3. Reduce precision

---

## 4. ECS Architecture

### Components (Data)

- Transform: position
- Motion: velocity / acceleration
- Body: collision
- Stats: hp / ammo
- PhysicsMode: NORMAL / ZERO_G / INVERTED
- Render: sprite / texture

---

### Systems (Logic)

- InputSystem
- PhysicsSystem
- GravitySystem
- CombatSystem
- RenderSystem

---

## 5. Rendering Pipeline

- Raycasting (DDA)
- ASCII shading
- Double buffering
- Z-shearing (no real pitch)

---

## 6. Engine Constraints

### Memory
- No growing allocation in frame loop
- Use reuse / pooling

### CPU
- Compute once, reuse
- Avoid repeated math

### Loop
- Must be bounded
