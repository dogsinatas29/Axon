## Core CoT (추론 과정 요약)

### 문제 A: GOVERNANCE_STALL (승인 후에도 무한 대기)
- 원인: `bootstrap_from_spec`에서 `approval_file` 경로가 `sandbox_root`에 매핑되는데,
  이전 PID(구버전 바이너리)의 루프가 `TEST4_GTK/TEST4_GTK/.axon_approval_pending`을 감시 중이었음.
  반면 서버의 SEAL 핸들러(DFS)는 올바른 `TEST4_GTK/.axon_approval_pending`에 `approved: true`를 씀.
  루프는 자신의 경로 파일을 읽지 못하므로 영구 STALL 상태에 빠짐.
- 추가 원인: 승인 성공 후 `std::fs::remove_file`로 파일을 삭제하는데, 삭제 후 루프가
  재진입하면 파일이 없으므로 `else` 브랜치(GOVERNANCE_STALL)로 빠짐.
- 수정: `approved: true` 감지 시 파일 삭제 제거 — break만 수행.

### 문제 B: SKELETON_FAIL 무한 루프 (Max retries reached)
- 원인: `generate_ir_with_context`에서 `qwen2.5:7b`가 spec 7833 바이트를
  4000 바이트로 강제 절삭(Truncating). LLM이 단 3개 컴포넌트만 생성.
  Validator가 6개 Core 컴포넌트 중 누락된 3개를 오류로 처리하며 무한 retry.
- 수정: Constraint-driven seed injection.
  Skeleton Validation 이후, `immutable_constraints.json`의 Core 컴포넌트 중
  IR에 없는 것을 Rust가 결정론적으로 삽입 (existence = deterministic).
  LLM은 semantic field enrichment만 담당 (semantics = probabilistic).

### 아키텍처 방향 (Ontology-first IR synthesis)
- 현재 패치: post-LLM seed (LLM 이후 누락 보완)
- 이상적 구조: pre-LLM seed (Constraints → Seed Skeleton → LLM fills fields → Merge)
  - Constraints > IR Schema > LLM output > spec prose (source of truth 우선순위)
  - 7B 모델에서도 안정적: discovery/ontology 약한 소형 모델이 기존 구조 보강/설명은 잘함
  - Validator → self-healing compiler로 진화 가능
