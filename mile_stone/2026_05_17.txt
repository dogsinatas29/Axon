use crate::schema::Language;
use super::ontology::{get_ontology, SemanticOntology};
use super::spec::SemanticSpec;
use super::violation::SemanticViolation;
pub struct SpecSemanticValidator;
impl SpecSemanticValidator {
    pub fn validate(spec: &SemanticSpec) -> Result<(), SemanticViolation> {
        let ontology = get_ontology(spec.language);
        for term in &spec.extracted_terms {
            let term_lower = term.to_lowercase();
            for forbidden in ontology.forbidden_vocabulary() {
                if term_lower.contains(&forbidden.to_lowercase()) {
                    return Err(SemanticViolation::ForbiddenVocabulary {
                        term: term.clone(),
                        language: spec.language,
                    });
                }
            }
        }
        for sys in &spec.build_systems {
            let allowed = ontology.allowed_build_systems();
            if !allowed.is_empty() && !allowed.iter().any(|a| sys.to_lowercase().contains(&a.to_lowercase())) {
                return Err(SemanticViolation::InvalidBuildSystem {
                    system: sys.clone(),
                    language: spec.language,
                });
            }
        }
        for task in &spec.task_vocabulary {
            if let Some(allowed) = ontology.allowed_task_kinds() {
                let task_lower = task.to_lowercase();
                let valid = allowed.iter().any(|tk| {
                    match tk {
                        axon_core::LanguageTaskKind::C(ct) => {
                            format!("{:?}", ct).to_lowercase().contains(&task_lower)
                                || task_lower.contains(&format!("{:?}", ct).to_lowercase())
                        }
                        axon_core::LanguageTaskKind::Rust(rt) => {
                            format!("{:?}", rt).to_lowercase().contains(&task_lower)
                                || task_lower.contains(&format!("{:?}", rt).to_lowercase())
                        }
                        _ => false,
                    }
                });
                if !valid {
                    return Err(SemanticViolation::InvalidTaskVocabulary {
                        task: task.clone(),
                        language: spec.language,
                    });
                }
            }
        }
        Ok(())
    }
}

1. Vocabulary Token Normalization Layer

현재:

"HeaderDecl"

↓

contains("header")

목표:

SemanticToken::HeaderDecl

즉 문자열 비교를 없애야 한다.

예:

pub enum SemanticToken {
    Header,
    Source,
    Module,
    Crate,
    Package,
    Integrator,
}

그리고 ontology는:

fn allowed_tokens(&self) -> &[SemanticToken]

기반으로 움직여야 한다.

2. Semantic Extractor 강화

현재:

JSON 문자열 스캔

수준이다.

목표는:

semantic intent extraction

이다.

즉:

현재	목표
substring scan	AST-like semantic extraction
raw vocabulary	normalized semantic intent
free text	canonical semantic tokens

예:

"create a rust module"

↓

SemanticToken::Module
3. SemanticViolation Hierarchy 확장

현재는:

ForbiddenVocabulary
InvalidBuildSystem

정도다.

하지만 실제론 더 필요하다.

예:

CrossLanguageContamination
IllegalTopology
ForbiddenArtifactIntent
InvalidEntrySemantic

이런 계층이 있어야 repair loop 품질이 올라간다.

4. Repair Loop 연동

지금은:

reject only

이다.

하지만 최종 목표는:

semantic self-repair

다.

예:

입력
Rust + HeaderDecl

↓

violation
ForbiddenVocabulary(HeaderDecl)

↓

repair hint
"Rust modules should use ModuleDecl instead of HeaderDecl"

↓

regenerated spec

이 흐름이 필요하다.

이건 반드시 필요하다.

현재 테스트는 unit 수준이다.

orphan headers	Topology failure

이걸 한 번에 고치면 시스템이 불안정해진다.

우선순위는 이렇게 가야 한다.

P0 — Artifact Firewall 먼저

이건 즉시 막아야 한다.

---CODE END===

같은 건 semantic 문제가 아니라 “emission contamination”이다.

현재 구조:

LLM output
   ↓
save()

처럼 보인다.

필요한 건:

LLM output
   ↓
ArtifactStripper
   ↓
SyntaxValidator
   ↓
save()

이다.

이건 governance 이전 문제다.

최소한 이건 바로 추가해야 한다:

const FORBIDDEN_ARTIFACTS: &[&str] = &[
    "```",
    "---CODE",
    "END===",
    "<think>",
    "</think>",
];

검출 시:

return Err(ArtifactContamination)
P1 — ContractSignatureValidator

이건 네 분석이 정확하다.

현재 시스템은:

header declaration
!=
implementation
!=
call site

를 전혀 검증 안 한다.

그래서:

db_init();

정의했는데:

init_db();

호출해도 통과한다.

이건 semantic validator가 아니라 ABI contract validator에 가깝다.

필요한 구조:

SymbolTable
   ├─ exports
   ├─ imports
   └─ signatures

그리고 validation:

caller imports
    ⊆
provider exports

C에서는 최소한:

함수명
arity
return type

까지는 검증해야 한다.

P2 — Syntax Gate Hard Fail

이건 무조건 들어가야 한다.

현재 Senior reviewer는 자연어 reviewer다.

필요한 건:

semantic review
+
compiler truth

이다.

즉:

gcc -fsyntax-only
cargo check
python -m py_compile

이 pipeline에 들어가야 한다.

LLM review는 compiler를 절대 대체 못 한다.

P3 — LanguageBinding

이건 맞는데, 지금 당장 손대면 범위가 커진다.

우선 최소 수정으로:

enum Language {
    C,
    Rust,
    Python,
}

도입하고:

match language {
    Language::C => HeaderGen,
    Language::Rust => ModuleGen,
    Language::Python => PackageGen,
}

정도부터 해야 한다.

그리고 아주 중요한 부분:

현재 네 pipeline은 실제로:

Natural Language Driven

이다.

하지만 필요한 건:

Constraint Driven

이다.

즉:

현재:

spec text
   ↓
LLM imagination

필요한 것:

spec
   ↓
semantic extraction
   ↓
typed constraints
   ↓
bounded generation

이다.

그리고 이 부분:

Senior가 syntax 오류를 놓침

이건 정상이다.

왜냐면 LLM reviewer는:

컴파일러 아님
parser 아님
linker 아님

이다.

따라서 architecture 자체가 잘못된 기대를 하고 있다.

Senior reviewer 역할은:

semantic coherence

이지:

compiler correctness

가 아니다.

그래서 추천 순서는 명확하다.

지금 해야 하는 실제 수정 순서
1단계

Artifact Firewall

markdown 제거
think tag 제거
code fence 제거
2단계

Compiler Gate

gcc -fsyntax-only
cargo check
py_compile

실패 시 hard fail.

3단계

ContractSignatureValidator

symbol export/import graph
unresolved symbol 차단
4단계

LanguageBinding

Rust → ModuleGen
C → HeaderGen
Python → PackageGen
5단계

Ontology Refactor

여기서부터 진짜 semantic architecture 단계다.

현재 가장 위험한 건 사실 이것이다:

잘못된 코드가 승인되는 것

이 아니라:

잘못된 코드가 구조적으로 "정상"처럼 축적되는 것

이다.

왜냐면 지금 pipeline은 semantic corruption을 계속 학습하게 되는 구조에 가깝다.
