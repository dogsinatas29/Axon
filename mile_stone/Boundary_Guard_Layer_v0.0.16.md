# Boundary Guard Layer v0.0.16

## 1. 목적

LLM이 시스템 구조(State / Task / Contract)를 임의로 변경하지 못하도록
런타임에서 강제 차단한다.

------------------------------------------------------------------------

## 2. 핵심 정의

Boundary Guard = "허용된 것만 실행되고, 나머지는 전부 거부된다"

------------------------------------------------------------------------

## 3. 보호 대상

-   State 구조
-   Task 정의 (read/write scope)
-   Contract schema
-   Planner DAG

------------------------------------------------------------------------

## 4. 검증 규칙

### 4.1 State 접근 검증

-   선언되지 않은 READ 접근 → reject
-   선언되지 않은 WRITE 시도 → reject

------------------------------------------------------------------------

### 4.2 State 변경 검증

-   허용되지 않은 key 생성 → reject
-   기존 구조 변경 → reject

------------------------------------------------------------------------

### 4.3 Task 구조 검증

-   정의되지 않은 task 생성 → reject
-   task schema 변경 → reject

------------------------------------------------------------------------

### 4.4 Contract 검증

-   필드 누락 → fail
-   타입 불일치 → fail

------------------------------------------------------------------------

## 5. 실행 흐름

1.  Task 실행 요청
2.  Guard 검증
3.  통과 시 실행
4.  실패 시 즉시 중단

------------------------------------------------------------------------

## 6. Runtime Hook

``` text
beforeTaskExecute(task, state):
  validateReadScope()
  validateWriteScope()
  validateContract()

afterTaskExecute(task, state):
  validateStateDiff()
```

------------------------------------------------------------------------

## 7. State Diff 검증

``` text
diff = new_state - old_state

if diff not in allowed_write_scope:
    reject
```

------------------------------------------------------------------------

## 8. 강제 정책

-   allowlist 기반 (whitelist)
-   default deny

------------------------------------------------------------------------

## 9. 실패 처리

-   즉시 중단 (fail-fast)
-   snapshot rollback
-   로그 기록

------------------------------------------------------------------------

## 10. 금지 사항

-   soft validation (경고만)
-   동적 schema 확장
-   runtime에서 contract 수정

------------------------------------------------------------------------

## 11. 결론

Boundary Guard는 선택이 아니라 필수다.

이 레이어가 없으면: - State 오염 - 구조 붕괴 - LLM uncontrolled behavior
발생

Guard는 시스템의 "물리적 안전장치"다.
