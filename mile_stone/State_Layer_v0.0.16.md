# State Layer (Decomposed) v0.0.16

## 1. 목적

Task 단위 실행에서 최소 상태 접근을 보장하고, Snapshot / Idempotency /
Scheduler 안정성을 확보한다.

------------------------------------------------------------------------

## 2. 핵심 원칙

### 2.1 Read / Write 분리

모든 Task는 명시적으로 선언한다.

-   READ: 참조하는 state
-   WRITE: 변경하는 state

------------------------------------------------------------------------

### 2.2 최소 접근

❌ 전체 state 접근\
✔ 필요한 key만 접근

------------------------------------------------------------------------

### 2.3 영향 범위 제한

Task 간 state write 영역은 겹치지 않도록 설계한다.

------------------------------------------------------------------------

## 3. State 구조

``` json
{
  "context": {},
  "planner": {},
  "tasks": {},
  "artifacts": {},
  "runtime": {}
}
```

------------------------------------------------------------------------

## 4. 영역 정의

### context

-   사용자 입력
-   글로벌 설정 → 거의 immutable

### planner

-   DAG 구조
-   dependency → planner만 수정 가능

### tasks

-   task 결과
-   상태 (pending / done / failed)

### artifacts

-   생성된 코드 / 파일 / 결과물

### runtime

-   scheduler 상태
-   queue
-   execution metadata

------------------------------------------------------------------------

## 5. Task 정의 예시

``` json
{
  "task_id": "build_component",
  "read": [
    "context.requirements",
    "planner.structure"
  ],
  "write": [
    "artifacts.component_code"
  ]
}
```

------------------------------------------------------------------------

## 6. 실행 검증

-   선언되지 않은 state 접근 → reject
-   허용되지 않은 write → reject

------------------------------------------------------------------------

## 7. Snapshot 연계

-   변경된 state만 추적
-   snapshot 경량화
-   복구 속도 향상

------------------------------------------------------------------------

## 8. Idempotency 연계

fingerprint = hash(input + read_state_only)

------------------------------------------------------------------------

## 9. 금지 사항

-   전체 state 전달
-   write 범위 미정의
-   runtime state 포함 fingerprint

------------------------------------------------------------------------

## 10. 결론

State 분해는 모든 안정성 계층의 기반이다.

Scheduler / Idempotency / Recovery / Guard는\
이 구조 위에서만 정상 동작한다.
