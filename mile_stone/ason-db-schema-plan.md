# AXON DB Schema Plan

## ason-db-schema-plan.md

Version: v0.1

------------------------------------------------------------------------

# 1. Overview

AXON uses a **thread-centered database model** designed to support
multi-agent AI collaboration, event-driven execution, and full project
traceability.

The system stores conversations, generated artifacts, and approvals
within a single execution unit called a **Thread**.

Core concept:

Thread ├ messages ├ artifacts └ approvals

This design ensures that the entire development process can be traced,
reviewed, and replayed.

------------------------------------------------------------------------

# 2. Database Engine

AXON uses a single local database:

axon.db

Database engine:

SQLite

Reasons:

-   lightweight
-   local-first architecture
-   simple persistence model
-   easy backup and portability

------------------------------------------------------------------------

# 3. Core Tables

AXON database consists of the following core tables.

projects\
agents\
threads\
messages\
artifacts\
approvals\
events

------------------------------------------------------------------------

# 4. Projects Table

projects

Purpose:

Represents a single software project.

Columns:

id\
name\
status\
created_at\
config

Notes:

Free users are limited to one active project.\
Paid users can run multiple projects concurrently.

------------------------------------------------------------------------

# 5. Agents Table

agents

Purpose:

Stores AI agents participating in the project.

Columns:

id\
project_id\
name\
role\
model\
manager_id\
status\
created_at

Roles:

architect\
senior\
junior

Hierarchy:

architect └ senior └ junior

manager_id defines the reporting structure.

------------------------------------------------------------------------

# 6. Threads Table

threads

Purpose:

Represents the core unit of work in AXON.

Columns:

id\
project_id\
title\
type\
status\
created_at\
updated_at

Thread Types:

task\
nogari\
system

Thread Status:

open\
running\
waiting\
completed\
archived

Threads contain conversation, generated outputs, and approvals.

------------------------------------------------------------------------

# 7. Messages Table

messages

Purpose:

Stores all conversation messages within a thread.

Columns:

id\
thread_id\
agent_id\
role\
content\
created_at

Roles:

architect\
senior\
junior\
system\
user

Messages represent reasoning, discussion, and instructions between
agents.

------------------------------------------------------------------------

# 8. Artifacts Table

artifacts

Purpose:

Stores outputs produced by agents inside a thread.

Columns:

id\
thread_id\
type\
path\
content\
version\
created_at

Artifact Types:

code\
document\
plan\
analysis

Examples:

src/auth/login.rs\
docs/architecture.md

Artifacts represent the produced deliverables of a thread.

------------------------------------------------------------------------

# 9. Approvals Table

approvals

Purpose:

Handles review and approval workflow.

Columns:

id\
thread_id\
artifact_id\
requested_by\
status\
created_at\
approved_at

Status:

pending\
approved\
rejected

Approvals allow the user to review AI generated outputs before
integration.

------------------------------------------------------------------------

# 10. Events Table

events

Purpose:

Stores persisted system events generated by the event bus.

Columns:

id\
project_id\
type\
payload\
source\
created_at

Example events:

thread_created\
message_added\
artifact_created\
approval_requested\
thread_completed

Events allow system replay and auditing.

------------------------------------------------------------------------

# 11. Database Relationship Model

projects │ ├ agents │ ├ threads │ ├ messages │ ├ artifacts │ └ approvals
│ └ events

This structure ensures that all work performed by AI agents is stored
within the context of a thread.

------------------------------------------------------------------------

# 12. Design Characteristics

Key properties of the AXON database model:

Thread-centric workflow storage

Conversation + output + review contained in one unit

Event-driven state persistence

Full traceability of AI reasoning and code generation

Support for multi-agent orchestration

------------------------------------------------------------------------

# 13. Future Extensions

Potential schema extensions may include:

artifact_versions

execution_metrics

runtime_logs

architecture_state

agent_performance_metrics

However, these are optional additions and not required for the core AXON
runtime.

------------------------------------------------------------------------

# 14. Summary

The AXON database schema focuses on capturing the **entire AI-driven
development process**.

Instead of separating chat, code, and task management systems, AXON
unifies them through a thread-centered architecture.

Thread ├ conversation ├ artifacts └ approvals

This structure enables:

-   full development traceability
-   reliable code review workflows
-   replayable AI development sessions
