# AXON Runtime Specification

## 1. Purpose

This document defines the **AXON Runtime**, the executable layer that
turns the AXON architecture into a working system.

While other specifications describe concepts (architecture, scheduler,
events, database), this document defines:

-   The runtime execution graph
-   Module boundaries
-   Ownership of system state
-   The main execution loop
-   Module interaction contracts

This document is intended to enable **direct implementation**.

------------------------------------------------------------------------

# 2. Core Principle

AXON runtime follows a strict rule:

**All system state lives on the Board.**

Modules may read or modify the board but must not maintain hidden
runtime state.

This enables:

-   deterministic behavior
-   full replay capability
-   transparent system debugging

------------------------------------------------------------------------

# 3. Runtime Architecture

AXON runtime consists of the following modules:

Runtime ├ Scheduler ├ Board Engine ├ Event Bus ├ Agent Engine ├
Workspace Manager └ Patch Engine

Responsibilities:

Scheduler\
Determines which thread runs next.

Board Engine\
Maintains the canonical system state.

Event Bus\
Propagates state changes to subsystems.

Agent Engine\
Executes AI workers.

Workspace Manager\
Handles project files and artifacts.

Patch Engine\
Applies code modifications generated by agents.

------------------------------------------------------------------------

# 4. Runtime Execution Graph

The execution flow of AXON follows this structure:

Board ↓ Scheduler ↓ Agent Engine ↓ Workspace Manager ↓ Patch Engine ↓
Board Update ↓ Event Bus

This loop continues until no runnable threads exist.

------------------------------------------------------------------------

# 5. Main Runtime Loop

Pseudo-code representation:

while true:

    events = event_bus.poll()

    board.apply_events(events)

    thread = scheduler.find_runnable_thread(board)

    if thread:

        step = scheduler.determine_step(thread)

        agent = agent_engine.assign(step)

        result = agent_engine.execute(agent, step)

        workspace.apply(result.artifacts)

        patch_engine.apply(result.patch)

        board.commit(result)

        event_bus.emit(result.events)

    else:

        runtime.idle()

------------------------------------------------------------------------

# 6. Module Boundaries

Each module has strict responsibilities.

Scheduler

Inputs: - board state

Outputs: - selected thread - next step definition

Scheduler must never modify board state directly.

Board Engine

Inputs: - events - commit results

Outputs: - updated board state

Board is the **single source of truth**.

Agent Engine

Inputs: - task description - workspace context

Outputs: - artifacts - patches - events

Workspace Manager

Inputs: - artifacts - project structure

Outputs: - filesystem updates

Patch Engine

Inputs: - code patches

Outputs: - applied modifications

Event Bus

Inputs: - events

Outputs: - broadcast notifications

------------------------------------------------------------------------

# 7. Data Ownership

Ownership rules prevent corruption.

Board Engine owns:

-   threads
-   milestones
-   project state
-   task status

Workspace owns:

-   source files
-   build artifacts

Agent Engine owns:

-   temporary reasoning
-   prompt execution

Scheduler owns:

-   scheduling decisions

No module may directly mutate data owned by another module.

All changes must pass through defined interfaces.

------------------------------------------------------------------------

# 8. Thread Lifecycle

Thread lifecycle:

draft ↓ junior_proposal ↓ senior_review ↓ architect_review ↓
boss_approval ↓ completed

Scheduler determines the next step based on thread state.

------------------------------------------------------------------------

# 9. Idle Runtime Behavior

When no runnable threads exist the runtime enters idle state.

Idle actions may include:

-   idle conversation system (Nogari)
-   planning discussions
-   dependency analysis

Idle processing must never modify board state without explicit events.

------------------------------------------------------------------------

# 10. Deterministic Execution

AXON runtime must be deterministic.

Given:

-   identical board state
-   identical event history

The runtime must produce identical results.

This enables:

-   replay
-   debugging
-   system auditing

------------------------------------------------------------------------

# 11. Replay Capability

Because all state transitions occur through events and board commits,
the entire runtime session can be replayed.

Replay procedure:

1.  load board snapshot
2.  replay event log
3.  re-run scheduler loop

This reconstructs the exact execution sequence.

------------------------------------------------------------------------

# 12. Implementation Order

Recommended module implementation order:

1.  Board Engine
2.  Scheduler
3.  Event Bus
4.  Agent Engine
5.  Workspace Manager
6.  Patch Engine
7.  Runtime Loop

This order allows incremental system validation.

------------------------------------------------------------------------

# 13. Conceptual Model

AXON operates like an automated software company.

Human user → CEO / Boss

Scheduler → Production Manager

Agents → Workers

Threads → Tasks

Milestones → Product goals

The runtime coordinates AI workers to advance software development.
