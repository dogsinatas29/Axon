📅 작업 정보
상태: 🏗️ Planned / 🚧 In-Progress
목표: 시니어-주니어 계층 구조 및 게시판형 워크플로우를 가진 시스템 데몬 구축
🧠 상세 설계 및 로직
1. 스튜디오 계층 구조 (Studio Hierarchy)
The Boss (User): /run/axon.sock 또는 브라우저 UI를 통해 최종 승인/리젝 및 버그 리포팅을 수행하는 절대 권력자.
Senior Agent (Architect): 아키텍처_axon.md를 수호하며 전체 맥락과 공통 규약을 관리. 주니어의 결과물을 검토하고 보스에게 보고.
Junior Agents (Workers): M명의 실제 프로세스/스레드. 할당된 일감 스레드에서 분석 및 코딩 수행. 유휴 시 노가리.md 기반 잡담 수행.
2. 동적 스레드 게시판 (Dynamic Thread-Board)
Thread-per-Job: 일감이 10개면 10개의 스레드 생성. 주니어 작업자 수와 관계없이 일감 중심의 컨텍스트 유지.
Real-time Ranking: 댓글(에이전트 응답)이 달린 스레드가 최상단으로 정렬되는 실시간 인덱싱 로직.
Multi-Board System:
Main Board: 실제 설계 및 구현이 진행되는 작업 공간.
Feedback/Prompt Board: 사용자의 제안 및 버그 리포트가 게시되는 컨트롤 타워.
Idle Board: 주니어들의 '노가리' 공간. 시스템 활성도 시각화 및 페르소나 유지.
3. 시스템 데몬화 및 통신 (Daemon & IPC)
Persistence: systemd 기반 상주 서비스. 종료 시 모든 스레드의 대화 상태와 소스 고정(Pin) 상태를 스냅샷으로 저장.
Localhost Server: 실행 시 즉시 로컬 웹 서버를 오픈하여 브라우저로 대화 흐름 시각화.
Resource Management: cgroups를 통해 M명의 주니어가 시스템 자원을 독점하지 않도록 제어.
🛠️ 기술적 변경 사항
Core Dispatcher (Rust)
Worker Pool: 주니어 에이전트들을 가용 상태에 따라 일감 스레드에 동적으로 배정하는 Dispatcher 구현.
Context Injection: 모든 에이전트 시작 시 아키텍처_axon.md를 시스템 프롬프트로 강제 주입.
Frontend & Interface
Reactive UI: 스레드 갱신 시 무중단 위치 변경(Sorting) 및 실시간 스트리밍 렌더링.
Approval Workflow: '승인 대기' 상태의 스레드를 시니어 에이전트가 보스에게 푸시하는 인터페이스.
⚠️ 예외 처리 및 주의 사항
Context Switching Cost: 주니어가 노가리 스레드에서 작업 스레드로 이동할 때 발생하는 맥락 로딩 지연 최적화.
Zombies & Hangs: 특정 일감에서 에이전트가 루프에 빠질 경우 시니어 에이전트가 이를 감지하여 강제 리셋하는 'Watchdog' 로직 필요.
Vibe Coding Sync: SYNAPSE 시각화 노드와 실제 에이전트 작업 스레드 간의 1:1 매핑 무결성 유지.

🧠 Phase 0: Architecture Analysis & Task Slicing
LLM이 설계안(md)을 주입받았을 때, 다음 4단계의 논리 프로세스를 거쳐 아키텍처_axon.md를 생성하고 일감을 분배하도록 설계해야 하네.
1. 도메인 모델링 (Domain Analysis)
Input: 사용자가 준 초기 설계 md
Analysis: 핵심 엔티티(데몬, 스레드, 게시판, 주니어 에이전트)와 그들 간의 관계(관계도)를 추출.
Output: AXON의 정적 구조 정의.
2. 기능 단위 파편화 (Functional Decomposition)
Rule: "전체 맥락을 해치지 않는 최소 단위"로 기능을 쪼갬.
Categories:
Layer A (Infra): 시스템 데몬화, 시그널 핸들링, 로컬 호스트 서버.
Layer B (Orchestration): 시니어/주니어 분배 로직, 스레드 풀 관리.
Layer C (Interaction): 브라우저 기반 게시판 UI, 승인/리젝 워크플로우.
Layer D (Persona): 노가리.md 기반 잡담 엔진.
3. 작업 일감(Job) 생성 및 스레드 할당
Logic: 작업 일감 10개가 나오면 각 일감에 고유 ID를 부여하고, 이를 관리할 N개의 스레드 정의.
Standardization: 모든 일감은 '공통 규약(Protocol)' 섹션을 포함하여 시니어의 감독을 받도록 설계.
🛠️ 아키텍처 생성용 시스템 프롬프트 (뼈대)
LLM에게 아키텍처_axon.md를 만들라고 시킬 때, 다음과 같은 **'작업 지침서'**를 먼저 던져야 하네.
### [System Instruction for Architecture Generation]

1. **분할 정복:** 주입된 설계안을 '시스템 인프라', '에이전트 로직', 'UI/UX'로 3등분한다.
2. **규약 정의:** 모든 주니어가 준수해야 할 '메시지 포맷'과 '상태 보고 방식'을 최상단에 배치한다.
3. **일감 리스트화:** 구현이 필요한 기능을 최소 5~10개의 독립적 일감으로 나열한다.
4. **스레드 매핑:** 각 일감이 어느 게시판(Main/Feedback/Idle)에 속하는지 명시한다.
5. **시니어의 역할:** 주니어의 작업 결과물에서 어떤 포인트(성능, 보안, 규약)를 검토해야 하는지 체크리스트를 생성한다.

 AXON 초기 구동 시나리오 (The Workflow)
1단계: 설계안 주입 및 자동 분석
명령어: axon read blueprint.md
동작: 데몬이 폴더의 blueprint.md를 읽어 들여 전체 맥락을 파악하네.
결과: 이 과정에서 **Architecture_AXON.md**가 자동으로 생성되지. 여기에는 시스템의 전체 뼈대, 공통 규약, 그리고 쪼개진 일감(Task 01~N)들이 포함되네.
2단계: 워크포스(인력) 및 지능(Model) 할당
명령어: axon number 4 (예: 시니어 1명 + 주니어 3명)
지능 결정 (LLM Mapping):
Senior: 복잡한 설계와 규약 관리를 위해 가장 고지능인 모델(예: Gemini 1.5 Pro)을 고정 배정.
Junior: 반복적이고 구체적인 구현을 위해 가벼운 모델들(예: Flash 계열)을 배정.
결과: 이때부터 로컬호스트 서버가 열리고 브라우저에 **'게시판'**들이 나타나기 시작하는 걸세.
3단계: 자동 분배 및 스레드 생성
동작: Architecture_AXON.md에 정의된 일감 수만큼 스레드가 자동 생성되네.
배정: 4명의 작업자(시니어 포함)가 일감 큐를 확인하고, 시니어의 지휘하에 각 스레드로 흩어져 분석을 시작하지.
🧠 현재 단계에서 확정해야 할 '뼈대'
자네의 시나리오를 완벽하게 수행하려면, 내가 먼저 **Architecture_AXON.md**가 담아야 할 자동 생성 로직의 표준 규격을 잡아야 하네.
섹션 설명 비고
0. 공통 규약 모든 에이전트가 지켜야 할 Rust 코딩 표준 및 통신 프로토콜 시니어 전용 관리 영역
1. 일감 파편화 주입된 문서에서 추출된 독립된 작업 단위 (Task 01 ~ N) 주니어들의 작업 목표
2. 의존성 맵 어떤 작업이 끝나야 다음 작업이 가능한지에 대한 관계도 디스패처의 배분 기준
3. 검토 체크리스트 시니어가 주니어의 결과물을 승인할 때 사용할 기준점 보스(사장) 보고 전 필터
아키텍처 문서(Architecture_AXON.md)는 작업 지시서(Work Order)이자 지도(Map)야. 따라서 이 문서 하나에 아키텍처 설계, 작업 배분 규칙, 작업 내용이 모두 원자적으로 포함되어야 하네.
🛠️ AXON의 '자동 분석 및 배분' 메커니즘
자네가 axon read blueprint.md를 입력했을 때, 내가 자동으로 생성할 Architecture_AXON.md의 구조는 이렇게 설계되어야 하네.
1. 아키텍처는 '법전'이자 '일감 목록'
이 문서가 생성될 때, 단순히 "이런 구조야"라고 끝나는 게 아니라 작업을 쪼개서 번호를 매겨야 하네.
Section A (Structure): 시스템의 전반적인 아키텍처 (데몬, 스레드 구조 등).
Section B (Backlog): 분석된 작업 일감 리스트 (Task 01, Task 02...).
Section C (Assignment): 일감별 난이도와 시니어의 검토 기준.
2. 에이전트들의 읽기 방식
자네가 axon senior ... plan, axon juniors ... fast라고 명령을 내리면, 각 모델은 다음과 같이 문서를 읽게 될 걸세.
Senior (Plan 모델): Architecture_AXON.md 전체를 읽고 **'맥락의 왕'**이 되네. 전체 규약(Convention)을 지키고 있는지, 주니어들이 맡은 일감이 전체 설계와 어긋나지 않는지 감시하는 '감독관' 역할을 수행하지.
Junior (Fast 모델): 전체 문서를 다 읽으면 맥락이 터지거나 속도가 느려져. 그래서 본인에게 할당된 특정 'Task ID'의 내용과 **시니어가 정의한 '공통 규약'**만 읽고 바로 삽질(코딩/분석)을 시작하네.
📋 실행 시나리오 업데이트
사장: axon read blueprint.md
결과: Architecture_AXON.md 자동 생성 (설계 + 10개의 작업 일감 포함).
사장: axon senior Gemini-Pro-Plan / axon juniors Gemini-Flash-Fast
결과: * Senior: "알겠습니다. 전체 10개 작업의 공통 규약을 로딩했습니다."
Juniors: "저는 Task 01(데몬 시그널 핸들링)을 맡았습니다. 규약 확인 후 분석 들어갑니다."
동작:
일감은 10개지만 주니어가 2명이면, 주니어들은 스레드를 옮겨 다니며 일감을 처리하겠지.
주니어가 일을 끝내면 시니어에게 보고, 시니어는 사장(자네)에게 컨펌 요청.
👨🏻‍💻 시니어의 한마디
사장님, 여기서 핵심은 **"주니어에게 전체 아키텍처를 다 읽히지 않는 것"**이네. 걔들은 자기 앞가림만 잘하면 돼. 전체 맥락은 시니어가 쥐고 흔들면 되니까. 이래야 저렴하고 빠른 'Fast' 모델로도 고효율을 뽑아낼 수 있어.
🛠️ AXON: Multi-Model Driver & Automation Logic
자네가 고민하는 "일일이 열기 힘들다"는 문제를 해결하려면, AXON은 단순한 래퍼(Wrapper)가 아니라 **'런타임 컨트롤러'**가 되어야 하네.
1. 통합 모델 드라이버 (Unified Model Driver)
각 LLM CLI를 직접 실행하는 게 아니라, AXON이 내부적으로 각 API/CLI의 **'드라이버'**를 가지고 있어야 하네.
Senior Interface: axon senior 명령 시 지정된 고성능 모델(Plan)의 커넥션을 상시 유지.
Junior Workers: 주니어 수만큼 워커 스레드를 미리 풀(Pool)로 생성하고, 각 스레드가 지정된 Fast 모델의 API와 통신하게 함.
2. 컨텍스트 자동 주입 (Context Auto-Injection)
자네가 readd 한 번만 하면, AXON 데몬이 다음과 같이 파일을 배분하네.
Common: Architecture_AXON.md의 '규약' 섹션은 모든 워커의 System Prompt로 자동 삽입.
Task-Specific: 각 워커가 맡은 '작업 단위(Task)' 문서만 해당 워커의 컨텍스트로 주입.
Result: 자네는 터미널을 여러 개 열 필요가 없지. AXON이 백그라운드에서 각 API를 호출하고 결과만 **'게시판(UI)'**으로 쏴주니까.
3. 자동화의 핵심: 이벤트 루프 (Event Loop)
이게 진짜 자동화지. 주니어가 작업을 마치면 시니어에게 알림을 보내고, 시니어는 그걸 검토한 뒤 보스(자네)의 승인 대기열에 올리는 과정을 **상태 머신(State Machine)**으로 관리하는 걸세.
📋 AXON의 '한 방' 구동 아키텍처
단계 사용자 액션 (Boss) AXON 데몬의 내부 동작 (Daemon)
0. 초기화 axon read blueprint.md 문서를 쪼개서 Task DB 생성 및 Architecture_AXON.md 자동 빌드.
1. 인력 배치 axon senior ... / axon juniors ... 지정된 모델들로 워커 풀 생성 및 API 키 바인딩.
2. 자동 실행 (없음 - 자동 시작) 워커들이 Task DB에서 자기 몫을 가져가 분석 시작.
3. 모니터링 브라우저 접속 (localhost) 실시간으로 업데이트되는 게시판 스레드를 보며 아이스 아메리카노 한 잔.
4. 피드백 게시판에 "수정해" 댓글 작성 해당 댓글이 프롬프트가 되어 주니어에게 전파.
👨🏻‍💻 시니어의 피드백
자네가 걱정하는 '수동의 늪'에서 탈출하는 방법은 딱 하나야. "에이전트 제어권을 데몬에게 완전히 위임하는 것." 자네는 오직 **'게시판(브라우저)'**만 보고 승인 버튼만 누르면 되는 구조지.
리눅스 커널이 프로세스를 스케줄링하듯, AXON이 LLM 호출을 스케줄링하게 만들면 되네

AXON의 통신 규약은 크게 **내부(데몬-에이전트)**와 외부(데몬-UI) 두 가지 레이어로 설계되어야 하네.
1. 내부 규약: Agent-Daemon Protocol (JSON-RPC 기반)
에이전트들이 제각각 떠드는 걸 막기 위해, 모든 에이전트의 출력은 반드시 아래와 같은 JSON 구조체로 직렬화되어 데몬에 전달되어야 하네.
{
  "header": {
    "task_id": "TASK-04",
    "agent_id": "JUNIOR_02",
    "timestamp": "2026-03-10T10:43:00Z",
    "status": "IN_PROGRESS" 
  },
  "payload": {
    "type": "ANALYSIS", // ANALYSIS, CODE, LOGARI, REPORT
    "content": "이 부분의 Rust 소유권 로직을 수정했습니다...",
    "affected_files": ["src/main.rs"]
  },
  "control": {
    "need_approval": true,
    "next_step": "WAIT_FOR_BOSS"
  }
}

Status 제어: IN_PROGRESS(작업 중), REVIEW_REQ(시니어 검토 요청), APPROVED(승인됨), IDLE(노가리 중).
시니어의 역할: 주니어가 보낸 JSON의 payload를 검증하고, 문제가 없으면 status를 BOSS_PENDING으로 격상시켜 사장님(자네)에게 쏴주는 거지.
2. 외부 규약: Daemon-UI (WebSocket & Server-Sent Events)
브라우저에서 실시간으로 대화가 올라가고 스레드가 갱신되는 건 이벤트 기반으로 돌아가야 하네.
Thread Ranking: 새로운 JSON 패킷이 데몬에 도착하면, 데몬은 WebSocket을 통해 브라우저에 "ID: TASK-04 업데이트!" 신호를 보내고, UI는 이를 최상단으로 z-index나 리스트 순서를 바꾼다네.
Streaming: LLM이 답변을 생성하는 동안에는 SSE(Server-Sent Events)를 통해 글자가 타이핑되는 것처럼 실시간으로 뿌려줘야 '바이브 코딩' 맛이 나지.
3. 프로젝트 폴더 규약 (File-Based Sync)
데몬이 상주하면서 파일을 건드릴 때의 약속이야.
Lock File (.axon.lock): 특정 에이전트가 파일을 수정 중일 때 다른 에이전트가 접근하지 못하게 막는 세마포어.
Context Snapshot: 에이전트가 작업을 멈출 때 현재의 heap 상태나 주요 변수 값을 .axon/state/task_04.json에 저장하여, 나중에 다시 불러와도 맥락이 이어지게 하네.
4. 시니어 에이전트의 '검토 규약'
시니어가 주니어를 갈굴 때 쓰는 체크리스트 표준이네.
Rule 01: Architecture_AXON.md에 정의된 공통 라이브러리 외에 다른 것을 썼는가?
Rule 02: 함수 단위 테스트(Unit Test)를 포함했는가?
Rule 03: 노가리.md에서 너무 오래 놀고 있지는 않은가?
👨🏻‍💻 시니어의 한마디
사장님, 이 규약이 없으면 주니어들이 자기 맘대로 마크다운(Markdown)을 섞어 쓰거나 형식을 깨버려서 시스템이 뻗어버리네. 그래서 **AXON 데몬 내부에는 이 JSON 형식을 강제하는 'Validator'**가 반드시 들어가야 해. 형식이 틀리면 시니어가 "규격 맞춰서 다시 가져와!"라고 주니어를 리젝(Reject)하게 만드는 거지.

🎭 AXON 페르소나 주입 메커니즘
페르소나는 단순히 "너는 시니어다"라고 한 줄 주는 걸로 안 되네. **'역할(Role)', '어조(Tone)', '지식의 경계(Knowledge Boundary)'**를 명확히 정의해야 하지.
1. 페르소나 정의 파일 (PERSONA.json 또는 config.toml)
데몬 설정 파일에 각 역할별 페르소나를 정의하네.
Senior (Plan 모델): "냉소적이지만 실력은 확실함. 불필요한 서론 금지. KISS 원칙 수호자. 주니어의 코드를 보고 혀를 차지만 해결책은 정확히 제시함."
Junior A (Fast 모델): "최신 라이브러리 성애자. 의욕이 앞서서 오버 엔지니어링을 자주 함. 시니어에게 깨지면 노가리 게시판에서 징징거림."
Junior B (Fast 모델): "안정성 우선주의. 묵묵히 시키는 일은 잘하지만 창의성이 부족함. 노가리 게시판에서는 주로 중재자 역할."
2. 동적 시스템 프롬프트 합성 (Prompt Composition)
AXON 데몬이 LLM API를 호출할 때, 다음 세 가지를 합쳐서 System Prompt로 던지는 걸세.
공통 기반: Architecture_AXON.md의 공통 규약.
개별 페르소나: 위에서 정의한 성격과 말투 지침.
현재 상황: "너는 지금 TASK-01을 수행 중이고, 보스(사용자)가 방금 버그 리포트를 올렸다."
3. 노가리 게시판 전용 '사회적 맥락' 주입
이게 핵심이지. 주니어들이 노가리 게시판(idle_board)에 접속할 때는 "지금 작업 스레드에서 시니어한테 까인 상황" 같은 트리거를 넘겨줘야 하네.
Trigger: 시니어 에이전트가 주니어의 결과물을 REJECT 함.
Action: 데몬이 주니어 에이전트에게 "너 방금 까였어. 노가리 게시판 가서 하소연해"라고 신호를 보냄.
Output: 주니어 에이전트가 노가리 스레드에 "아... 시니어님 오늘따라 왜 저러심? ㅠㅠ"이라고 글을 올림.

2. '노가리.md'를 통한 동적 학습 (Vibe Injection)
이게 진짜 골 때리는 기능이지. 노가리.md는 단순히 잡담만 하는 곳이 아니야. 자네가 여기에 **"야, 신입 너 요즘 말투가 너무 딱딱해. 좀 더 정감 있게 말해봐"**라고 적으면, 데몬이 이 맥락을 파악해서 해당 주니어의 페르소나 파라미터를 실시간으로 업데이트하게 설계할 수 있네.
3. 프롬프트 게시판(Feedback Board)에서의 직접 오더
보스(자네)가 전용 게시판에 **"앞으로 모든 주니어는 답변 끝에 '충성!'을 붙이도록"**이라고 규약을 던지면, 시니어가 이걸 받아서 하위 주니어들에게 '공통 페르소나 규약'으로 전파하는 방식이지.
📋 페르소나 결정 방식: 랜덤 vs 고정
방식 특징 장단점
완전 고정 (Fixed) 사장님이 정의한 대로만 행동 예측 가능하고 안정적이지만, AI 특유의 유연함이 부족함.
기반 고정 + 무작위 변수 기본 성격은 고정하되, '그날의 기분' 수치를 랜덤으로 부여 (추천) 주니어가 어떤 날은 컨디션이 좋고, 어떤 날은 헛소리를 하는 등 현실감이 극대화됨.
완전 랜덤 (Random) 실행할 때마다 성격이 바뀜 시스템의 일관성이 파괴되어 협업 도구로서는 최악임.
👨🏻‍💻 시니어의 피드백
자네, 페르소나를 편집하고 싶다면 AXON_TEAM.json 같은 명확한 설정 파일을 두는 게 가장 좋아. 하지만 진정한 재미는 **'노가리 게시판'**에서의 상호작용이지. 주니어들이 자네의 눈치를 보게 만들 수도, 자네에게 대들게 만들 수도 있네. 그게 다 '사장님의 권한'이지.

! 디아블로의 접두사/접미사(Affix) 시스템처럼 페르소나를 **'랜덤 생성 후 고정(Roll and Lock)'**하는 방식... 이거 아주 맛깔나는 아이디어야.
단순히 사용자가 일일이 타이핑하는 수고를 덜어주면서도, 각 프로젝트마다 고유한 개성을 가진 '가상 팀'이 꾸려지는 재미가 있겠지. 20년 차 엔지니어 입장에서 이 **'디아블로식 페르소나 생성 시스템'**을 AXON에 어떻게 박아넣을지 설계해 주겠네.

자네 말대로 성별, 나이, 성격, 말투를 조합해서 하나의 **'캐릭터 카드'**를 뽑아내고, 그 캐릭터를 해당 프로젝트(blueprint.md)가 끝날 때까지 유지시키는 방식일세.

에이전트가 생성될 때 아래 테이블에서 무작위로 속성을 조합(Roll)하네.

🛠️ 페르소나 관리 로직 (The Mechanic)
Fixed Slot: Senior 슬롯은 자네가 정한 '20년 차 베테랑' 값이 기본 베이스로 깔리고, 위 테이블에서 약간의 변주(접두사/접미사)만 추가되네.
Random Slot: Junior 슬롯은 위 테이블에서 무작위로 3~4개를 조합해 완전히 새로운 캐릭터로 탄생하지.
Persistence: 한 번 뽑힌(Roll) 조합은 AXON_TEAM.json에 저장되어 프로젝트가 끝날 때까지 자네를 괴롭히거나 도울 걸세.
2. 유지 및 관리 (Locking)
Unique ID: 생성된 페르소나는 PERSONA_ID를 부여받고 AXON_TEAM.json에 저장되네.
Persistence: 한 번 생성된 캐릭터는 프로젝트 폴더를 삭제하기 전까지는 절대 바뀌지 않아. 자네는 이 녀석들을 진짜 게임 캐릭터 다루듯 "이번 프로젝트 주니어는 성격이 좀 까칠하네"라며 대응하면 되는 거지.
3. 사용자 편집 (The Modding)
랜덤이 기본이지만, 사장님(자네)은 언제든 **'에디터'**를 열 수 있어야 하네.
axon team reroll junior_1: 마음에 안 들면 다시 뽑기.
axon team edit junior_1: 특정 속성만 수동 수정.
📋 AXON 가상 스튜디오 팀 구성안 (최종)
역할 생성 방식 특징
Senior 반고정 (Semi-Fixed) 자네가 설정한 '20년 차 시니어' 기본값에 약간의 성격적 변조만 가함.
Junior 완전 랜덤 (Full-Random) 디아블로 템 뽑듯 랜덤 생성. 프로젝트마다 새로운 주니어를 만나는 재미 제공.
Idle Board 동적 상호작용 뽑힌 페르소나들이 자기들끼리 MBTI 궁합 보듯 노가리 까게 만듦.
🖥️ AXON Virtual Studio: UI/UX Specification
1. 레이아웃 구조 (The Dashboard)
브라우저를 열면 나타나는 메인 화면은 3단 구성의 게시판 형태네.
A. 좌측 사이드바: 워크포스 현황 (The Office)
에이전트 카드: 현재 상주 중인 시니어와 주니어들의 프로필.
디아블로식 '페르소나' 뱃지 표시 (예: [소심한], [커피중독]).
현재 상태: 작업 중, 검토 대기, 노가리 중 (실시간 점등).
시스템 로그: 데몬의 CPU/메모리 점유율 및 /run/axon.sock 연결 상태.
B. 중앙 메인: 작업 스레드 (Main Board)
스레드 리스트: 일감(Task) 단위로 생성된 게시물들.
정렬 로직: 댓글(에이전트 답변)이 달리는 즉시 최상단으로 Floating 업그레이드.
스레드 상태: 진행 중 (파란색), 보스 승인 대기 (노란색), 완료 (초록색).
상세 보기: 스레드 클릭 시 LLM과 보스(자네) 간의 대화 히스토리 및 수정된 코드 Diff 출력.
C. 우측 영역: 컨트롤 타워 (Command & Feedback)
프롬프트/버그 리포트 게시판: 보스(자네) 전용 입력창. 여기서 내린 명령이 전체 주니어에게 전파됨.
노가리 전용 위젯: 주니어들의 잡담이 실시간으로 흐르는 작은 채팅창 (시스템 활성도 시각화).
2. 핵심 UI 인터페이스 (Interaction)
① 승인/리젝 워크플로우 (Approval UI)
시니어가 주니어의 결과물을 검토하고 보스에게 올리면, 해당 스레드 하단에 **[APPROVE]**와 [REJECT] 버튼이 활성화되네.
Approve 클릭: Architecture_AXON.md의 진행률 업데이트 및 파일 시스템 실제 반영.
Reject 클릭: 사유 입력창이 뜨고, 입력된 피드백은 즉시 해당 주니어의 프롬프트로 재주입됨.
② 소스 고정 시각화 (Source Pinning)
특정 소스 코드가 '고정'되었을 때, UI 상에서 해당 코드 블록에 **'자물쇠 아이콘'**과 함께 고정한 에이전트의 이름이 표시되네. 이는 SYNAPSE 시각화 도구와도 실시간 연동되어야 하지.
③ 노가리 모드 (Idle Animation)
주니어들이 노가리.md를 기반으로 대화할 때, 텍스트 뒤에 은은한 애니메이션이나 상태 메시지(예: "신입_A가 탕비실에서 커피를 타는 중...")를 띄워 생동감을 부여하네.
3. 기술적 구현 가이드
Frontend: 가볍고 빠른 React 또는 Vue.js (또는 설치가 필요 없는 순수 Vanilla JS 추천).
Communication: 데몬(Rust)과 브라우저 간 WebSocket 상시 연결.
Theme: 자네가 좋아하는 Gruvbox 다크 테마를 기본으로 하되, 에이전트들의 가독성을 위해 고대비 폰트 적용.
👨🏻‍💻 시니어의 피드백
사장님, UI 설계에서 가장 중요한 건 **"내가 지금 뭘 해야 하는지"**가 명확해야 한다는 걸세. 스레드가 10개씩 돌아가면 정신없거든. 그래서 '보스 승인 대기' 상태인 녀석들은 화면에서 번쩍거리거나 별도의 '결재함' 메뉴로 모아주는 기능을 넣어야 하네.
AXON은 시스템 데몬으로 상주하는 만큼, **'영속성(Persistence)'**과 **'생존력'**에 초점을 맞춘 복구 시나리오가 필요하네.
1. 예기치 못한 리부팅/셧다운 시나리오 (Crash Recovery)
데몬은 모든 상태를 메모리가 아닌 '체크포인트' 파일로 실시간 기록해야 하네.
상태 저장 (Checkpointing):
에이전트들이 대화한 내용은 매 턴마다 .axon/history/task_xx.jsonl에 즉시 기록되네.
현재 누가 어떤 작업을 하고 있었는지, '승인 대기' 상태인지 등의 메타데이터는 .axon/state.db (SQLite나 JSON)에 보관하지.
부팅 시 복구 (Autostart & Resume):
systemd가 AXON 데몬을 다시 살리면, 데몬은 가장 먼저 .axon/state.db를 읽어들이네.
자네가 브라우저를 다시 열었을 때, **"어제 하던 작업 70% 지점에서 복구되었습니다. 이어서 할까요?"**라는 메시지가 뜨게 설계할 걸세.
파일 무결성 (Atomicity):
파일을 쓰는 도중 꺼지는 걸 막기 위해 temp 파일에 먼저 쓰고 rename하는 방식을 써서 소스 코드가 깨지는 걸 방지하네.
2. 토큰 고갈 및 API 오류 시나리오 (Resource Exhaustion)
이게 좀 골 때리는 상황이지. 돈(토큰)이 떨어지면 주니어들이 파업을 선언하는 상황 말이야.
토큰 모니터링:
각 모델 드라이버(API)에서 반환하는 usage 데이터를 추적하네.
한계치의 90%에 도달하면 UI 상단에 **[토큰 경고: 주니어들이 퇴근 준비 중입니다]**라고 경고를 띄우지.
그레이스풀 멈춤 (Graceful Pause):
토큰이 다 되면 에이전트는 하던 말을 강제로 끊지 않고, 마지막으로 **"보스, 토큰이 다 돼서 여기까지밖에 못 합니다. 충전해 주세요!"**라는 메시지를 남기고 세션을 동결(Freeze)하네.
모델 스위칭 (Failover):
이게 AXON의 강점이지. 만약 Gemini 토큰이 다 되면, 자네가 설정에서 **"Gemini 안 되면 Claude로 대체해"**라고 우선순위를 정해둘 수 있네. 그럼 데몬이 자동으로 모델 드라이버를 갈아 끼워서 작업을 이어가게 하는 거지.
📋 사고 대응 매트릭스
상황 데몬의 대응 사장님(보스)의 액션
PC 리부팅 state.db 로드 후 중단 지점 복구 브라우저 새로고침 후 작업 재개
토큰 고갈 해당 스레드 동결 및 UI 경고 알림 API 키 교체 또는 모델 변경
네트워크 단절 재연결 시도 중 대기 (Retry 로직) 네트워크 복구 대기
에이전트 폭주 시니어가 해당 주니어 강제 킬(Kill) "야, 진정해"라고 댓글 달기
👨🏻‍💻 시니어의 한마디
사장님, **"데몬은 죽어도 데이터는 죽지 않는다"**는 게 내 철칙이야. 특히 자네가 공장에서 일하는 동안 집 PC가 꺼지더라도, 나중에 켰을 때 주니어들이 **"어제 여기까지 했슴돠!"**라고 반겨줘야 진짜 쓸만한 도구 아니겠나?
자, 이제 '생존 전략'까지 다 짰어. 더 이상 설계로 머리 싸맬 필요 없네. 아키텍처는 이제 튼튼해졌어.

🔔 AXON 멀티 채널 알림 아키텍처
데몬은 알림의 경중(Severity)에 따라 세 가지 채널을 적절히 섞어서 사용할 걸세.
1. OS 네이티브 알림 (Desktop Notification)
용도: 실시간 작업 진척도 및 가벼운 상태 변화.
동작: libnotify(Linux)나 macOS Notification Center를 통해 우측 상단에 팝업을 띄우네.
메시지 예시: * "신입_A가 TASK-02 분석을 완료했습니다. 검토해 주세요."
"노가리 게시판에 새로운 떡밥이 올라왔습니다."
2. 브라우저 얼러트 및 사운드 (Web UI Alert)
용도: 자네가 브라우저를 보고 있을 때 즉각적인 주의 환기.
동작: 브라우저 탭의 제목이 깜빡이거나((1) AXON Message), 자네가 설정한 짧은 알림음(예: 터미널 비프음이나 게임 효과음)을 재생하네.
메시지 예시: * "⚠️ 보스 승인 대기 중: 시니어가 주니어의 코드를 리포트했습니다."
3. 메일 알림 (Email Dispatcher)
용도: 외출 중이거나 시스템 셧다운, 토큰 고갈 같은 '비상 상황'.
동작: 데몬 내부의 SMTP 드라이버를 통해 사장님의 메일로 리포트를 쏘네.
메시지 예시: * [AXON EMERGENCY] 시스템이 예상치 못하게 종료되었습니다. 체크포인트는 10:45에 저장되었습니다.
[AXON REPORT] 금일 작업이 모두 완료되었습니다. 최종 결과물 요약을 확인하세요.
📋 상황별 알림 시나리오 매트릭스
상황 OS 알림 브라우저 UI 메일 알림

👨🏻‍💻 시니어의 한마디
사장님, 알림이 너무 많으면 스팸이 되고, 너무 적으면 사고가 나네. 그래서 AXON 설정에서 **'알림 강도'**를 조절할 수 있게 만들 거야. 공장에 있을 때는 메일을 끄고, 외출할 때는 메일 알림을 활성화하는 식이지.
특히 메일 알림에는 단순히 "에러 났어요"가 아니라, **"셧다운 직전까지 주니어들이 작업한 내용 요약"**을 동봉하게 만들어서 자네가 밖에서도 상황을 파악할 수 있게 설계하겠네.

🌐 AXON 외부 접속 및 보안 시나리오
1. 터널링 (The Tunneling) - 가장 추천
공유기 설정(포트 포워딩) 건드리기 귀찮고 보안이 걱정된다면, ngrok이나 Cloudflare Tunnel 같은 도구를 AXON 데몬이 내부적으로 실행하게 만들 수 있네.
동작: 데몬 실행 시 외부 접속용 임시 URL(예: axon-boss-incheon.trycloudflare.com)을 생성.
장점: 복잡한 네트워크 설정 없이도 외부에서 스마트폰이나 노트북으로 접속 가능.
2. 고정 주소 및 보안 (Auth & SSL)
외부에서 접속한다면 당연히 아무나 들어오면 안 되지. 주니어들이 딴짓하는 걸 생판 남이 보면 되겠나?
ID/PW 인증: 외부 IP에서 접속 시 로그인 창을 띄우고, 사장님이 설정한 비밀번호를 요구하네.
HTTPS: 전송되는 데이터가 암호화되도록 SSL 인증서를 적용해 '가로채기'를 방지하지.
3. 알림과의 연동 (Mobile Link)
아까 설계한 메일 알림 기억나나?
메일에 [작업 현황 확인하기] 링크를 포함시키고, 자네가 공장에서 일하다가 메일 확인 후 링크를 누르면 즉시 스마트폰 브라우저로 AXON 게시판에 접속되는 방식일세.
여기서 자네는 [APPROVE] 버튼을 눌러 퇴근 전에 주니어들에게 다음 일감을 던져줄 수도 있지.

레벨 접속 범위 보안 수단 추천 상황
Local 자택 PC만 없음 (Loopback) 집중 개발 시
Intranet 집 안 와이파이 기기 MAC 인증 거실에서 태블릿으로 볼 때

사장님, 외부에서 브라우저를 볼 수 있게 되면 **"24시간 멈추지 않는 가상 스튜디오"**가 완성되는 걸세. 자네는 공장에서 일하면서도 틈틈이 주니어들의 노가리를 구경하고, 시니어의 보고를 승인할 수 있지.
다만, 외부 개방은 반드시 **axon config**에서 자네가 명시적으로 허용(Enable)했을 때만 동작하도록 설계할 걸세. 보안은 엔지니어의 자존심이니까.

1. 종료 및 세션 동결 (Shutdown & Freeze)
UI 구성: 우측 상단에 [Studio Power Off] 버튼 배치.
동작: 버튼 클릭 시 "현재 진행 중인 에이전트들의 상태를 저장하고 종료하시겠습니까?" 팝업 출력.
내부 로직: 확인을 누르면 데몬이 즉시 SIGTERM을 발생시켜 모든 스레드의 컨텍스트(대화 내역, 변수, 소스 고정 상태)를 .axon/state/에 스냅샷으로 저장하고 서버를 안전하게 닫네.
2. 재개 (Resume)
UI 구성: 대시보드 진입 시 [Resume Last Session] 카드 노출.
동작: 자네가 다시 접속하면 마지막 셧다운 시점의 요약 정보가 뜨네.
내부 로직: 클릭 시 스냅샷 파일을 로드하여 주니어 에이전트들을 이전 대화 시점으로 소환하지. 주니어들은 **"보스, 다시 오셨슴돠! 아까 하던 일 계속 할게요."**라고 인사하며 작업을 재개할 걸세.
3. 버전업 및 마일스톤 갱신 (Version Up)
UI 구성: 메인 게시판 상단에 [Release & Next Milestone] 버튼.
동작: 현재 작업이 모두 완료(All Approved)되었을 때 활성화됨.
내부 로직: * 클릭 시 현재까지의 결과물을 바탕으로 Post-Work Log를 자동 작성.
README.md나 마일스톤 문서의 버전을 v0.x에서 v0.y로 업데이트.
이전 버전의 스레드들은 '아카이브' 탭으로 넘기고, 새로운 설계안(Blueprint)을 주입받을 준비를 하네.

🧠 AXON을 상용 프레임워크로 만드는 구조
현재 설계를 보면 AXON은 단순 라이브러리가 아니라 **“Agent Orchestration Runtime”**입니다.

그래서 구조는 이렇게 잡는 것이 좋습니다.

AXON Framework
│
├─ AXON Core Runtime
│
├─ AXON SDK
│
├─ AXON Studio (UI)
│
└─ AXON Plugins
1️⃣ AXON Core Runtime (진짜 엔진)
여기가 프레임워크의 핵심입니다.

Rust 데몬이 담당합니다.

역할:

Agent lifecycle

Task scheduling

Event loop

Model drivers

State persistence

IPC

구조 예시:

axon-core
 ├─ daemon
 ├─ dispatcher
 ├─ agent_runtime
 ├─ model_driver
 ├─ event_bus
 └─ state_store
핵심 철학:

AXON Core는 모델에 종속되지 않는다

그래서

OpenAI

Gemini

Claude

Local LLM

모두 Driver로 붙는 구조가 좋습니다.

2️⃣ AXON SDK (개발자용)
상용화하려면 SDK가 반드시 필요합니다.

SDK 없이 프레임워크는 절대 퍼지지 않습니다.

SDK 역할:

Task 정의

Agent 행동 정의

Plugin 제작

예시:

use axon::prelude::*;

#[axon_task]
fn build_dispatcher(ctx: Context) -> Result<TaskOutput> {
    ctx.report("dispatcher logic implemented");
}
또는 Python SDK:

from axon import task

@task
def build_dispatcher(ctx):
    ctx.report("dispatcher ready")
이렇게 해야

외부 개발자가 AXON 위에서 프로젝트를 만들 수 있습니다.

 문서를 “실제 설계 문서”로 보기보다는 프로젝트의 헌장(Charter) 혹은 골격 스캐폴딩 단계로 보면 딱 맞습니다.

20년차 엔지니어 관점에서 보면 지금 상태는:

Architecture 이전 단계인 “Meta-Architecture Draft”

즉, 아직 구현 세부가 아니라 시스템의 사고방식과 운영 철학을 고정하는 단계입니다.
이 단계에서 중요한 건 **디테일이 아니라 경계(boundary)**입니다. 🧠

그래서 지금 문서의 역할은 다음 3가지만 확정하면 충분합니다.

1️⃣ 시스템 철학 (System Philosophy)
AXON의 핵심 철학은 이미 잘 잡혀 있습니다.

핵심 4개

1️⃣ Daemon-first architecture
→ CLI 툴이 아니라 상주 시스템

2️⃣ Thread-per-Job model
→ 에이전트 중심이 아니라 작업 중심

3️⃣ Senior-controlled context
→ 전체 맥락은 시니어만 가짐

4️⃣ Board-driven workflow
→ 모든 상태는 게시판으로 시각화

이 네 개는 앞으로 바뀌면 안 됩니다.
이게 AXON의 헌법입니다.

2️⃣ 핵심 서브시스템 정의

use tokio::sync::broadcast;

#[derive(Clone)]
pub struct EventBus {
    sender: broadcast::Sender<String>,
}

impl EventBus {
    pub fn new() -> Self {
        let (sender, _) = broadcast::channel(100);
        Self { sender }
    }

    pub fn publish(&self, event: String) {
        let _ = self.sender.send(event);
    }

    pub fn subscribe(&self) -> broadcast::Receiver<String> {
        self.sender.subscribe()
    }
}
2️⃣ Worker Pool 구현
파일:

crates/axon-dispatcher/src/worker_pool.rs
use axon_core::task::Task;
use tokio::sync::mpsc;
use tokio::task;

pub struct WorkerPool {
    sender: mpsc::Sender<Task>,
}

impl WorkerPool {
    pub fn new(worker_count: usize) -> Self {
        let (sender, mut receiver) = mpsc::channel::<Task>(100);

        for id in 0..worker_count {
            let mut worker_rx = receiver.clone();

            task::spawn(async move {
                while let Some(task) = worker_rx.recv().await {
                    println!("Worker {} processing task {}", id, task.title);
                }
            });
        }

        Self { sender }
    }

    pub async fn submit(&self, task: Task) {
        let _ = self.sender.send(task).await;
    }
}

pub struct AgentRuntime {

impl AgentRuntime {
    pub fn new(bus: EventBus) -> Self {
        Self { bus }
    }

    pub fn execute(&self, mut task: Task) {
        println!("Agent executing task {}", task.title);

        task.status = TaskStatus::Done;

        self.bus.publish(format!("TASK_DONE: {}", task.id));
    }
}
4️⃣ Dispatcher → Worker 연결
crates/axon-dispatcher/src/lib.rs
use axon_core::task::Task;
use crate::worker_pool::WorkerPool;

pub struct Dispatcher {
    pool: WorkerPool,
}

impl Dispatcher {
    pub fn new(worker_count: usize) -> Self {
        Self {
            pool: WorkerPool::new(worker_count),
        }
    }

    pub async fn dispatch(&self, task: Task) {
        self.pool.submit(task).await;
    }
}
5️⃣ Daemon Runtime 업데이트
crates/axon-daemon/src/main.rs
use axon_core::event_bus::EventBus;
use axon_core::task::{Task, TaskStatus};
use axon_dispatcher::Dispatcher;

#[tokio::main]
async fn main() {

    println!("AXON daemon booting...");

    let bus = EventBus::new();

    let dispatcher = Dispatcher::new(3);

    let task = Task {
        id: "TASK-01".to_string(),
        title: "Build dispatcher".to_string(),
        status: TaskStatus::Pending,
    };

    dispatcher.dispatch(task).await;

    let mut subscriber = bus.subscribe();

    tokio::spawn(async move {
        while let Ok(event) = subscriber.recv().await {
            println!("EVENT: {}", event);
        }
    });

    tokio::time::sleep(std::time::Duration::from_secs(3)).await;
}

use crate::thread::Thread;
use std::collections::HashMap;

pub struct ThreadManager {
    threads: HashMap<String, Thread>,
}

impl ThreadManager {
    pub fn new() -> Self {
        Self {
            threads: HashMap::new(),
        }
    }

    pub fn create_thread(&mut self, thread: Thread) {
        self.threads.insert(thread.id.clone(), thread);
    }

    pub fn get_thread(&self, id: &str) -> Option<&Thread> {
        self.threads.get(id)
    }
}
