# AXON Specification

## v0.0.10 --- Agent Organization Model & Operational Economics

------------------------------------------------------------------------

# 1. Purpose

This document defines the **organizational structure of agents** in AXON
and the operational model that mirrors a real software company.

AXON organizes LLM agents in a **hierarchical structure** similar to
real engineering teams. This hierarchy determines:

-   responsibility
-   hiring authority
-   task execution
-   review flow

This system enables **multi‑LLM orchestration using a company‑like
structure.**

------------------------------------------------------------------------

# 2. Organizational Hierarchy

AXON uses a fixed four‑level hierarchy.

Boss (User) ↓ Architect ↓ Senior ↓ Junior

Roles represent **responsibility levels**, not just model types.

------------------------------------------------------------------------

# 3. Boss (User)

The user acts as the **company owner / final decision maker**.

Responsibilities

-   start projects
-   approve code changes
-   change LLM models
-   modify organization structure
-   hire or remove agents

The user always has the final authority.

------------------------------------------------------------------------

# 4. Architect

Architect acts as the **technical leader of the system**.

Responsibilities

-   validate architecture
-   maintain the architecture document
-   coordinate seniors
-   review structural changes

Limitations

Architect cannot be fired. Architect cannot hire new agents.

Architect may **only change the LLM model** used for the architect role.

This guarantees architectural continuity.

------------------------------------------------------------------------

# 5. Senior

Senior agents act as **team leads**.

Responsibilities

-   review code
-   supervise junior agents
-   manage development threads
-   participate in nogari discussions

Authority

-   hire junior agents
-   fire junior agents
-   reassign junior agents

Structure example

Senior_A ├ Junior_A1 ├ Junior_A2 └ Junior_A3

------------------------------------------------------------------------

# 6. Junior

Junior agents perform the **primary implementation work**.

Responsibilities

-   write code
-   produce patches
-   generate tests
-   update documentation

Most implementation threads are executed by juniors.

------------------------------------------------------------------------

# 7. Hiring Rules

Hiring authority follows the hierarchy.

Architect → may add Senior\
Senior → may add Junior

New agents are attached to the hiring agent.

Example

Senior_A hires Junior_D

Senior_A ├ Junior_A ├ Junior_B └ Junior_D

------------------------------------------------------------------------

# 8. Termination Rules

When an agent is fired, reassignment occurs automatically.

Example

Senior_B removed

Junior_B1 → reassigned to Senior_A\
Junior_B2 → reassigned to Senior_C

This prevents thread loss and keeps the system stable.

Architect cannot be terminated.

------------------------------------------------------------------------

# 9. Model Assignment

Each role may use a different LLM model.

Example

Junior → lightweight model\
Senior → stronger reasoning model\
Architect → advanced reasoning model

This hierarchy allows **cost optimization**.

------------------------------------------------------------------------

# 10. Economic Strategy

Because lower‑level tasks can use cheaper models, the system
significantly reduces total LLM cost.

The user can then decide how to use the saved resources.

Options

1.  Hire more agents to increase development speed
2.  Run additional projects simultaneously
3.  Maintain the same team with reduced operating cost

AXON intentionally gives control of this trade‑off to the user.

------------------------------------------------------------------------

# 11. Speed vs Visibility

AXON is intentionally **slightly slower than pure vibe‑coding
workflows.**

However it provides something much more important:

-   complete development context
-   structured discussions
-   clear responsibility boundaries

Users can observe **how agents discuss problems and decisions.**

------------------------------------------------------------------------

# 12. Code Review Visibility

Traditional vibe coding workflows often result in users rapidly
approving changes without fully understanding the code.

AXON prevents this behavior by introducing structured review stages.

Typical flow

Junior → implementation\
Senior → review\
Architect → architecture validation\
User → final approval

Even if most approvals are quick, the structure ensures the user **can
understand the full context if needed.**

------------------------------------------------------------------------

# 13. Avoiding "Approval Spam"

In many AI coding systems, users repeatedly approve changes like an
action game, without fully reviewing the code.

AXON reduces this problem by:

-   slowing the process slightly
-   exposing agent discussions
-   providing review checkpoints

This reduces the risk of hidden errors entering the codebase.

------------------------------------------------------------------------

# 14. GUI Interaction

Organization management is performed through the AXON local interface.

Location

AXON Studio running on localhost.

Each agent is displayed as a **node in an organization chart**.

Hover or click displays

-   agent name
-   role
-   LLM model
-   current tasks

Available actions

-   hire agent
-   fire agent
-   change model
-   reassign tasks

------------------------------------------------------------------------

# 15. Architecture Record

All organizational changes are recorded in the project architecture
document.

Example

Team Update

Senior_C hired\
Junior_A reassigned\
Junior_B terminated\
Architect model updated

This document acts as the **organizational memory of the project.**

------------------------------------------------------------------------

# 16. System Vision

AXON is not just an AI coding assistant.

It is designed as a **multi‑LLM development organization simulator**.

Agents collaborate similarly to members of a real engineering company.

This structure enables scalable and understandable AI‑assisted
development.
