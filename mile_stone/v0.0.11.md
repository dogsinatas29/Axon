# AXON Specification

## v0.0.11.md

Event System Consolidation

------------------------------------------------------------------------

# 1. Purpose

This document consolidates the **event system concepts already defined
in v0.0.3.md**. The goal is to clearly identify the role of the Event
Bus inside AXON and prepare for the formal specification that will
follow in `axon-event-spec.md`.

AXON is fundamentally an **event‑driven runtime system**.

Almost every action inside the runtime produces an event.

------------------------------------------------------------------------

# 2. Event‑Driven Architecture

The AXON runtime operates using a central Event Bus.

Basic flow:

Agent ↓ Event Generated ↓ Event Bus ↓ Subscribers

Subscribers may include:

Scheduler\
Agent Runtime\
Nogari Listener\
UI Streaming\
Logging System\
Persistent Storage

This design allows all subsystems to react independently to runtime
events.

------------------------------------------------------------------------

# 3. Core Event Flow

The runtime flow described in v0.0.3.md can be summarized as:

Task Created ↓ Dispatcher Queue ↓ Worker Pool ↓ Agent Runtime ↓ Event
Generated ↓ Event Bus ↓ System Subscribers

The Event Bus therefore becomes the **central coordination layer of
AXON**.

------------------------------------------------------------------------

# 4. Example Event Types

The following events already appear in v0.0.3.md.

TaskCreated\
AgentAssigned\
AgentResponse\
ReviewRequested\
BossApproval

Additional events referenced in scenarios:

THREAD_CREATED\
MESSAGE_POSTED\
PATCH_PROPOSED\
THREAD_COMPLETED\
PATCH_APPROVED\
PATCH_REJECTED\
QUOTA_EXCEEDED

These events represent the operational lifecycle of threads and agents.

------------------------------------------------------------------------

# 5. Agent Listener Model

Agents react to events through listeners.

Example structure:

Junior agents listen for:

THREAD_ASSIGNED

Senior agents listen for:

JUNIOR_MESSAGE_POSTED

Architect agents listen for:

SENIOR_REVIEW_DONE

This creates a **reactive multi‑agent workflow**.

Event ↓ Agent Listener ↓ Agent Action ↓ New Event

------------------------------------------------------------------------

# 6. Nogari Event Integration

Nogari threads are also driven by the event system.

Example flow:

Thread Engine ↓ Event Occurs ↓ Event Bus ↓ Nogari Listener ↓ Persona
Reaction ↓ Nogari Message Posted

Nogari therefore functions as a **specialized event subscriber**.

------------------------------------------------------------------------

# 7. Event Logging

The event system can optionally persist logs.

Example structure:

.axon/ events/ event_001.json event_002.json

Example event:

{ "type": "patch_rejected", "thread": 21, "actor": "Senior" }

Event logs allow:

Runtime debugging\
Project replay\
Audit trails

------------------------------------------------------------------------

# 8. Event System Role

Within AXON architecture, the Event Bus acts as:

Runtime coordination layer

It connects:

Thread Engine\
Agent Runtime\
Scheduler\
UI\
Storage\
Nogari System

All major runtime components communicate through events.

------------------------------------------------------------------------

# 9. Relationship With Thread Model

Threads represent the **unit of work**, while events represent **state
changes**.

Thread lifecycle events may include:

THREAD_CREATED\
MESSAGE_POSTED\
ARTIFACT_CREATED\
APPROVAL_REQUESTED\
THREAD_COMPLETED

Threads store the state, while events describe the transitions.

------------------------------------------------------------------------

# 10. Current Specification Status

Based on v0.0.3.md, the following parts are already defined
conceptually:

Event Bus architecture\
Event‑driven runtime model\
Agent listener model\
Example event types\
Event logging concept\
Nogari integration

However, the **formal event schema is not yet defined**.

------------------------------------------------------------------------
