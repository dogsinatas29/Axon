
# v0.0.17 Validation & Interaction Layer

## 1. 문제 정의

“완전 공격형 검증”을 그대로 적용할 경우 시스템은 안정이 아니라 마비 상태에 빠진다.  
특히 주니어(Executor)는 반복적인 Reject 루프에 갇히며 실행이 진행되지 않는다.

핵심 문제:

검증 강도와 실행 가능성 사이의 균형이 없음

---

## 2. 해결 전략: Progressive Validation

### Stage A — Sanity Check
- 실행 가능성만 검증
- 명백한 오류만 차단

Reject 조건:
- 실행 불가능
- 치명적 상태 붕괴

---

### Stage B — Structural Check
- 상태 정합성 검증
- Idempotency / Snapshot 중심

Reject 조건:
- 재현 가능한 실패 시나리오 존재

---

### Stage C — Adversarial Check
- 실패 가정 기반 공격 검증
- Release 전 1회 수행

---

## 3. 주니어 보호 원칙

- 주니어는 Stage A만 통과하면 실행 가능
- Stage B/C 실패는 설계 문제로 간주
- 주니어는 설계 수정 금지

---

## 4. 실행 제어 규칙

### Reject 조건 제한
Reject ONLY if:
- 실행 불가능
- 데이터 손상 가능
- 무한 루프 가능

---

### 리뷰 출력 제한
- 최대 issue 3개
- 각 issue 5줄 이하

---

### 수정 범위 제한
- 지적된 부분만 수정 가능
- 나머지 코드 변경 금지

---

### 반복 제한
- 동일 Task 최대 3회 루프
- 초과 시 강제 종료

---

## 5. 운영 전략

- 설계 단계: 70% 정확도 허용
- 구현 단계: 빠른 실행
- 실행 단계: 실제 오류 기반 보정

---

## 6. 결론

완벽한 검증보다 중요한 것은 멈추지 않는 시스템이다.

Progressive Validation을 통해:
- 시스템 마비 방지
- 비용 통제
- 실행 지속성 확보
