# AXON v0.0.18 Conflict Resolution Loop Design

## 🎯 PURPOSE

Resolve 3-way merge conflicts using a controlled LLM retry loop.

------------------------------------------------------------------------

# 1. PROBLEM

When merge detects:

BASE != CURRENT\
BASE != NEW\
CURRENT != NEW

→ Conflict 발생

자동 병합 불가 → 명시적 해결 필요

------------------------------------------------------------------------

# 2. CORE STRATEGY

LLM을 "충돌 해결기"로 사용

단, 자유 생성이 아니라:

👉 **제약된 입력 + 강제 출력 계약**

------------------------------------------------------------------------

# 3. INPUT STRUCTURE (LLM)

``` text
[MERGE CONTEXT]

FILE: <filename>

[BASE]
<code>

[CURRENT]
<code>

[NEW]
<code>
```

------------------------------------------------------------------------

# 4. OUTPUT CONTRACT (STRICT)

``` text
[OUTPUT]

FILE: <filename>

```python
<merged_code>
```


    ❗ 규칙:
    - 하나의 파일만 출력
    - 설명 금지
    - 반드시 단일 코드 블록

    ---

    # 5. LOOP STRUCTURE

    ```python
    def resolve_conflict(file, base, current, new, max_retry=3):
        for attempt in range(max_retry):
            prompt = build_prompt(file, base, current, new)
            output = call_llm(prompt)

            parsed = parse_output(output)
            if not parsed:
                continue

            _, merged_code = parsed[0]

            if not validate_code(merged_code):
                continue

            return merged_code

        return None  # 실패

------------------------------------------------------------------------

# 6. VALIDATION RULES

LLM 결과는 반드시:

1.  compile 가능
2.  dependency 유지
3.  syntax 오류 없음

------------------------------------------------------------------------

# 7. FAILURE HANDLING

### 조건

-   파싱 실패
-   코드 invalid
-   빈 출력

→ 즉시 retry

------------------------------------------------------------------------

## MAX RETRY 초과 시

``` text
→ HARD FAIL
→ 상위 pipeline rollback
```

------------------------------------------------------------------------

# 8. RETRY PROMPT 강화

각 retry마다 추가 정보 제공:

### Attempt 1

기본 context

### Attempt 2

-   "이전 시도 실패" 명시

### Attempt 3

-   stricter instruction

------------------------------------------------------------------------

예:

``` text
Previous attempt failed.

You MUST produce valid Python code.
Do NOT omit any required logic.
```

------------------------------------------------------------------------

# 9. MERGE 정책

LLM이 따라야 할 기준:

1.  CURRENT 변경 유지 (사용자 수정 보존)
2.  NEW 변경 반영 (LLM 의도 유지)
3.  BASE는 기준선 (참고만)

------------------------------------------------------------------------

# 10. PIPELINE 통합

``` text
3-WAY MERGE
→ conflict 발생
    → LLM resolve loop
        → 성공 → merge 완료
        → 실패 → RETRY 전체 task
```

------------------------------------------------------------------------

# 11. RISKS

1.  LLM 반복 실패
2.  의미 손실 merge
3.  dependency 깨짐

------------------------------------------------------------------------

# 12. MITIGATION

-   retry 제한
-   strict validation
-   fail-fast

------------------------------------------------------------------------

# 13. RESULT

이 루프는:

-   자동 conflict 해결
-   상태 보존
-   인간 개입 최소화

👉 안정적인 self-healing merge 시스템 구축
