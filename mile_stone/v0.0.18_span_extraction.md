# AXON v0.0.18 Span Extraction Priority Design

## 🎯 PURPOSE

Define correct implementation order for semantic merge system.

------------------------------------------------------------------------

# 1. CONCLUSION

Span extraction must come BEFORE merge algorithm.

------------------------------------------------------------------------

## Correct Order

1.  Accurate span extraction\
2.  Safe replacement\
3.  Merge logic

------------------------------------------------------------------------

# 2. WHY EXTRACTION FIRST

Semantic merge depends on:

"Knowing exactly where to modify code"

Without accurate span:

-   Wrong region replaced
-   Structure corruption
-   Debugging impossible

------------------------------------------------------------------------

# 3. FAILURE CASES

## 3.1 Wrong Replacement

``` python
def a():
    pass

def b():
    pass
```

Incorrect span → both functions affected

------------------------------------------------------------------------

## 3.2 Decorator Loss

``` python
@cache
def func():
    pass
```

Decorator detached if span incorrect

------------------------------------------------------------------------

## 3.3 Nested Function Break

``` python
def outer():
    def inner():
        pass
```

Inner replacement breaks outer structure

------------------------------------------------------------------------

# 4. EXTRACTION REQUIREMENTS

## Must handle:

-   Function start (def line)
-   Decorators
-   Function end (indentation)
-   Nested structures
-   Class methods

------------------------------------------------------------------------

# 5. IMPLEMENTATION STRATEGY

## Step 1 --- AST-based span

``` python
node.lineno
node.end_lineno
```

------------------------------------------------------------------------

## Step 2 --- Decorator adjustment

``` python
node.decorator_list
```

Adjust start line if decorators exist

------------------------------------------------------------------------

## Step 3 --- Token refinement (optional)

-   Precise slicing
-   Comment-safe extraction

------------------------------------------------------------------------

# 6. REPLACEMENT MODEL

``` python
new_code = (
    original[:start]
    + replacement
    + original[end:]
)
```

------------------------------------------------------------------------

# 7. KEY INSIGHT

This phase is NOT about merging.

It is about:

👉 "precise code slicing"

------------------------------------------------------------------------

# 8. RISK OF WRONG ORDER

If merge is implemented first:

-   Replace boundaries incorrect
-   Hard-to-debug corruption
-   Exponential complexity

------------------------------------------------------------------------

# 9. FINAL PRINCIPLE

Correct merge is only possible AFTER correct extraction.

------------------------------------------------------------------------

# 🚀 RESULT

-   Safe function-level modification
-   Structure preserved
-   Reliable foundation for semantic merge
