## 🏛️ Sovereign Protocol - Global Intelligence Specification (v0.0.20)
**"모델은 생성하고, 시스템은 학습하며, 레지스트리는 증명한다."**

### 🧠 핵심 철학 (Core Philosophy)
- **Individual Error to Organizational Asset**: 개별 프로젝트의 시행착오를 조직 전체의 규칙 자산으로 치환.
- **Constraint-driven Quality**: 모델의 품질 편차를 견고한 제약 시스템(Constraint IR)으로 보정.
- **Adaptive Validation**: 정적 검증기에서 실제 실패 데이터를 기반으로 진화하는 적응형 검증기로의 전환.

### 📊 정량적 기대치 (Quantitative Expectations)
- **Fail Rate**: 동일 유형 오류의 재발 방지를 통해 전체 실패율 40~70% 감소 기대.
- **Cold Start**: 신규 프로젝트 투입 시 초기 실패 축적 기간의 획기적 단축.
- **Consistency**: 프로젝트 간 '정책 Drift' 제거 및 조직 단위의 일관된 품질 유지.

### 2. 지능의 3단계 계층 구조 (Memory Hierarchy)
1. **단기 기억 (Hot Rule Cache)**:
   - **Role**: 즉각적인 실수 교정 (Hint Layer).
   - **Specs**: In-memory LRU/Weight 기반 캐시, 1ms 미만 지연, 자동 망각(Decay) 로직 탑재.
2. **중기 기억 (Promotion Engine - The Trial)**:
   - **Role**: 검증된 실수를 로컬 법전으로 승격.
   - **Specs**: Candidate Pool, Scoring (Freq + Effectiveness), Shadow Validation (Trace Replay)을 통한 엄격한 검증.
3. **장기 기억 (Global IR Registry)**:
   - **Role**: 조직 단위의 공통 지능 공유 및 표준화.
   - **Specs**: SemVer(Semantic Versioning) 기반 버전 관리, `Global -> Profile -> Project Override` 계층적 해석.

## 🧠 상세 설계 및 로직
- **Failure Memory Injection**: 최근 실패 사례를 프롬프트에 동적으로 주입하여 "즉각적 학습" 효과 구현.
- **Trial-based Promotion**: Hot Cache의 힌트가 실제로 성공을 이끌어냈는지 증명(Shadow Test)된 후에만 IR로 승격.
- **Deterministic Materialization (v0.0.22 Integration)**: 
  - **Stage 3.5 Stub Generation**: 병렬 빌드 시의 의존성 충돌을 막기 위해 모든 설계 대상 파일을 빈 껍데기(Stub)로 미리 생성.
  - **Strict JSON Contract**: 요원의 출력을 순수 JSON 배열로 강제하여 파싱 불확실성을 제거하고 100% 신뢰도의 코드 추출 보장.

## ⚠️ 예외 처리 및 운영 제약 (1050TI Optimized)
- **Parallel Bottleneck**: 1050TI/Ollama 환경의 병렬 처리 한계(`NUM_PARALLEL: 1`)를 고려하여, 워커 수를 제한하거나 타임아웃을 넉넉히 설정하여 큐 대기 에러 방지.
- **Stub Dependency**: `main.py` 실행 시 다른 Stub 파일들이 존재하므로 `ImportError`는 방지되나, 미구현 함수 호출 시의 `AttributeError`는 단계적 완성 과정으로 수용.
