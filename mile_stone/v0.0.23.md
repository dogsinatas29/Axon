## 🧠 상세 설계 및 로직 (IR-First Architecture)
### 1. IR 매핑 및 규약 강제 (Stage 6.1)
- LLM 패치를 파일에 반영하기 전, **IR Contract** 준수 여부를 먼저 검증.
- `architecture.md`에 정의된 심볼(함수)이 모두 구현되었는지, 정의되지 않은 심볼이 추가되었는지 체크.

### 2. AXON Patch v2 + IR Transformer
- LLM 출력(Patch v2) -> IR Change -> Updated IR 순서로 변경 사항 전파.
- 파일 시뮬레이션은 업데이트된 IR을 바탕으로 수행.

### 3. 고도화된 Validator (Holistic Validation)
- Syntax 뿐만 아니라 **Interface Mismatch**, **Symbol Drift**를 IR 레벨에서 차단.

## 🛠️ 기술적 변경 사항
- **axon-core (IR Schema)**: `HashMap` 기반의 `ProjectIR`, `Component`, `Function` 구조로 전면 개편.
- **axon-core (IR Engine)**: `IRChange` 열거형 및 `apply_changes` 엔진 구현.
- **axon-core (Transformer)**: Patch v2를 `IRChange`로 변환하는 `patch_to_ir_changes` 구현.
- **axon-core (Validator v2)**: 컴포넌트 존재성, 심볼 규약, 의존성 정합성을 검증하는 IR 중심 Validator 구현.
- **axon-daemon**: Stage 6.1 `IR Mapping Validation` 단계 신설 및 `axon_ir_mapper.py` 연동 (Rust 통합 대기 중).
- **axon-agent**: 시스템 프롬프트에 `IR Contract Enforcement` 지침 주입.

## ⚠️ 예외 처리 및 주의 사항
- **Symbol Drift**: LLM이 유용한 헬퍼 함수를 추가하더라도 IR에 없으면 일단 'Drift'로 간주하여 반려하거나 IR 업데이트 요청 유도.
- **Minimal Parsing**: Rust Transformer는 현재 최소한의 정규식 패턴으로 함수를 추출하며, 복잡한 파싱은 향후 고도화 예정.

## 🏛️ M4: Physical Verification Pipeline (Optimistic Automation)
**"물리와 논리의 일치, 그리고 실물 환경에서의 최종 무결성 보장."**

### 1. 전략: 낙관적 자동화, 비관적 개입 (Optimistic Automation, Pessimistic Intervention)
- **기계 검증 95%**: 파일 경로, 권한, OS 환경, 의존성 해결 등 물리적 배포 단계에서 발생하는 모든 노가다성 검증은 `PHYSICAL_VALIDATE` 레이어에서 자동으로 수행.
- **시니어 개입 5%**: 물리 하네스가 성공하면 시니어를 방해하지 않고 즉시 완료(Finalize). 실패 시에만 시니어에게 "물리적 환경 부적합" 인터럽트를 전송하여 수동 개입 및 롤백 수행.

### 2. 파이프라인 확장
1. **AXON_PASS (Logical Approval)**: 주니어의 제안서가 로직 및 사양을 충족함을 시니어가 1차 승인.
2. **MATERIALIZE_FS (Physical Commit)**: 승인된 코드를 실제 프로젝트 디렉토리에 물리적으로 작성.
3. **PHYSICAL_VALIDATE (Final Harness)**: 작성된 실물을 대상으로 실제 런타임 환경 검증 실행.
4. **FINAL_GATE (Auto/Manual)**: 성공 시 자동 Lock-in, 실패 시 시니어에게 제어권 이관.

### 3. 기술적 방어선 (Bug Eradication)
- **Stub Detection**: `axon_patch_simulator.py` 레벨에서 `# AXON STUB` 문구 잔존 여부를 감시하여 게으른 구현을 원천 차단.

- **단순성**: 최대 거절 사유 3개 이내.

## 🏛️ M5: IR-based Execution Planner (Deterministic Scheduling)
**"LLM에게 순서를 맡기지 않는다. 시스템이 아키텍처의 의존성을 해석하고 실행 순서를 강제한다."**

### 1. IR → DAG → Execution Plan 레이어
- **DAG Builder**: `architecture.md`의 IR 정보를 바탕으로 컴포넌트 간 의존성 그래프를 생성.
- **Topological Sort**: 의존성 그래프를 분석하여 `input → validation → calculate → db`와 같은 안전한 생성 순서 도출.
- **Layered Parallelism**: 동일한 의존성 계층(Layer)에 있는 태스크들만 병렬 실행을 허용하여 수율과 안정성 확보.

### 2. Execution Planner 핵심 알고리즘 (Topological Execution)
- **Node Status**: `Pending → Ready → InProgress → Approved/Completed`.
- **Ready Criteria**: 모든 `dependencies` 노드가 `Approved` 상태일 때만 해당 노드를 `Ready`로 격상.
- **Atomic Loop**: `acquire_lock(file) → generate → physical_validate → unlock`.

### 3. Validation Hook (Stub 방지 최종 방어선)
- **Size Gate**: 파일 크기 < 200바이트 기각.
- **Content Gate**: "TODO", "Implementation pending" 잔존 시 기각.
- **Structure Gate**: 유효한 함수 정의(`fn `) 누락 시 기각.

### 4. Failure & Replan 전략
- **Retry Policy**: 검증 실패 시 최대 2회 재시도, 이후 시니어 인터럽트.
- **Dynamic Replan**: 의존성 노드 실패 시 하류(Downstream) 태스크들을 자동으로 보류(Hold)하고 실행 계획 재구성.
