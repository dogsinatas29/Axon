## 🧠 상세 설계 및 로직 (Phase 0 - 3가지)

### 1️⃣ Destructive Diff 차단 (`extract_actual_functions`)

`rewrite` 액션 시 기존 파일에 존재하던 함수가 누락되면 **즉시 REJECT**.

```rust
fn detect_function_removal(old: &str, new: &str) -> bool {
    let old_funcs = extract_actual_functions(old);
    let new_funcs = extract_actual_functions(new);
    old_funcs.iter().any(|f| !new_funcs.contains(f))
}
```

적용 위치: `PHASE A (Dry Run)` — 물리 파일 쓰기 전

### 2️⃣ Rewrite = Full file 강제

기존 Signature Matching(태스크 명세 기준)에 더해,  
**실제 파일 내용 기준**으로 기존 함수 보존 여부를 이중 검증.

실패 조건:
```
if rewrite && old_func_missing_in_new:
    REJECT immediately (before any disk write)
```

### 3️⃣ LLM 프롬프트에 기존 코드 주입 (Context Injection)

에이전트 호출 전, 물리 파일의 현재 내용을 프롬프트에 삽입.

```
You are rewriting the ENTIRE file.
You MUST preserve all existing functions.

Existing file:
<현재 코드 삽입>
```

→ "방어"가 아닌 **"유도"** 전략. 에이전트가 상태를 알게 함.

## ⚠️ 예외 처리 및 주의 사항

- `extract_actual_functions`는 Rust/Python 함수를 regex로 탐지하므로, 다른 언어(JS, Go 등)는 추후 확장 필요

### [Priority 3] Logic Validator 추가 (Next Phase)
- **비즈니스 정합성**: 단순 실행 여부를 넘어, 요구사항(spec.md)에 기술된 로직(예: 나이 계산식, 날짜 범위)이 정확한지 유닛 테스트 기반으로 검증.

### 4️⃣ Dependency-Aware Validation Policy

## 🔒 AXON v0.0.25 Concurrency & Integrity Hardening (Final)

### 1️⃣ File Lease (Pessimistic Locking)
- **동시성 충돌 방지**: `file_locks` 테이블을 통한 배타적 쓰기 권한 제어를 도입했습니다.
- **Lease 메커니즘**: 
    - 태스크 시작 시 워커가 해당 파일의 리스(Lease)를 획득합니다.
    - 다른 워커가 작업 중인 파일은 `LOCK_DENIED`로 차단되며, 태스크는 자동으로 대기열로 복귀(Requeue)합니다.
- **안전 장치**: Deadlock 방지를 위해 리스 만료(Lease Expiry) 시스템을 탑재하여 워커 비정상 종료 시에도 자동으로 락이 해제됩니다.

### 2️⃣ Version Gate (Optimistic Locking)
- **Stale Write 차단**: 작업 시작 시점의 파일 해시(`base_hash`)를 기록하고, 커밋 직전에 현재 해시와 대조합니다.
- **데이터 무결성**: 다른 워커가 먼저 커밋하여 파일이 변경된 경우, 늦게 끝난 워커의 작업 결과가 최신 코드를 덮어쓰지 못하도록 `STALE_WRITE` 에러를 발생시키고 롤백을 수행합니다.

### 3️⃣ Atomic Write Sandbox
- **물리 파일 보호**: 모든 쓰기 연산은 `Lock + Version` 검증을 통과해야만 SSOT로 승격됩니다.
- **자동 롤백**: 검증 실패 시 물리적 파일을 작업 전 상태로 즉시 복구합니다.

### 🔒 AXON v0.0.25 Multi-file Lock Set Hardening (Complete)
- **Lock Set (Multi-file)**: 단일 파일 락을 넘어, 의존성 관계에 있는 여러 파일을 하나의 원자적 단위로 잠급니다.
- **Deterministic Atomic Lock**: 락 획득 시 파일 경로를 정렬(`sort()`)하여 데드락을 방지하고, 모든 파일에 대한 락을 일괄 확보합니다.
- **Impact-aware Version Gate**: `HashMap<String, String>`을 사용하여 잠긴 모든 파일의 초기 상태를 캡처하고, 커밋 직전 전체 일관성을 검증합니다.
- **Global Scheduler Filtering**: 스케줄러가 태스크의 `lock_files` 전체를 확인하여 단 하나라도 잠겨있으면 디스패치를 선제적으로 차단합니다.


### 🔄 AXON v0.0.25 Conflict-to-Rework (C2R) Pipeline (Complete)
- **Conflict Eventification**: 충돌(StaleWrite, LockDenied)을 단순 실패가 아닌 정형 데이터(`conflict_events`)로 승격하여 박제합니다.
- **Impact-aware Rework**: 충돌 발생 시 `DepGraph`를 통해 영향 범위를 즉시 분석하고, 관련 파일을 모두 포함하는 고우선순위 Rework 태스크를 자동 생성합니다.
- **Fast-track Scheduling**: 스케줄러가 `REWORK` 태스크를 최우선으로 배치하여 시스템의 자가 복구 속도를 극대화했습니다.
- **Loop Control**: 동일 태스크에 대한 Rework가 3회를 초과할 경우 수동 개입을 요청하여 무한 루프를 방지합니다.

### 🔒 AXON v0.0.25 Lock Manager & Jittered Backoff (Complete)
- **Deterministic Multi-file Lock**: `Storage::acquire_lock_set`에서 파일 경로를 알파벳 순으로 정렬하여 다중 잠금을 시도함으로써 교착 상태(Deadlock)를 원천 차단합니다.
- **Jittered Backoff Strategy**: 락 충돌 발생 시 즉시 실패하지 않고, 재시도 횟수에 비례한 지터(Jitter) 포함 백오프(50ms ~ 1s) 후 재큐잉합니다. 이를 통해 리브락(Livelock)을 방지하고 부하를 자연스럽게 분산합니다.
- **1-hop Lock Expansion**: 수정 대상 파일뿐만 아니라 `DepGraph`를 통해 식별된 직접적인 하위 의존 파일(1-hop)을 잠금 범위에 포함하여, 병렬 작업 중 발생할 수 있는 의존성 깨짐을 사전에 방지합니다.

- **Hotspot Priority Scoring**: 리워크 횟수, 실패율, 의존성 영향도(Fan-in)를 기반으로 파일별 우선순위를 계산합니다. 병목 현상을 일으키는 'Hotspot' 파일을 우선적으로 스케줄링하여 프로젝트 전체 속도를 최적화합니다.
- **Task Coalescing**: 동일 파일에 대한 중복 리워크 요청을 자동으로 병합하여 큐 오염을 방지하고 최신 상태의 작업을 보장합니다.
- **Dependency-Aware Initialization**: 부트스트랩 시 `DepGraph`로부터 의존성 정보를 읽어와 초기 파일 우선순위를 지능적으로 설정합니다.


### 📈 AXON v0.0.25 Fan-in Based Intelligence Scoring (Complete)

- **SSOT Driven Priority**: 아키텍처 사양(IR)에 명시된 정적 의존성만을 활용하여 스케줄링의 결정론적(Deterministic) 무결성을 유지합니다.
