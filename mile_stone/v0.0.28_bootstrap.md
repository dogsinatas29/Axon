# v0.0.28_bootstrap.md

# Axon v0.0.28 — Bootstrap Execution Plan

## Purpose

This document defines the practical bootstrap sequence for implementing:

```text
Skeleton Hard Stop
+
Semantic Validator
```

inside the Axon pipeline.

The goal of v0.0.28 is not feature expansion.

The goal is:

```text
pipeline stabilization
```

through deterministic validation and corruption containment.

This bootstrap document converts the conceptual documents into an executable implementation roadmap.

Related documents:

- v0.0.28_problem.md
- v0.0.28_goal.md
- v0.0.28_validation_strategy.md
- v0.0.28_skeleton_hardstop_semantic_validator.md

---

# Core Bootstrap Principle

Current Axon behavior:

```text
Generate first
Validate later
```

v0.0.28 target behavior:

```text
Validate before propagation
```

No downstream stage should execute on corrupted IR.

---

# Immediate Mission

Prevent this pipeline:

```text
spec.md
→ invalid Skeleton IR
→ corrupted architecture.md
→ hallucinated task graph
→ stub cascade
```

from ever occurring again.

---

# Bootstrap Phase Overview

# Phase 1 — SkeletonGuard
# Phase 2 — Semantic Validator
# Phase 3 — HeaderGen Isolation
# Phase 4 — Language-Aware Constraints
# Phase 5 — Verification Harness

Implementation should proceed strictly in this order.

---

# Phase 1 — SkeletonGuard

# Objective

Create deterministic structural validation for Skeleton IR.

This is the first enforcement layer.

---

# Required Behavior

If Skeleton IR is invalid:

```text
ABORT PIPELINE IMMEDIATELY
```

No retries.

No fallback tasks.

No hallucinated continuation.

---

# Required Validators

## 1. Empty Component Check

Reject:

```python
if component_count == 0:
    reject()
```

---

## 2. Placeholder Leakage Check

Reject:

```python
".."
"TBD"
"placeholder"
"example"
```

Example:

```python
if component.path == "..":
    reject()
```

---

## 3. Empty Function Check

Reject:

```python
if len(component.functions) == 0:
    reject()
```

---

## 4. Invalid Path Check

Reject:

```text
- malformed paths
- unsupported extensions
- illegal filenames
```

---

## 5. Duplicate Symbol Check

Reject:

```text
- duplicate exports
- duplicate ownership
- duplicate component IDs
```

---

# Mandatory Result

Current behavior:

```text
FAIL → continue
```

Must become:

```text
FAIL → HARD STOP
```

This is the highest priority change in v0.0.28.

---

# Phase 2 — Semantic Validator

# Objective

Ensure IR remains semantically connected to the specification.

This prevents:

```text
semantic drift
```

and:

```text
generic fallback hallucination
```

---

# Stage 1 — Keyword Similarity

Compare:

```text
spec keywords
vs
IR keywords
```

---

# Example

Specification:

```text
calculator
validator
header
age
C
```

Generated IR:

```text
frontend
deployment
react
typescript
```

Result:

```text
semantic mismatch
→ reject
```

---

# Implementation Goal

Create a lightweight deterministic scoring layer.

Example:

```python
score = keyword_overlap(spec, ir)

if score < THRESHOLD:
    reject()
```

---

# Stage 2 — Language Constraint Validation

If specification declares:

```text
Language: C
```

then reject:

```text
.tsx
package.json
npm
Node.js
React
```

---

# Stage 3 — Artifact Expectation Validation

For C/C++ projects:

Require:

```text
.h
.c
```

pairs.

Reject if missing.

---

# Expected Result

The system should no longer produce:

```text
frontend.tsx
```

inside C projects.

---

# Phase 3 — HeaderGen Isolation

# Objective

Prevent invalid architecture.md from generating hallucinated task graphs.

---

# Required Rule

HeaderGen must refuse invalid input.

Example:

```python
if architecture.invalid():
    abort_task_generation()
```

---

# Current Problem

Observed behavior:

```text
Skeleton = FAILING
HeaderGen = HEALTHY
```

This must become impossible.

---

# Required Pipeline Rule

```text
Skeleton failure
=
global pipeline failure
```

No downstream stages may continue.

---

# Phase 4 — Language-Aware Constraints

# Objective

Introduce language-native IR awareness.

Current IR is too generic.

---

# Required C/C++ Concepts

The IR must understand:

```text
- translation units
- header/source ownership
- include topology
- exported symbols
- linker boundaries
```

---

# Example IR

```json
{
  "component_type": "translation_unit",
  "header": "database.h",
  "source": "database.c",
  "exports": [],
  "includes": []
}
```

---

# Why This Matters

This enables deterministic detection of:

```text
- orphan headers
- missing implementations
- unresolved symbols
- stub-only outputs
```
