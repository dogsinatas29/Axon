# Axon v0.0.28 — Skeleton IR Collapse & Fail-Open Pipeline Analysis

## Overview

During analysis of the `TEST2` project, a critical pipeline failure pattern was discovered in the Axon bootstrap process.

The observed issue was initially reported as:

- `.h` files are not being generated in C/C++ projects
- Stub-only outputs are appearing
- `architecture.md` is effectively empty

However, deeper inspection revealed that the actual root cause occurs **before HeaderGen**.

The real failure point is:

```text
spec.md
  ↓
Skeleton IR Generation   ← FAILURE
  ↓
architecture.md
  ↓
HeaderGen
  ↓
Task Decomposition
  ↓
.c / .h generation
```

---

# Observed State

## spec.md

The source specification correctly defined a C modular project:

```text
main.c
database.c/h
calculator.c/h
validators.c/h
```

The project itself was a simple age calculator system written in C.

---

Generated output:

```json
{
  "components": [
    {
      "id": "..",
      "name": "..",
      "path": "..",
      "functions": []
    }
  ],
  "constraints": []
}
```

This indicates total IR collapse.

The generated IR contains:

- placeholder leakage
- invalid component structure
- empty function graph
- invalid path data
- no language-aware decomposition

# Critical Discovery

The most important observation:

```text
Skeleton = Failing
HeaderGen = Healthy
```

This means the pipeline detected invalid IR generation but still continued execution.

This behavior creates a dangerous fail-open architecture.

Current observed behavior:

```text
IR generation failure
→ invalid architecture.md
→ HeaderGen continues
→ generic hallucinated tasks

→ stub cascade
```

Generated tasks included:

```text
Database Design
API Development
Frontend Development
Deployment
```

These tasks are completely unrelated to the original C project specification.

---

# Root Problems

# 1. Skeleton Validation Missing

Current behavior appears to validate only:

```text
"Did the model return JSON?"
```

Instead of validating semantic integrity.

Required validations:

```text
- component_count > 0
- path != ".."
- functions non-empty
- keyword overlap with spec
- language-specific artifact existence
```

For C/C++ projects specifically:

```text
- .h / .c pair validation
- include graph validation
- translation unit validation
```

must exist.

---

# 2. Fail-Open Pipeline Design

Current behavior:

```text
Skeleton FAIL
→ continue pipeline
```

Correct behavior:

```text
Skeleton FAIL
→ hard stop
```

Reason:

`architecture.md` is the root IR for all downstream stages.

If it becomes corrupted:

- task generation becomes hallucinated
- implementation becomes disconnected from spec
- validators lose semantic grounding

---

# 3. Generic Fallback Contamination

The generated tasks indicate the model escaped into generic software priors.

Observed behavior:

```text
software project
→ generic CRUD/web assumptions
→ hallucinated architecture
```

This commonly occurs during:

```text
- context collapse
- malformed prompts
- schema anchoring failure
- token truncation
- placeholder corruption
```

The `".."` output strongly suggests template leakage or placeholder contamination.

---

# 4. architecture.md Is Not Language-Aware

Current IR schema appears optimized for:

```text
Python
JavaScript
Generic app workflows
```

But lacks native representation for C/C++ concepts:

```text
- translation units
- header/source separation
- include dependencies
- exported symbols
- compilation boundaries
```

As a result:

```text
C modular decomposition fails
```

during Skeleton generation.

---

# Proposed Hardening

# Phase 1 — SkeletonGuard

Immediate validation layer:

```python
if component.path == "..":
    reject()

if component_count == 0:
    reject()

if language == "C":
    require_header_units()

if keyword_overlap(spec, ir) < threshold:
    reject()
```

And:

```text
Skeleton FAIL => ABORT PIPELINE
```

must become mandatory behavior.

---

# Phase 2 — Language-Aware IR

Suggested C/C++ IR schema:

```json
{
  "component_type": "translation_unit",
  "header": "database.h",
  "source": "database.c",
  "exports": [],
  "includes": []
}
```

This allows:

- dependency-aware generation
- proper include graph planning
- modular decomposition
- linker-aware reasoning

---

# Phase 3 — HeaderGen Isolation

HeaderGen must refuse invalid architecture input.

Example:

```python
if architecture.invalid():
    abort_task_generation()
```

Without this, hallucinated tasks contaminate downstream planning.
