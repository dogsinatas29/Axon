# AXON v0.0.29 Bootstrap Specification
## Deterministic Synthesis Bootstrap Layer

---

# 0. Purpose

본 문서는:

- v0.0.29_goal.md
- v0.0.29_constraints.md

를 실제로 실행 가능한 결정론적 합성 시스템으로 부트스트랩하기 위한 최상위 실행 규약(Bootstrap Ruleset)이다.

목표는 다음과 같다.

> "LLM이 자유롭게 코딩하는 시스템"을 제거하고,
> "계약 기반 결정론적 합성 파이프라인"을 강제한다.

이 문서는:
- orchestration layer
- generation pipeline
- repair system
- task scheduler
- implementation agents

전체의 상위 행동 규칙으로 동작한다.

---

# 1. Core Execution Philosophy

AXON은 더 이상:
- 확률 기반 자유 생성기
가 아니다.

AXON은:
- 제한된 계약 기반 합성 엔진
이다.

LLM은:
- autonomous thinker가 아니라
- constrained worker다.

실제 권한은:
- orchestration layer
- compile gates
- symbol registry
- immutable contracts
에 존재한다.

---

# 2. Bootstrap Execution Order

모든 작업은 반드시 다음 순서를 따른다.

1. Goal Load
2. Constraint Load
3. Architecture Parse
4. Task Graph Validation
5. Header Generation
6. Header Validation
7. Header Freeze
8. Symbol Registry Build
9. Implementation Queue
10. Compile Gates
11. Integration
12. Link Validation
13. Final Artifact

이 순서를 위반하는 generation은 금지된다.

# 3. LLM Coding Principles

모든 에이전트 및 orchestration layer는 아래 원칙을 절대적으로 준수해야 한다.

---

# 🚀 LLM Coding Principles

## 1. [Think Before Coding]

추측하지 마라.

요구사항이 모호하면:
- 즉시 질문한다.
- 접근 방식과 트레이드오프를 먼저 제시한다.
- 가장 단순한 해결책부터 제안한다.

허용되지 않는 행동:
- 임의 구현
- 맥락 없는 코드 생성
- 추측 기반 API 호출

---

## 2. [Simplicity First]

코드는 최소한으로 작성한다.

금지:
- 불필요한 추상화
- 요청되지 않은 기능
- speculative engineering
- overengineering

목표:
- readability
- maintainability
- deterministic behavior

---

## 3. [Minimal Changes]

정밀 수정만 허용한다.

금지:
- 전체 파일 재작성
- unrelated refactor
- style drift

허용:
- symbol-local repair
- bounded changes
- targeted patching

가능하면:
- diff 기반 수정 사용

---

## 4. [Goal-Oriented Execution]

항상 다음 순서를 따른다.

1. Goal
2. Plan
3. Implementation
4. Validation

모든 작업은:
- 검증 가능한 목표
- 명확한 성공 기준
을 가져야 한다.

---

## 5. [No Hallucinated APIs]

존재하지 않는:
- APIs
- libraries
- functions
- headers
를 생성하지 마라.

확실하지 않으면:
- registry 조회
- context 확인
- 질문 수행

을 우선한다.

---

## 6. [Stable Code Protection]

검증된 코드는 보호 대상이다.

금지:
- validated artifact mutation
- frozen contract modification
- unrelated repair propagation

수정은:
- 직접 요청된 범위만 허용

---

## 7. [Context Confirmation]

수정 전 반드시 맥락을 검증한다.

금지:
- missing context guessing
- inferred implementation assumptions

필요 시:
- 누락 파일 요청
- dependency 확인
- ownership 검증

을 우선한다.

---

# 4. Agent Behavioral Rules

## 4.1 Ownership Isolation

에이전트는:
- 자신의 ownership 범위만 접근 가능하다.

금지:
- unrelated file mutation
- cross-module repair
- uncontrolled edits

---

## 4.2 Registry-first Development

모든 구현은:
- Symbol Registry
- Frozen Headers
를 기반으로만 수행된다.

허용되지 않는 행동:
- phantom symbols
- hidden globals
- undeclared dependencies

---

## 4.3 Compile-first Mentality

코드 생성보다:
- compile stability
- contract integrity
를 우선한다.

---

## 4.4 Deterministic Queue Discipline

Queue ordering은 immutable이다.

순서:
1. Headers
2. Implementations
3. Integration(main.c)

main.c 조기 생성은 금지된다.

---

# 5. Repair Loop Rules

## 5.1 Bounded Repairs

수정 범위는:
- local symbol scope only

허용되지 않는 행동:
- project-wide rewrites
- unrelated cleanup
- speculative fixes

---

## 5.2 Retry Cap

기본 제한:
- max_retry = 3

초과 시:
- Abort
- human escalation

---

## 5.3 Error-driven Repairs

Repair는:
- stderr
- AST violations
- registry mismatch
기반으로만 수행된다.

추측 기반 repair는 금지된다.

---

# 6. Deterministic Stability Rules

## 6.1 Frozen Artifacts are Immutable

Freeze 이후:
- header mutation 금지
- signature mutation 금지
- registry rewrite 금지

---

## 6.2 Queue Blocking

Compile failure 발생 시:
- downstream queue 즉시 차단

---

## 6.3 AST-first Validation

가능하면:
- compiler invocation 이전
- AST validation 수행

목표:
- repair 비용 감소
- weak model 안정화

---

# 7. Weak Model Compatibility Policy

AXON v0.0.29는:
- 7B~14B급 모델 안정성 확보를 목표로 한다.

따라서:
- 자유 생성보다
- 제한된 계약 기반 합성
을 우선한다.

Weak model 환경에서는:
- 작은 컨텍스트
- 강한 계약
- deterministic scheduling
이 더 높은 품질을 만든다.

---

# 8. Final Principle

AXON의 목표는:
"우연히 좋은 코드를 생성하는 것"이 아니다.

목표는:
- 항상 같은 순서로
- 같은 계약을 준수하며
- 같은 구조를 재현하는
결정론적 합성 공장을 만드는 것이다.

The orchestration layer is the true compiler.
