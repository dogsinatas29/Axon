# AXON v0.0.29 Goal Specification

## Deterministic Synthesis Pipeline & Contract-Driven Build Architecture

---

# 0. Objective

AXON v0.0.29의 핵심 목표는 다음과 같다.

> "가끔 성공하는 확률론적 코드 생성기"를 폐기하고,
> "항상 동일한 품질을 재현 가능한 결정론적 코드 합성 공장"으로 전환한다.

현재 시스템의 주요 실패 원인은 LLM 자체의 품질 문제가 아니라 다음과 같은 오케스트레이션 붕괴에 있다.

* dependency-blind generation
* implementation task omission
* signature drift
* hallucinated includes
* header/implementation boundary collapse
* uncontrolled context sharing
* missing compile gates

v0.0.29에서는 이를 해결하기 위해:

* 중앙 계약 레지스트리(Symbol Authority)
* Header Freeze
* Dependency-aware Ordering
* Compile-success Gating
* AST-level Validation

을 핵심 인프라로 도입한다.

---

# 1. Primary Design Philosophy

## 1.1 Determinism over Creativity

AXON은 더 이상 "창의적인 코드 생성기"가 아니다.

목표는:

* 자유 생성이 아니라
* 제한된 계약 기반 합성이다.

Weak model일수록:

* 작은 컨텍스트
* 강한 계약
* 제한된 생성 범위
  가 더 높은 안정성을 만든다.

---

## 1.2 Contracts are Immutable

한 번 생성 및 검증된 인터페이스는 수정 불가능하다.

특히:

* function signatures
* public symbols
* include ownership
* extern declarations

은 중앙 레지스트리에 의해 관리된다.

Implementation 단계는 절대 계약을 수정할 수 없다.

---

## 1.3 Compilation is a First-Class Gate

컴파일 실패는 downstream symptom이 아니라 상부 빌드 게이트다.

모든 단계는:

* compile success
* contract integrity
* symbol validity

를 만족해야 다음 큐로 진입 가능하다.

---

# 2. Deterministic 11-Stage Pipeline

## Stage 1 — Architecture IR Parsing

Input:

* architecture.md

Action:

* architecture.md를 deterministic IR 구조체로 파싱
* component DAG 생성
* ownership metadata 생성

Outputs:

* Task Graph
* Symbol Expectations
* Include DAG

---

## Stage 2 — Task Graph Validation

목표:

* ImplGen 시작 전 전체 구조 무결성 검증

검증 항목:

* expected task count
* dependency DAG
* include DAG cycles
* orphan components
* unresolved ownership

실패 시:

* 즉시 Abort
* 하위 generation queue 차단

---

## Stage 3 — Header Generation

목표:

* 모든 .h 인터페이스 선행 생성

생성 대상:

* public APIs
* structs
* typedefs
* extern declarations

규칙:

* implementation 금지
* declaration only

---

## Stage 4 — Header Compile Check

목표:

* 헤더 자체의 독립 무결성 검증

검증 항목:

* syntax validation
* duplicate definitions
* include consistency
* header guards
* cyclic include detection

실패 시:

* Freeze 금지
* Repair Queue 진입

---

## Stage 5 — Header Freeze

목표:

* 검증 완료된 인터페이스를 Immutable 상태로 고정

효과:

* 이후 단계에서 수정 불가
* signature drift 차단
* cascading mutation 차단

---

## Stage 6 — Symbol Registry Build

목표:

* 중앙 Symbol Authority 구축

Registry 포함 항목:

* public functions
* structs
* typedefs
* extern globals
* ownership metadata
* include authority

규칙:

* whitelist 기반 접근만 허용
* registry 외 symbol 생성 금지

---

## Stage 7 — Dependency-aware Implementation Queue

목표:

* generation ordering 강제

순서:

1. Headers
2. Implementations
3. main.c

규칙:

* dependency satisfied 상태만 queue 승격
* unresolved symbol 상태에서는 generation 금지

---

## Stage 8 — Per-file Compile Gate

목표:

* 구현체 생성 즉시 개별 검증 수행

검증 항목:

* syntax compile
* symbol existence
* illegal includes
* ownership violations
* linker-local validation

실패 시:

* Repair Loop 진입
* 다음 queue block

---

## Stage 9 — Integration (main.c)

목표:

* 최종 엔트리포인트 합성

규칙:

* registry 기반 symbol 참조만 허용
* hallucinated functions 금지
* dependency-blind generation 금지

main.c는:

* 가장 마지막에 생성된다.

---

## Stage 10 — Link Gate

목표:

* 전체 object linking 검증

검증 항목:

* unresolved references
* duplicate symbols
* missing implementations
* ABI mismatch

---

## Stage 11 — Final Artifact

목표:

* deterministic executable artifact 생성

출력:

* binary
* logs
* build metadata
* registry snapshot

---

# 3. Core Guardrails

## 3.1 Symbol Authority

중앙 계약 레이어.

효과:

* hallucinated APIs 차단
* signature drift 차단
* phantom includes 차단

---

## 3.2 Header Freeze

인터페이스 고정 레이어.

효과:

* cascading mutation 차단
* implementation-side contract rewrite 금지

---

## 3.3 Context Isolation

각 agent는:

* 자신의 ownership 범위만 본다.

허용:

* assigned header
* symbol registry

금지:

* unrelated project-wide mutation

---

## 3.4 Compile-success Gating

컴파일 성공 이전에는:

* 다음 단계 진입 금지

---

## 3.5 AST-level Validation

컴파일 이전 lightweight validation 수행.

탐지 대상:

* unknown symbols
* illegal includes
* forbidden globals
* duplicate definitions
* invalid ownership

목표:

* expensive compiler invocation 감소

---

## 3.6 Repair Loop (Bounded)

최대 Retry:

* 3회

수정 제한:

* symbol-local scope only
* bounded line modification

목표:

* repair hallucination cascade 방지

---

# 4. Semantic Garbage Prevention

현재 가장 위험한 failure mode는:

> "Compile succeeds, but implementation is meaningless."

예:

* return 0;
* TODO stubs
* Hello World placeholders
* empty implementations

이를 방지하기 위해:

* lightweight semantic gates 추가

---

## Implementation Completeness Gate

탐지 항목:

* TODO markers
* empty function bodies
* placeholder strings
* constant-only fake logic
* unreachable implementations

목표:

* deterministic garbage 차단

---

# 5. Explicit Non-Goals

v0.0.29는 다음을 목표로 하지 않는다.

* AGI-level autonomous coding
* perfect semantic reasoning
* infinite self-healing loops
* unrestricted creativity

우선 목표는:

1. deterministic generation
2. compile stability
3. contract integrity
4. reproducible orchestration

이다.

---

# 6. Success Criteria

v0.0.29 성공 조건:

## Required

* no missing implementation files
* no hallucinated includes
* no signature drift
* deterministic generation ordering
* compile-success gated pipeline
* successful final linking

## Stretch Goals

* semantic completeness validation
* AST-level repair
* reproducible build snapshots

---

# 7. Final Principle

AXON의 목표는:
"좋은 코드를 우연히 생성하는 시스템"이 아니다.

목표는:

* 제약된 환경 안에서
* 항상 동일한 결과를 재현하며
* 인터페이스를 절대 배신하지 않는
* 결정론적 코드 합성 공장이다.

LLM은 자유 생성기가 아니라:

* 제한된 계약을 실행하는 worker로 취급된다.

The orchestration layer is the true intelligence.
