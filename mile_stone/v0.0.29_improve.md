# AXON v0.0.29 Improve Plan
## Deterministic Orchestration Refactor

---

# Overview

v0.0.29 validation hardening exposed a deeper architectural issue:

The primary instability is no longer code validation quality.

The real failure point is probabilistic orchestration.

Current AXON behavior still allows the LLM to implicitly decide:

- which implementation tasks exist
- which files should be materialized
- task ordering
- implementation completeness

This caused:

- missing implementation tasks
- IntegratorGen premature execution
- materialization timeouts
- placeholder fallback generation
- artifact mismatch failures

---

# Root Cause

Observed runtime flow:

```text
IR_MAP:
- include/calculator.h
- src/calculator.c
- include/database.h
- src/database.c
- include/validators.h
- src/validators.c
- src/main.c
```

However:

```text
HeaderGen:
✓ calculator.h
✓ database.h
✓ validators.h

ImplGen:
✗ calculator.c missing
✗ database.c missing
✗ validators.c missing

IntegratorGen:
✓ main.c only
```

Result:

```text
Build:
❌ MATERIALIZATION_TIMEOUT
❌ Missing artifact: src/calculator.c
```

The IR correctly contained all required modules.

The scheduler failed to materialize implementation tasks.

This proves:

- IR generation is mostly functional
- validation gates are functional
- orchestration is the unstable layer

---

# Strategic Shift

Old philosophy:

```text
Detect bad generation after it happens
```

New philosophy:

```text
Remove the freedom to generate incorrectly
```

This changes AXON from:

```text
LLM-driven orchestration
```

to:

```text
Daemon-driven deterministic execution
```

---

# New Architecture

## Previous Flow (Agent-dependent)

```text
PROJECT_START
    ↓
IR Generation
    ↓
ImplGen asks LLM to create implementation tasks
    ↓
LLM may omit tasks
    ↓
IntegratorGen starts prematurely
    ↓
Timeout / missing artifacts
```

---

## New Flow (Daemon-enforced)

```text
PROJECT_START
    ↓
IR Generation
    ↓
Deterministic required_modules extraction
    ↓
Task Injection Interceptor
    ↓
Hard implementation tasks inserted directly into queue
    ↓
LLM only performs execution work
    ↓
IntegratorGen waits for completion
```

---

# Core Principle

The LLM must NEVER decide:

- what files should exist
- what tasks are required
- what order tasks should execute

The LLM should ONLY decide:

- implementation details
- internal function logic
- local code structure

---

# Required Runtime Changes

## 1. Deterministic IR Output

IR stage must explicitly produce:

```json
{
  "required_modules": [
    "calculator",
    "database",
    "validators"
  ]
}
```

This becomes the authoritative execution contract.

---

## 2. Task Injection Interceptor

Before ImplGen execution:

The daemon runtime must iterate through required_modules.

Example:

```rust
Task::new(
    TaskType::GenerateCFile,
    "src/calculator.c"
)
```

Tasks are inserted directly into the runtime queue.

This bypasses LLM planning entirely.

---

## 3. LLM Worker Simplification

Workers become deterministic executors.

Old role:

```text
Planner + Generator
```

New role:

```text
Generator only
```

Worker responsibility becomes:

```text
"Implement src/calculator.c"
```

instead of:

```text
"Figure out what should exist"
```

---

# Required Gates

## Hard Task Completeness Gate

Before IntegratorGen:

```rust
expected_tasks == completed_tasks
```

Example:

```text
expected:
- calculator.c
- database.c
- validators.c

completed:
- calculator.c
- validators.c
```

Result:

```text
FAIL
```

IntegratorGen MUST NOT execute.
