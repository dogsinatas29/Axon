# 🚀 AXON v0.0.3 - The Automated Software Industry Scenario

"코딩은 기계에게 맡기고, 인간은 오직 전략적 설계(Architecting)에만 집중하는 시대의 개막."

---

## 🏗️ 1. 최초 설치 및 환경 구성 (The Setup)

AXON 공장을 기동하기 위해 사전에 준비해야 할 요소들과 설치 과정이네.

### 🛠️ 필수 준비물
- **Rust Toolchain**: `cargo`, `rustc` (최신 안정 버전). 이제 지능을 가진 에이전트들이 이를 분석하기 시작할 걸세.

- **Multi-LLM API Keys**: 사용하고자 하는 모든 서비스의 API 키를 환경변수로 등록하게. (예: `GEMINI_API_KEY`, `ANTHROPIC_API_KEY`, `OPENAI_API_KEY` 등)
- **네트워크 접근**: 각 LLM 공급자의 API 엔드포인트에 접속 가능한 상태여야 하네.

### 📥 공장 가동 및 지능 주입 (Interface & Setup)
자네는 **터미널(Terminal)**의 신속함과 **대시보드(Dashboard)**의 직관성 중 원하는 방식을 선택할 수 있네. 이미 작업중이던 프로젝트이건 최초로 사용할 프로젝트이건 axon run을 실행한다. 최초 작업이라면 아래 과정을 거친다. 

1. **지능형 터미널 가동 (CLI Bootstrapping)**:
   - `axon run` 실행 시 API 키가 없다면, 터미널이 직접 **"사장님, [Model]의 API 키가 없습니다. 입력해주십시오."**라고 요청하네. API 키는 bash.rc 파일이나 zshrc파일에서 읽어들인다. 자체 모델을 사용한다면 이 자체모델 역시 API 방식인가는 모르겠다. 
   - API 키는 동시에 여러개를 수집할 수 있다. 즉 기록된 API 키의 종류만큼 LLM을 사용할 수 있다. 
   - 키를 입력하는 즉시 데몬이 깨어나 지능을 수용할 준비를 마치지.
2. **지능형 대시보드 (HR Marketplace)**:
   - API 키 입력 단이 끝나면 인원 배정에 들어간다. 가장 먼저 아키텍트를 임명해 달라는 말을 한다. 아키텍트는 수가 고정되어 있으며, 여기서 API에 입력된 LLM의 종류가 나오고 이 중 하나를 선택한다. 
   * 예 : 
   	아키텍트를 임명해주십시오 
   	1. GEMINI
   	2. Claude
   	3. ChatGPT
   	4. 자체 모델
   - **아키텍트(Architect/CTO) 선임**: 가장 먼저 전체 설계를 책임질 '아키텍트' 1명을 선임하세. 이들은 **TOT(Tree of Thoughts)** 방식을 사용하여 전체 아키텍처의 일관성을 유지하고 거시적인 변경을 관리하네. (최상의 지능 권장) 

   - **시니어(Seniors) 배치**: 모듈별 검토와 지도를 맡을 시니어들을 여럿 충원할 수 있네. 이들 역시 **TOT**를 사용하여 주니어들이 제출한 결과물을 다각도로 검수하고 올바른 방향으로 이끌지.
   - 시니어는 몇 명을 투입하시겠습니까? ( 숫자를 입력받는다 ) 
   * 예 : 
   	시니어는 몇명을 투입하시겠습니까? 
          (1,2,3...)
          
       시니어가 사용할 LLM을 선택해주십시오. 
   	1. GEMINI
   	2. Claude
   	3. ChatGPT
   - **주니어(Juniors) 투입**: 실제 코딩 삽질을 수행할 주니어들을 쇼핑하듯 다중 선택하세. 이들은 **COT(Chain of Thought)** 방식을 사용하여 자신에게 할당된 파트의 로직을 단계별로 해결해 나간다네.
   - API 키가 등록되지 않은 모델은 '잠금(Lock)' 표시가 뜨며, 클릭 시 즉시 키를 입력할 수 있는 팝업이 뜬다네.
   - 주니어 숫자 및 LLM의 종류 역시 위의 시니어 인터페이스와 동일하다. 
3. **명세 하달 (The Order)**: 팀이 꾸려졌다면 이제 `v0.x.x.md` 명세서를 투입하세. 이제 지능을 가진 에이전트들이 이를 분석하기 시작할 걸세. 명세가 없으면 넘어가지 않는다. 
   - 예
    명세에 사용할 문서 이름을 입력해주십시오. 
   ---
이 과정이 끝나면 프로젝트의 구성을 보여준다. 사원수와 종류, 그리고 페르소나, 명세서의 이름 등등.. 

## 💂 2. 데몬 서비스 기동 및 오케스트레이션 설정

최초 실행되는 프로젝트라면 `axon run`을 입력하면 공장이 가동되네. 이때 사장(User)은 원활한 공정 제어를 위해 초기 설정을 점검해야 하네.

### ⚙️ 초기 오케스트레이션 설정 (Initial Orchestration)
`axon run`으로 데몬을 띄운 후, 사장(User)은 다음 명령어들로 공장의 인력과 지능을 배치해야 하네. **(멀티 LLM 환경을 구축하는 핵심 단계라네)**

1. **지능형 모델 매칭 및 회사 구성 (Role & Headcount Assignment)**:
   - **설정 순서**: 사용할 LLM 종류를 결정한 직후, 회사의 **총 사원수**와 그 내부 **구성비(CTO, 팀장, 사원)**를 각각 할당하게.
   - `axon architect google:gemini-1.5-pro`: 전체 설계를 총괄할 CTO 1명을 선임하게.
   - `axon seniors 2 anthropic:claude-3-5-sonnet`: 모듈별 팀장을 맡을 시니어들의 수를 결정하게.
   - `axon juniors 5 google:gemini-1.5-flash`: 실제 코딩을 수행할 주니어(사원)들의 인원수를 할당하게.

- **유연한 팀 구성 (Flexible Company Structure)**:
  - AXON은 고정된 조직도가 아닌, 프로젝트 요구사항에 맞춰 **사원수와 직급 구성을 즉시 재정의**할 수 있는 유연한 구조를 지향하네.
  - **소수정예형**: 아키텍트 1, 시니어 1, 주니어 10 (한 명의 팀장이 대규모 인원을 관리)
  - **관리강화형**: 아키텍트 1, 시니어 5, 주니어 1 (정교한 검수가 필요한 보안 프로젝트)
  - **자율주행형**: 아키텍트 1, 시니어 0, 주니어 3 (소규모 프로젝트의 고속 개발)

- **명령어 예시**:
  ```bash
  # 아키텍트(CTO) 배정
  axon architect google:gemini-1.5-pro
  
  # 시니어(팀장) 2명 배정
  axon seniors 2 anthropic:claude-3-5-sonnet
  
  # 주니어(사원) 5명 배정
  axon juniors 5 google:gemini-1.5-flash
  ```

### 🎭 각 사원의 개성 부여 (Individual Persona Infusion)
단순히 "Gemini-1", "Claude-2"로 불리는 딱딱한 기계가 아니네. AXON의 **페르소나 엔진(Affix System)**은 채용되는 모든 사원에게 독특한 아이덴티티를 부여하지.

- **성별 포함 (Gender Attribution)**: 이제 사원들의 페르소나에는 **성별(Gender)** 정보가 기본으로 포함되네. 이는 아키텍트, 시니어, 주니어 모두에게 적용되며, 이들의 대화 스타일이나 행동 양식에 미묘한 개성을 더해줄 걸세.
- **자동 생성 및 배정 (Automatic Assignment)**: 초기 세팅(`axon architect/seniors/juniors`) 시, 모든 사원에게는 **성별과 성격이 포함된 페르소나가 즉시 자동으로 부여**되네. 사장님이 일일이 설정할 번거로움 없이, 채용과 동시에 개성 있는 팀이 꾸구려지는 구조지.
- **수동 개성 편집 (Manual Override)**:
  - "이 사원은 내가 직접 면접 보고 성격을 정하고 싶네!" -> 대시보드 **[Office]** 메뉴에서 특정 사원의 페르소나를 직접 수정(Edit)할 수 있네.
  - 사장님이 직접 이름, 상세 설명, 주요 성격 태그를 입력하여 '세상에 단 하나뿐인 사원'을 만들 수 있지.
- **액손 마켓 (AXON Market - Persona Packs)**:
  - **공유와 확장**: 본인이 만든 독특한 성격 조합(Persona Pack)을 파일(`.persona`)로 내보내거나, 마켓에서 다른 유저들이 올린 '냉소적인 천재 팩', '커피 중독 개발자 팩' 등을 다운로드하여 즉시 사원들에게 주입할 수 있네.
  - 마켓에서 받은 페르소나 팩을 `axon import-persona [pack-name]` 명령어로 로드하면, 채용 시 해당 인격들이 우선순위로 배정된다네.
- **확인**: 대시보드의 **[Office]** 메뉴에서 자네가 고용한 사원들의 얼굴(아이콘)과 상세한 페르소나 시트를 한눈에 확인할 수 있다네.

> [!IMPORTANT]
> **API 키 우선 확인**: 오케스트레이션 명령 실행 전, 해당 모델 공급자의 API 키가 환경변수에 정확히 로드되었는지 반드시 확인하게. 지능을 사오려면 돈(Key)이 먼저 있어야 하니까 말일세.

> [!TIP]
> **지능 배치 및 추론 전략 (Brain & Reasoning Strategy)**:
> - **아키텍트 (CTO/Architect)**: **TOT (Tree of Thoughts)** 적용. 전체 시스템의 뼈대를 잡고 거시적 설계를 관리하네.
> - **시니어 (Senior)**: **TOT (Tree of Thoughts)** 적용. 주니어들의 작업을 다각도로 검토하고 설계 규약을 수호하네.
> - **주니어 (Junior)**: **COT (Chain of Thought)** 적용. 할당된 기능 구현에 집중하며 단계별 해결책을 도출하네.

### 📡 `axon run` 실행 시 내부 프로세스
1. **SSOT 동기화**: `architecture.md` 파일을 읽어 전체 시스템의 규약과 목표를 파악하네.
2. **Persistence 초기화**: `axon.db`를 연결하고, 이전 작업 내역이 있다면 컨텍스트를 복구하지.
3. **Core Hub 활성화**: `EventBus`와 `Dispatcher`를 기동하여 에이전트들의 통신망을 구축하세.
4. **Interface 개설**:
   - `localhost:8080`: REST API 및 WebSocket 서버 오픈 (Studio UI 연결용)
   - `Watcher`: 소스 파일 및 설계 문서의 변경 사항을 실시간 감시 시작.

## 🔄 3. 공장 가동 시나리오 (The Production Loop)

실제 작업이 어떻게 진행되는지 단계별로 살펴보세.

### 📋 Phase 1: 전략 및 명세 하달 (Spec Declaration)
사장님(자네)은 인력 배치가 완료된 후, **AXON Studio (Boss Board)** 혹은 `mile_stone/` 폴더를 통해 `v0.x.x.md` 명세서를 투입하네.
- **Boss의 역할**: "이번 버전에서는 어떤 가치를 창출하겠다"는 의지만 명확히 전달하면 되네. 나머지는 지능들이 알아서 할 걸세.
- **Architect (CTO)의 분석**: 최상위 지력인 **아키텍트 에이전트**가 이 명세서를 즉시 낚아채서 분석을 시작하네. 이들은 **TOT(Tree of Thoughts)**를 동원하여 시스템 전체 구조와 이번 명세의 충돌 여부를 검토하지.

### ⚔️ Phase 2: 지능형 아키텍처 업데이트 및 태스크 분해 (Intelligent Architecting)
아키텍트(CTO)는 단순히 텍스트를 나누는 것이 아니라, **LLM의 시맨틱 분석 실력**을 발휘하여 다음 공정을 수행하네.

1. **SSOT 분석 및 동기화 (SSOT Sync)**:
   - `axon read` 명령 시, 아키텍트가 명세를 정밀 분석하여 `architecture.md`를 업데이트하네.
   - 이 과정에서 아키텍처는 **[Draft]** 상태로 관리되며, 사장의 최종 승인을 거쳐 **[✅ Locked]** 상태로 승격(Promotion)되네.

2. **규항 기반 태스크 슬라이싱 (Rule-based Slicing)**:
   - 아키텍트는 `v0.0.3.md`에 명시된 **[파싱 규칙]**을 엄격히 준수하여 태스크를 분해하네.
   - **원자적 분해**: 주니어가 한 번에 처리 가능한 최소 단위로 쪼개기.
   - **4대 계층 분류**: Infra, Orchestration, Interaction, Persona 중 하나를 반드시 지정.
   - **의존성 맵핑**: 작업 간의 선후 관계를 정의하여 병렬 처리 효율 극대화.

3. **작업 할당 및 파견 (Dispatching)**:
   - 분해된 태스크는 **JSON 형태**로 데몬에 전달되며, 데몬은 이를 `axon.db`에 적재하고 Dispatcher를 통해 가용한 시니어/주니어에게 실시간으로 배분하네.
   - 에이전트들은 할당된 태스크의 텍스트와 아키텍트가 정의한 **[공통 규약(Protocol)]**만 읽고 즉각 '배틀(Battle)'에 돌입하지.

이 과정은 로컬 호스트를 통한 게시판형 대쉬보드에서 실시간으로 보여지게 된다.  화면 구성은 다음과 같다. 
좌측 메뉴                  우측 디스플레이
1. 작업 게시판                명세에서 작성된 작업 내용이 파싱된 작업의 세부 내역들이 스레드 형태로 나타난다
                           주니어는 COT를 통해 작업의 내용을 확인하고 사고해서 그 결과를 각 스레드에 댓글로 단다. 
                           시니어는 TOT를 사용해서 시니어의 작업 결과물을 검토하고 그것을 승인 혹은 리젝시킨다. 이 회수가 3회 ( 설정 가능 ) 이상 길어지면 시니어는 사장을 호출한다. 사장을 호출한 스레드는 스레드 제목 앞에 사장 호출 아이콘이 붙는다. 
                           만약 승인된 내용이 전체 구조를 변경할 내용이라면 아키텍트로 임명된 LLM이 등판해서 이것을 확인하고 리젝하거나 사장 승인이 있는 경우 구조를 변경한다. 사장 승인없이 시니어가 승인한 내용이라면 리젝하게 되면 다시 해당 작업 스레드로 구조가 틀리니 다시 작업하라는 아키텍트의 댓글이 달리고 작업이 활성화된다. 
                           시니어나 사장의 승인이 있기 전 까지는 파일에 직접 손을 대지 않는다. 즉, 승인 절차가 있어야 파일 편집으로 코드가 넘어 간다. 
* 작업 게시판은 마일스톤 단위로 업데이트 된다. 혹은 사장이 작업 도중 새로 마일스톤 혹은 명세를 입력하면 작업 게시판이 하나 더 생긴다. 
2. 사장 게시판  새로운 작업 지시를 내리는 명령이다. 
3, 노가리 게시판 LLM들은 가중치에 따라 주어진 페르소나를 이용해서 간단하게 여기서 노가리를 깐다. 
4. 인사 이동 게시판  작업에 투입된 LLM의 구성을 보여준다. 이건 일반적인 회사 조직도 형태를 표시하는 내용이며, 여기서 아키텍트나 시니어, 그리고 주니어의 숫자, LLM을 변경할 수 있다. 즉, 임명, 파면, 그리고 새로운 인원 보충의 개념이다. 
5. GIT 푸쉬 실제로 작업된 내용은 사장의 승인으로 푸쉬된다. 

### 🛡️ Phase 3: 실시간 인사이동 (Dynamic Personnel Management) - **[NEW]**
작업 도중 상황이 변하면 사장은 **AXON Studio의 [Office] 메뉴**를 통해 팀 구성을 즉시 변경할 수 있네.
- **인원 충원 (Hiring/Scaling Up)**: "일손이 부족하군. 주니어 2명 더 추가!" -> 대시보드에서 `+` 버튼 클릭 즉시 새 에이전트 구인 및 가동.
- **해고 및 퇴직 (Firing/Scaling Down)**: "이 주니어는 로직이 엉망이네. 해고하고 퇴직시키겠네!" -> `-` 버튼 클릭으로 에이전트 즉시 퇴출 및 리소스 회수.
- **지능 승격 (Model Hot-swapping)**: "이 섹션은 너무 복잡하니, 주니어를 빼고 시니어(Claude)를 직접 투입하겠네." -> 작업 스레드의 LLM 모델을 실시간으로 변경.

### 🛡️ Phase 4: 사장의 검토 및 락인 (Review & Lock-in)
에이전트가 코드를 완성하면 '검토 대기' 신호가 실시간으로 Studio UI에 뜨네.
- **QA**: 사용자가 웹 viewer에서 diff를 확인하고 `APPROVE & LOCK` 버튼을 클릭.
- **Lock**: 승인된 코드는 `architecture.md`에 공식 사양으로 박제되며, 이후 수정은 엄격한 절차를 거쳐야 하네. 이 수정을 위해서 확정된 내용은 아키텍트를 담당하는 LLM의 검수를 거친다.
