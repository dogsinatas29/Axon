# 🚀 AXON v0.0.30 — UI Intervention Architecture

## 📌 Core Question

When should human intervention occur?

And where should that intervention happen?

```text
Terminal?
or
Browser-based localhost UI?
```

This decision fundamentally changes the entire AXON user experience.

---

# 🧠 Core Conclusion

The intervention layer must be divided by responsibility.

```text
Terminal = Bootstrap + Runtime Console
Browser UI = Semantic Arbitration Layer
```

---

# 🔥 Why Terminal-Only UX Fails

AXON v0.0.30 no longer asks users to merely inspect logs.

The user now performs:

```text
Semantic Arbitration
```

The user is expected to decide:

- struct memory layouts
- ownership policy
- dependency classification
- ABI visibility
- lifecycle contracts
- unresolved semantic gaps

This is effectively:

```text
Mini Architecture Review
```

Terminal UX rapidly collapses into:

```text
1.
2.
3.
y/n
retry?
```

This creates:

- context fragmentation
- decision trace loss
- semantic history instability
- impossible diff visualization
- sealed contract drift

---

# 🌐 Why Browser UI Is Necessary

The browser becomes:

```text
Semantic Operating Console
```

It enables:

- semantic diff visualization
- ownership inspection
- dependency arbitration
- contract sealing
- decision history tracking
- interrupt prioritization
