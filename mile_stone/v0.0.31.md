### 5. Phase 1 → Phase 2 Auto-Resume Race Condition 숙청

#### 문제
Boss approve/reject/retry 핸들러에서 `pipeline_running.store(true)`를 `run_background()` 호출 전에 실행.
`run_background()` 내부에서 `running.load() == true` 체크 → 즉시 return → Phase 2가 실행되지 않음.

#### 조치 파일: `crates/axon-daemon/src/server.rs`
- `approve_thread` (Phase 1→2 자동 재개, line 254-267)
- `approve_pipeline_review` (line 574-576)
- `reject_pipeline_review` (line 652)
- `retry_pipeline_review` (line 733)

#### 수정 전/후
```
수정 전: Boss approve → pipeline_running.store(true) → run_background() → running.load()=true → return ❌
수정 후: Boss approve → run_background() → running.load()=false → running.store(true) → Phase 2 실행 ✅
```

### 10. WAL Flush Barrier (비동기 I/O 동기화 장벽)

#### 문제
`update_thread_status`가 `save_thread`로 Thread 상태를 비동기 WAL 큐에 넣은 후 즉시 return.
WAL 큐가 플러시되기 전에 파이프라인이 Auto-resume되거나 데몬이 재시작되면,
과거 스냅샷 데이터를 읽어 Phase 2 완료 태스크를 Pending으로 오독하는 '초기화 대참사' 발생 가능.

#### 조치 파일: `crates/axon-daemon/src/pipeline.rs`
- `update_thread_status` 함수 내 `save_thread` 직후 `storage.flush().await;` 장벽 주입
- "과거의 모든 디스크 쓰기 연산이 100% 완료되기 전에는 그 어떤 다음 페이즈 연산 노드도 진입하지 못한다"는 결정론적 선후 관계 보장

### 11. Phase 흐름 (v0.0.31 완성)

```
Phase 1: Header declarations (.h) → Boss approval gate → WAL flush barrier
    ↓
Phase 2: Source implementations (.cpp) → Boss approval gate → WAL flush barrier
    ↓
Phase 3: Integrator (CMakeLists.txt topology) → Build validation
```

### 15. 주니어 에이전트 추가 영향 분석 (2026-05-26)

#### 현재 구성
| 역할 | 수량 | 모델 |
|------|------|------|
| Architect | 1 | qwen2.5:7b-instruct-q4_K_M |
| Junior | 2 | qwen2.5:7b, deepseek-coder-v2:lite |
| Senior | 1 | deepseek-coder-v2:lite |

#### 동시 처리 메커니즘
```rust
// pipeline.rs:586
let semaphore = Arc::new(Semaphore::new(juniors.len()));
// pipeline.rs:598
let junior = juniors[idx % juniors.len()].clone();
```
- Semaphore 제한: `juniors.len()` (현재 2)
- 작업 분배: Round-robin (`idx % juniors.len()`)

#### 주니어 추가 시 영향

##### ✅ 긍정적 영향
1. **동시 처리량 증가**: 주니어 수 = 동시 행 가능 태스크 수
2. **Phase 완료 시간 단축**: 태스크 수 ÷ 주니어 수
3. **Draft 태스크 즉시 처리**: 대기 중인 태스크(`task_007`~`011`) 빠른 처리

##### ⚠️ 주의사항
1. **리소스 소비**: 주니어 1명당 약 2-4GB 메모리
2. **Diminishing Returns**: 태스크 수 < 주니어 수이면 이득 없음
3. **Senior 병목**: Junior가 빠르게 완료하면 Senior 검토 큐 쌓일 수 있음

#### 권장 사항
- 현재 상태: 2명 Junior, 5개 Draft 태스크
- 권장: **3명으로 증가** (1명 추가)
- 이유: Draft 5개 처리에 2명으로는 3회차 필요, 3명이라면 2회차로 처리 가능

#### 설정 방법
```bash
# 데몬 재시작 시
How many Juniors to hire? [default: 3]: 3
```
또는 `axon_config.json` 직접 수정.

### 28. GLOBAL_HARNESS 포인터 역참조/음수 리터럴 오판정 수정 (Side-Effect 유격 차단) (2026-06-01)

#### 문제

`is_empty_or_comments_only` 수정 1차 버전에서 `l.starts_with("*") || l.starts_with("-")` 조건이 블록 주석 컨텍스트와 무관하게 **모든** `*`/`-` 시작 라인을 주석으로 처리.

**사이드 이펙트 사례:**
```c
void init(GtkApplication* app) {
*ptr = 10;          // ← 라인 선두 '*' → 주석으로 오인 제거
    g_object_unref(app);
}
```

이 경우 `*ptr = 10;` 라인이 제거되어 `g_object_unref(app);` 만 남고, 최종적으로 **"주석뿐"으로 오인 → 자동 REJECT** 재발 가능성.

#### 매니저 검속 의견
> "C/C++ 소스 코드 중간에서 포인터 연산(*ptr = 10;)이나 감산 연산(-1)이 라인 선두에 올 때 코드가 주석으로 오인되어 잘려 나가는 '사이드 이펙트 유격'이 터질 수 있네! 블록 주석 제거 로직 내부에서만 해당 밸브가 작동하도록 닫아두어야 하네."

#### 조치 파일: `crates/axon-daemon/src/pipeline.rs`

##### 변경 전 (버그)
```rust
if is_line_comment || l.starts_with("*") || l.starts_with("-") {
    continue;  // ← 모든 '*'/'-' 라인 무조건 제거
}
```

##### 변경 후 (수정)
```rust
// 블록 주석 상태 관리를 라인 단위로 상단 이동
let mut in_block_comment = false;

for line in trimmed.lines() {
    let l = line.trim();

    // 블록 주석 시작/종료 감지 (라인 단위)
    if !in_block_comment && l.starts_with("/*") {
        in_block_comment = true;
        if l.contains("*/") { in_block_comment = false; }
        continue;
    }
    if in_block_comment {
        if l.contains("*/") { in_block_comment = false; }
        continue;
    }

    // 언어별 라인 주석만 제거 (블록 주석 외부)
    let is_line_comment = if is_python {
        l.starts_with("#")
    } else if is_lua {
        l.starts_with("--") && !l.starts_with("---")
    } else {
        l.starts_with("//")
    };

    if is_line_comment { continue; }

    // '*' and '-' at line start are NO LONGER stripped here.
    // They are valid code: *ptr = 10;  -1  -x + y
    cleaned.push_str(l);
}
```

### 31. Phase 완료 팝업/알림 미표시 버그 수정 (localStorage 레이스 컨디션) (2026-06-01)

#### 문제

Phase 1 완료 후 성공 팝업이 표시되지 않음.

#### 근본 원인 분석

1. **localStorage 레이스 컨디션**: `shownPopups` 초기 상태가 `{}` (빈 객체)이고, `/api/status` fetch가 완료된 후에야 localStorage에서 값을 로드함. 팝업 감지 useEffect가 localStorage 로드 전에 실행되어 `shownPopups[1] === undefined` → `!undefined === true` → 팝업 표시 시도. 하지만 이후 localStorage 로드 완료 시 `{1: true}` (이전 실행 기록)가 적용되면 팝업이 영구 스킵됨.

2. **두 useEffect 간 충돌**: 
   - Effect 1: Phase 완료 시 팝업 표시 (`isPhase1Complete && !shownPopups[1]`)
   - Effect 2: Phase 미완료 시 `shownPopups[1]` 리셋
   - Effect 2가 Effect 1의 `shownPopups` 상태를 덮어쓰며 무한 리렌더링 루프 발생 가능

3. **DB 스키마 확인**: `list_all_threads()`는 모든 스레드 반환 (Completed 포함). `/api/threads` 엔드포인트는 정상 작동. DB 문제 아님.

## v0.0.32: Failure-driven Patch Radius Gate 연결

### 배경

`validate_patch_radius()` 함수는 v0.0.31에서 정의되었으나 **실제 파이프라인에서 호출되지 않는 데드엔드**였다. 감사 결과 근본 원인은 `PatchContract.allowed_regions`가 어디에서도 생성되지 않았기 때문이다.

### 설계 결정: Failure-driven vs Design-driven

**옵션 A/C (Design-driven):** `architecture.md` → `Component.patchable_regions` → `PatchContract.allowed_regions`
- 설계 시점에 수정 영역을 미리 정의하는 모델
- IR 스키마 변경, 파서 수정, 설계 문서 수정 필요

**옵션 B (Failure-driven):** `FailedDiagnostic.error_line` → `(line-3, line+3)` → `allowed_regions`
- 실패 시점에 자동으로 수정 영역 결정
- IR/Schema 무변경, 기존 `.failed.json` 재활용

**결론: 옵션 B 채택** — 현재 AXON의 `FailedDiagnostic → Pinpoint Diagnosis → Patch Repair` 체인과 철학적으로 일치.

### diagnostic_to_regions() 함수 (`pipeline.rs`)

```rust
// Regions are 1-based inclusive line numbers (e.g. line 42 → (39, 45))
fn diagnostic_to_regions(diags: &[FailedDiagnostic]) -> Vec<(usize, usize)> {
    diags.iter()
        .filter_map(|d| d.error_line.map(|line| (line.saturating_sub(3), line + 3)))
        .collect()
}
```

`FailedDiagnostic.error_line: Option<usize>`를 `validate_patch_radius()`가 기대하는 `&[(usize, usize)]`로 변환하는 어댑터 계층.

### load_failed_regions() 함수 (`pipeline.rs`)

```rust
fn load_failed_regions(sandbox_root: &PathBuf, target: &str) -> Vec<(usize, usize)> {
    let json_path = PathBuf::from(format!("{}.json", failed_path));
    match std::fs::read_to_string(&json_path) {
        Ok(json) => match serde_json::from_str::<Vec<FailedDiagnostic>>(&json) {
            Ok(diags) => Self::diagnostic_to_regions(&diags),
            Err(e) => { tracing::debug!(...); Vec::new() }
        },
        Err(e) => { tracing::debug!(...); Vec::new() }
    }
}
```

**변경 전:**

```
    ↓
[auto_reject_reason 체크] → Senior Review
```

**변경 후:**
```
[COMPILATION_GATE]
    ↓
[PATCH_RADIUS]  ← NEW: retries>0 시 .failed.json → regions → validate_patch_radius()
    ↓
[auto_reject_reason 체크] → Senior Review
```

### 삽입 코드 (`pipeline.rs:1672-1690`)

```rust
if auto_reject_reason.is_none() && retries > 0 {
```

3. 오탐이 없으면 Phase 3에서 `AUTO_REJECT`로 격상 검토

### Module State Taxonomy 정의

> AXON은 이제 함수 단위가 아닌 **상태 단위**로 분류한다. "호출 여부"만으로 데드엔드를 판단하면 Passive Guard나 Dormant 기능을 오탐한다.

| 상태 | 의미 | 판단 기준 |
|------|------|----------|
| **Active** | 실행 경로에 연결, 파이프라인에서 실제 동작 | 호출됨 + 흐름에 영향 |
| **Passive** | 실행되지만 관측만 수행 (`tracing::warn!`), 흐름 미차단 | 호출됨 + 결과 무시 |
| **Dormant** | 구현 완료, 활성화 조건 미충족으로 미연결 | 미호출 + 조건 명시 |
| **Archived** | 역사 보존용, 컴파일/링크/호출 배제 | 컴파일 트리 제외 |
| **Legacy** | 과거 시스템 호환, 메인 파이프라인 미연결 | legacy 전용 경로에서만 호출 |
| **Dead** | 진짜 고아 코드, 삭제 후보 | 미호출 + 의미 소멸 |

### Module State Taxonomy 정의

> AXON은 이제 함수 단위가 아닌 **상태 단위**로 분류한다. "호출 여부"만으로 데드엔드를 판단하면 Passive Guard나 Dormant 기능을 오탐한다.

| 상태 | 의미 | 판단 기준 |
|------|------|----------|
| **Active** | 실행 경로에 연결, 파이프라인에서 실제 동작 | 호출됨 + 흐름에 영향 |
| **Passive** | 실행되지만 관측만 수행 (`tracing::warn!`), 흐름 미차단 | 호출됨 + 결과 무시 |
| **Dormant** | 구현 완료, 활성화 조건 미충족으로 미연결 | 미호출 + 조건 명시 |
| **Archived** | 역사 보존용, 컴파일/링크/호출 배제 | 컴파일 트리 제외 |
| **Legacy** | 과거 시스템 호환, 메인 파이프라인 미연결 | legacy 전용 경로에서만 호출 |
| **Dead** | 진짜 고아 코드, 삭제 후보 | 미호출 + 의미 소멸 |
