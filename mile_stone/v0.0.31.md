## 🧠 상세 설계 및 로직
### 1. P5-6e: Semantic Equivalence Kernel (의미론적 동치성 커널)
- **목표:** AST-native mutation의 안정성 확보를 위해 "Semantic Identity Lattice"를 구축. 단순한 구조적 일치(structural identity)를 넘어, 의미론적으로 동일한 변화인지 수치적으로 입증.
- **분류 체계 (Taxonomy):** `SemanticMutationClass` 도입 (FormattingOnly, ImportOrdering, SignatureMutation 등).
- **거리 측정 (Distance Score):** 0.00 (whitespace) ~ 0.99 (execution semantic mutation)까지의 `semantic_distance_score` 산출.
- **정규화 (Canonicalization):** 파서(Printer) 출력에 의존하지 않는 `AST -> Semantic Token Graph -> Canonical Semantic Form` 파이프라인.

### 2. P5-7 ~ P6: Atomic Governance & Failure Cascade (운영체제급 방어 기제)
- **Causal Ledger & Rollback:** 모든 상태 전이를 인과율 원장(`StateTransitionRecord`)에 기록하고, 오너십/위상 붕괴 시 물리적으로 되돌림.
- **Atomic I/O:** `tempfile` 생성, `fsync`, `rename` 스왑 패턴 강제로 권위 상태 파일의 조용한 파괴(Silent Corruption) 방지.
- **Circuit Breaker:** `FailureBudget` 기반 무한 수리 루프 및 연쇄 붕괴 격리(Quarantine).
- **Zombie Eviction:** `OwnershipLease` 기반의 TTL 및 하트비트 체크로 교착(Deadlock) 방지.

### 3. P6-SIM-T1: Integration Crash Suite
- `atomic_crash.rs`: `fsync` 전/후 크래시 주입 테스트.
- `lease_eviction.rs`: 프리징된 워커로부터 오너십 정상 회수 검증.
- `circuit_breaker.rs`: 정해진 실패 임계치 초과 시 에스컬레이션 작동 검증.
- `determinism_replay.rs`: 다수 리플레이 후 `SYSTEM_STATE_HASH` 100% 일치 증명.
- `ripple_containment.rs`: 허브 심볼 붕괴 시 확산 반경(Ripple Radius) 억제 검증.

AXON을 관측소(Observatory) 단계에서 벗어나, "시스템을 죽이지 않으면서 레거시를 안전하게 진화시키는 시스템(Runtime-safe software evolution system)"으로 승격.
모든 변경을 텍스트 수정이 아닌 **"위상 변이 트랜잭션(Topology Mutation Transaction)"**으로 취급.

**상세 집도 내용:**
- **Phase J: Runtime Causality Compression & Telemetry Sensor**
  - `signal_nesting.rs`, `refcount_logger.rs`, `deferred_lineage.rs`, `widget_snapshot.rs` 등 물리적 런타임 센서 구축.
  - `RuntimeCausalityCompressor`: Raw Telemetry(ASLR, Jitter 포함)를 정규화하여 `DestroyWithoutStabilization`, `DeferredOrphanDispatch` 등 Canonical Pathology Event로 압축.
  - `live_attach_harness.rs`: 1000회 실행 시 노이즈 하에서도 동일한 병리로 수렴(`compressed_lineage_identity == 1.0`)하는지 결정론 증명.
  - `observer_effect.rs`: LD_PRELOAD 센서 자체가 큐 순서를 왜곡하는지(Observer Effect) 검증.

- **Phase K: Controlled Software Evolution Workflow**
  - **Mutation Intent Layer (`intent_dsl.rs`)**: `ADD_IDLE_CALLBACK`, `ATTACH_SIGNAL` 등 위상 변화 의도만 선언적으로 명시.
  - **Mutation Boundary Lock (`boundary_lock.rs`)**: 의도가 기존 소유권이나 큐 경계를 침범하는지 검증.
  - **Safe Mutation Envelope (`safe_envelope.rs`)**: 검증 통과 시 발급되며, 변이는 이 봉투 내부의 엄격한 스코프 안에서만 수행됨.
  - **Evolution Workflow (`workflow.rs`)**: 의도 -> 락 검증 -> 안전 봉투 발급 -> 런타임 변이 -> 리플레이 및 관측 -> 승인으로 이어지는 실전 개발 루프 구축.
  - **Runtime Drift Diff Visualizer (`drift_visualizer.rs`)**: 병리 발생 사유를 인간이 즉시 이해할 수 있도록 위상 변형(Topology Drift)을 시각화하여, 고무도장(Rubber Stamping) 승인 방지.

- **Phase L: Proof-Driven Evolution & Human Approval Workflow**
  - **Topology Mutation Contract**: 텍스트 코드 Diff 리뷰를 대체하는 새로운 승인 규격. 의도(Intent), 리스크 예측(Risk Forecast), 리플레이 증거(Replay Evidence), 법의학 리포트(Forensic Report)를 묶어 런타임 인과율 리뷰 시대로 전환.
  - **Mutation Transaction Ledger (`ledger.rs`)**: 인간 관제사의 승인/거절 내역과 트랜잭션 기록을 영구 보존하는 원장. 불안전한 위상 변형 승인을 하드락(Hard Reject)으로 원천 차단.
  - **Runtime Evolution Dashboard (`dashboard.rs`)**: `[SAFE]` vs `[DANGEROUS]` 상태와 런타임 위상 보존 여부(Queue Drift, Ownership Drift 등)를 시각화하는 진화 레이더(Evolution Radar) 구현.
  - **Paradigm Shift 박제**: AXON을 "AGI IDE 판타지"에서 **"결정론적 런타임-안전 소프트웨어 진화 커널(Deterministic Runtime-Safe Software Evolution Kernel)"**로 공식 선언 (GEMINI.md 최상단 기록 완료).

- **Phase M: Proof Artifact Schema (Portable Evolution Certificate)**
  - **`.axon-proof/` 번들 구축**: Git Commit을 대체할 런타임 진화 증명 패키지(Proof Bundle) 직렬화 파이프라인.
  - `proof.verdict.json` (스키마 버전 및 최종 판결), `mutation.intent.json`, `lineage.delta.json`, `queue.diff.json`, `ownership.diff.json` 정의.
  - `replay.trace.bin`: 독립적 재검증(Machine Replayable Verification, 예: `axon verify proof.axon`)을 위한 압축된 캐노니컬 바이너리 트레이스.
  - **진화 워크플로우 정립**: CI/CD(compile -> test -> deploy) 구조를 넘어 `mutate -> replay -> prove -> govern -> merge` 라는 완전히 새로운 소프트웨어 진화 프레임워크 확립.

- **Phase N: Production Pressure Validation (GTK2 실전 수술 단계 진입)**
  - 이론(Topology theory) 확장을 중지하고, AXON 커널이 실전 환경에서 살아남는지 증명하는 단계로 전환.
  - **SAFE Mutation**: Timeout 추가, Reconnect 로직 변경 등 안전한 진화 시도.
  - **DANGEROUS Mutation**: Human/LLM이 주입한 의도적인 고아 위젯(Orphan Widget), 파괴 순서 오류, 큐 순서 역전(Queue ordering inversion) 등의 패치를 AXON이 정확하게 거부(Hard Reject)하는가 검증.
  - **E2E 브라운필드 유지보수**: GTK2 앱 -> Intent -> Mutation -> Replay -> Proof Bundle -> Dashboard Review -> Approval -> Merge 의 실전 루프 검증 돌입.

- **Phase O: Operational Integration & Thin Governance Hook (조직 전개 준비)**
  - 거대한 데스크탑 플랫폼이나 Custom VCS 대신, 기존 개발 워크플로우에 마찰 없이(Frictionless) 삽입되는 `Thin Governance Hook` 전략 확립.
  - 통합 CLI (`axon verify`)를 통해 `.axon-proof` 번들을 검증하고, `SAFE_TO_MERGE` 판결 이외에는 Native Exit Code 1을 반환하도록 설계.
  - `.git/hooks/pre-push` 템플릿을 생성하여, 증명되지 않은 패치는 CI 이전에 물리적으로 차단되는 'Invisible Infrastructure' 구현 완료.

- **Phase P: Phase 2.5 - Operational Kernel Hardening (커널 연마 및 섀도우 검증)**
  - CI 연동으로 인한 'Premature Freeze'를 방지하고, 전면 조직 전개(Rollout) 이전에 커널의 런타임 인과율 판별력(Precision)을 극대화.
  - **Tiny Mutation Corpus 확보:** GTK, Win32, Async, Timer, Plugin Lifecycle 등 10~20개의 '작고 통제된 실제 진화' 핫스팟 샘플 집중 수집.
  - **Precision 교차 증명:** `SAFE_TO_MERGE` (안전한 패치)와 `TOPOLOGY_DRIFT_REJECTED` (위험한 위상 파괴)를 균형 있게 발생시켜 커널의 거부 정밀도 입증.
  - **Collapse Lineage Taxonomy 숙성:** `OrphanedAsyncTask`, `StaleHWNDDispatch` 등의 병리 분류 체계가 실증 데이터를 통해 자연스럽게 통폐합되도록 대기.
  - **Real Repository Pilot (XChat):** 전체 저장소를 만지지 않고, `Safe Timeout Stabilization` 단일 핫스팟에 대해서만 `MAX_CHANGED_LINES <= 20`, `MAX_NEW_TIMERS <= 1`, `NO_NEW_GLOBAL_STATE` 등의 엄격한 Envelope 제약 하에 완벽한 위상 보존(Replay Identity 1.0) 실측 성공.

- **Phase Q: Causal Lineage Normalization (분류 체계 압축 및 설명력 극대화)**
  - AXON의 관측을 `Event-centric`에서 **`Causal-family-centric`**으로 전환하여 Taxonomy 파편화 방지 및 압축률 상승.
  - **Root vs Symptom 계층 분리:** `CancellationCollapse`, `OwnershipCollapse` 등의 근본 원인(Root Lineage)을 정의하고, `ZombieRetryLoop`, `DeferredOrphanDispatch` 등을 파생 증상(Symptom)으로 매핑(Deterministic Mapping).
  - **Proof Artifact 스키마 개편:** `lineage.delta.json` 구조를 `CausalFamily` 중심으로 재편하여 인간 관제사(Reviewer)가 "어떤 근본 원인으로 어떤 증상이 발생했는지"를 직관적으로 이해할 수 있도록 **Governance Explainability** 극대화.
  - **Benign Corpus 공식화:** `SafeBenignPropagation` (안전한 재귀, 선택 동기화 등)을 Root Lineage에 편입시켜 AXON이 무조건 차단만 하는 Paranoid Kernel로 타락하는 것을 방지.

- **Phase R: Archive Semantic Continuity (과거 데이터의 시맨틱 연속성 확보)**
  - **Taxonomy Migration Manifest v2.0.0:** 과거의 레거시 이벤트 이름들을 신규 Root Lineage 체계로 정규화하는 `lineage_taxonomy.rs` 구축.
  - **Catastrophe Archive Normalization:** 기존의 1차원 문자열 배열로 저장되던 붕괴 아카이브를 정규화하여, 과거의 지식과 미래의 판독 기준을 동일한 Causal Ontology 위에 통일. Semantic Bifurcation 방지.
  - **Root-Weighted Similarity Scoring:** 단순 증상 이름의 텍스트 비교가 아닌 **인과 구조(Root Lineage)** 기반의 Similarity Scoring을 구현. 증상(`Symptom`)이 달라도 근본 원인(`Root`)이 같으면 높은 유사성을 도출하여 예측 면역력(Predictive Immunity) 강화.

- **Phase S: Frictionless Governance Hook (운영 마찰 최소화 체계)**
  - AXON 커널의 런타임 증명력을 거대한 대시보드나 CI 플랫폼 확장이 아닌, 개발자의 로컬 환경과 워크플로우에 조용히 스며드는(Invisible) 형태로 압축 전개.
  - **Governance Cognition Latency 최소화:** CLI 출력에서 `SAFE`, `REJECTED`, `CATASTROPHIC`, `PROOF_CORRUPTED`의 명확한 상태 헤더와 Exit Code(0~3)를 매핑.
  - **Root Lineage First Rendering:** 인간이 즉시 인과를 파악할 수 있도록, 원인 불명 증상 나열을 폐기하고 `Root Cause ➔ Symptoms` 순서로 직관적 렌더링.
  - **Safe Minimalism:** `[SAFE]` 상태일 때는 불필요한 로그를 감추어, 개발자의 피로도(Fatigue)를 낮추고 거버넌스 출력을 무시하는 현상 방지.
  - **Actionable Corruption UX:** `.axon-proof` 손상 시 어떤 스키마나 필드가 누락되었는지 정확하게 지적하여 툴링(Tooling) 신뢰도를 확보.
  - **Hook Bypass Semantics:** 긴급 상황을 위한 `AXON_BYPASS=1` Override를 허용하되, Audit Trail을 남겨 운영 리스크를 투명하게 통제.
  - **Thin Wrapper Integration:** `.git/hooks/pre-push` 템플릿과 `.github/workflows/axon_governance.yml`을 구축하여, 거버넌스 오버헤드 없이 순수 Transport Layer로서만 동작하도록 구현.

- **Phase T: Operational Survivability & Latency Discipline (운영적 생존성 및 지연 시간 규율)**
  - AXON의 성패를 가르는 가장 핵심적인 검증 포인트인 **"Governance Friction Ceiling(개발자가 거버넌스를 귀찮아하지 않는 한계치)"**을 수학적으로 증명하는 단계.
  - `governance_latency_harness.rs`를 구축하여 기능 확장의 유혹을 차단하고 오직 **Latency Discipline**에 집중.
  - **Proof Loading Cost (Deserialization):** 1,000개의 Causal Family 증명서 파싱을 약 `2.76ms` (Proof당 `2.76µs`)만에 돌파하여 증명서 적재 병목 제로(0)화 입증.
  - **Similarity Scoring Amplification:** Root Lineage 기반 이진 스코어링 최적화를 통해 10,000개의 Archive 데이터 비교 시 단 `43.65µs` 소요. 아카이브 성장에 따른 거버넌스 지연(Governance Slowdown) 리스크 완벽 제거.
  - **Replay Scope Amplification Ceiling:** 변경 라인(Changed Lines) 대비 파급 노드(Affected Nodes)가 비정상 폭증할 시(예: 실행 비용 11.0s), 즉각 `[!] REGRESSION ALERT`를 발생시켜 커널 스스로 무거워지는 것을 방어하는 기제 작동 증명.
  - **Local Pre-Push Latency Projection:** 전체 과정을 통과하는 데 약 `2.66s`가 소요되어, 마찰 없는(Frictionless) 기준점인 `Ideal < 3.0s` 이내에 도달함을 확인.
  - **결론:** AXON은 기능적으로 똑똑한 프로토타입을 넘어, **개발자 피로도(Developer fatigue)를 유발하지 않는 생존성(Survivability) 높은 커널**로 완성됨을 증명.

## 📝 Post-Work Log (작업 후 기록) - Phase 2: WAL Flush Barrier (Bootstrap-Pipeline Race Condition Fix)

### 1. 문제 정의
- Bootstrap이 `save_tasks_from_decomposed()`로 11개 태스크를 생성 후 `run_v3()` 완료.
- 직후 `pipeline.run_background()`가 `list_all_tasks()` 조회 시 SQLite에 아직 flush되지 않아 `No pending tasks` → 즉시 종료.
- 원인: `enqueue_durable()`가 `mpsc` 채널에 적재만 하고 return. `durable_worker`는 100ms timeout 또는 50개 배치 도달 시에만 실제 SQLite commit 수행.

### 2. 해결: 명시적 Flush 동기화 Barrier
**아키텍처:**
```
bootstrap 완료
  → storage.flush()
    → flush_tx에 oneshot::Sender 전송
      → durable_worker가 flush_rx 수신
        → pending batch 즉시 flush_batch_durable() (SQLite commit)
        → oneshot::Sender::send(()) 응답
  → flush() await 완료 (disk 동기화 확보)
→ pipeline.run_background() (안전하게 SELECT)
```

**변경 파일:**
- **`crates/axon-storage/src/lib.rs`**:
  - `Storage`에 `flush_tx: mpsc::UnboundedSender<oneshot::Sender<()>>` 필드 추가
  - `durable_worker` 루프에 `flush_rx.recv()` select branch 추가 — 수신 시 즉시 batch flush 후 ack 응답
  - `flush()` 메서드 신설: oneshot 채널 생성 → `flush_tx` 전송 → ack 대기
- **`crates/axon-daemon/src/lib.rs`** `start_with_spec()`:
  - `manager.run_v3()` 완료 후 `self.storage.flush().await` 호출
- **`crates/axon-daemon/src/server.rs`** `submit_spec()`:
   - `manager.run_v3()` 성공 후 `pipeline_storage.flush().await` 호출

## 📝 Post-Work Log (작업 후 기록) - Phase 5.5: WAL Flush FIFO Drain (Bootstrap-Pipeline Race Condition 완전 해결)

### 1. 문제 재정의
- Bootstrap이 `save_tasks_from_decomposed()`로 11개 태스크 생성 후 `flush()` 호출.
- `flush()` → `flush_tx`에 oneshot 전송 → durable worker의 `tokio::select!`가 flush_rx를 rx보다 먼저 선택하면 **batch가 비었거나 일부만 찬 상태로 flush** → ack 반환.
- pipeline이 `list_all_tasks()` 조회 시 아직 쓰이지 않은 태스크는 안 보임 → `No pending tasks` → 즉시 종료.
- SQLite DB에는 **과거 TEST4_GTK 태스크만 존재**하고, `project_id = "spec"`인 현재 세션 태스크는 0건.

### 2. 근본 원인: `tokio::select!`의 경합 조건
```
durable worker loop:
  tokio::select! {
      Some(env) = rx.recv()      → batch.push(env)       [쓰기 메시지]
      Some(ack) = flush_rx.recv() → flush_batch(); ack     [flush 신호]
      _ = timeout                → flush_batch()           [100ms timeout]
  }

❗ flush_tx (unbounded)로 보낸 flush 신호가 rx 메시지보다 먼저 select!에 Poll::Ready가 되면:
   → batch = [] 상태로 flush → ack 전송
   → SQLite에 0건 기록
   → 메인 스레드는 "✅ WAL queue flushed" 로그 출력 (거짓 긍정)
   → pipeline: No pending tasks
```

### 3. 해결: `flush_rx` 핸들러 내 `rx.try_recv()` 선행 소진
- **파일**: `crates/axon-storage/src/lib.rs` — durable worker loop (line 115)
- **수정 사항** (단 4줄 추가):
  ```rust
  Some(ack_sender) = flush_rx.recv() => {
      // ★ FIFO 보장: flush signal 이전의 모든 tx 메시지를 먼저 처리
      while let Ok(env) = rx.try_recv() {
          batch.push(env);
      }
      if !batch.is_empty() {
          Self::flush_batch_durable(&db_writer, &writer_log, &writer_dlq, &mut batch);
      }
      let _ = ack_sender.send(());
  }
  ```
- **기전**: `mpsc::Receiver::try_recv()`는 non-blocking receive. flush signal이 `flush_rx`로 먼저 도착해도, flush 핸들러 진입 후 `rx.try_recv()`가 `TryRecvError::Empty`가 될 때까지 모든 선행 쓰기 메시지를 수집 → 일괄 flush → ack. FIFO 순서가 절대적으로 보장됨.
- **영향**: WAL flush가 이제 진정한 의미의 sync barrier로 작동. Bootstrap → Pipeline 사이의 태스크 소멸 문제 완전 제거.
- **Zero-cost in normal path**: flush가 호출되지 않은 정상 경로에서는 `flush_rx.recv()`가 실행되지 않으므로 `try_recv()` 호출도 발생하지 않음.

### P0 #1: Boss Approve Auto-Resume (`server.rs:192-258`)

**파일**: `crates/axon-daemon/src/server.rs` — `approve_thread()` 핸들러

**문제**: Boss가 thread를 승인하면 task 상태는 `Completed`로 변경되지만, 파이프라인은 이미 `return`으로 종료된 상태. Phase Gating 체크를 통과해도 Phase 2가 자동 시작되지 않음.

**해결**: `approve_thread()` 내에서 task 완료 처리 후 `tokio::spawn`으로 백그라운드 체크 시작:
```rust
// Boss Approve 후 Phase Gating 재체크
let all_tasks = state_for_spawn.storage.list_all_tasks().unwrap_or_default();
let phase1_tasks: Vec<_> = all_tasks.iter().filter(|t| {
    t.project_id == project_id &&
    t.target_file.as_deref().map(|f| f.ends_with(".h") || f.ends_with(".hpp")).unwrap_or(false)
}).collect();

if !phase1_tasks.is_empty() {
    let all_completed = phase1_tasks.iter().all(|t| t.status == TaskStatus::Completed);
    if all_completed {
        let phase2_pending: Vec<_> = all_tasks.iter().filter(|t| {
            t.project_id == project_id &&
            t.status != TaskStatus::Completed &&
            !t.target_file.as_deref().map(|f| f.ends_with(".h") || f.ends_with(".hpp")).unwrap_or(false)
        }).collect();

        if !phase2_pending.is_empty() && !state_for_spawn.pipeline_running.load(Ordering::SeqCst) {
            state_for_spawn.pipeline_running.store(true, Ordering::SeqCst);
            let mut pipeline = ExecutionPipeline::new(...);
            pipeline.run_background();
        }
    }
}
```

**데이터 흐름**:
```
Boss가 task_005 승인 → task_005 = Completed
  → tokio::spawn { Phase 1 모두 Completed? → YES }
  → Phase 2 pending task 존재? → YES (task_006~011)
  → pipeline_running.store(true) + ExecutionPipeline 재spawn
  → Phase 2 자동 시작 ✅
```

```rust
pub fn detect_senior_header_hallucination(
    target_file: &str,
    _feedback: &str,
    proposed_code: &str,
) -> bool {
    if !target_file.ends_with(".h") && !target_file.ends_with(".hpp") { return false; }
    
    let has_function_body = proposed_code.lines().any(|line| {
        let trimmed = line.trim();
        if trimmed.starts_with('#') || trimmed.starts_with("//") || trimmed.is_empty() { return false; }
        if trimmed.contains('{') {
            let before_brace: String = trimmed.split('{').next().unwrap_or("").to_string();
            if before_brace.contains(')') && !before_brace.trim().starts_with("struct")
                && !before_brace.trim().starts_with("enum") && !before_brace.trim().starts_with("class") {
                return true;
            }
        }
        false
    });
    
    if !has_function_body {
        tracing::warn!("🛡️ [HALLUCINATION_GUARD] Senior rejected header {}, but proposed code has NO function bodies.", target_file);
        return true;
    }
    false
}

- `#[serde(untagged)] enum RawSymbol`을 선언하여 문자열과 객체를 모두 유연하게 받아들이는 다형성 파서를 구현. `allowed_includes` 및 `metadata` 속성의 맵핑도 함께 복원.

#### [Fix 2] 수동 인터락 대기 파일명 고유화 (Backend) — `crates/axon-daemon/src/pipeline.rs`

#### [Fix 3] 워크 보드 승인 API 핸들러 수동 인터락 폴백 추가 (Backend) — `crates/axon-daemon/src/server.rs`
- `approve_thread`, `reject_thread`, `retry_thread` API 핸들러가 호출되었을 때, 해당 태스크가 인메모리 `pending_reviews`에 존재하지 않는 경우 고유 승인 대기 파일(`.axon_approval_pending_{thread_id}`)을 직접 갱신하도록 폴백을 작성했습니다.
- 수동 인터락 대기 상태에서 승인을 진행하면 백그라운드 폴링 루프가 정상적으로 해제되며, 승격(Promotion, 파일 이동) 및 다음 공정(Phase 2)으로의 이행이 안전하게 재개됩니다.

### 🛠️ 변경 파일 목록 (Affected Files)

#### Phase 1 — 저장소 실체화
##### `crates/axon-daemon/src/intelligence/corpus/catastrophe_archive.rs`
- `normalize_and_archive()`: stub `Ok(())` → `runtime/catastrophe_archive.jsonl`에 JSONL append 저장으로 실체화
- `load_archived_roots()` 신규 구현: JSONL 파일을 읽어 `Vec<RootLineage>` 반환
- `check_against_archive()` 편의 함수 추가: load + is_predictable_catastrophe 원스텝 호출

#### Phase 2 — 유사도 개선
##### `crates/axon-daemon/src/intelligence/replay/lineage_taxonomy.rs`
- `CausalSimilarityScorer::calculate()`: binary(1.0 / 0.0) → **가중치 매트릭스** 방식으로 교체
  - 동일 계통 = 1.0
  - Lifecycle 계열 연관 (OwnershipCollapse ↔ CancellationCollapse) = 0.4
  - Flow 계열 연관 (QueueOrdering ↔ RecursivePropagation) = 0.3
  - Lifecycle ↔ Flow 간 약한 연관 = 0.2
  - 무관 계통 / SafeBenignPropagation = 0.0

#### Phase 3 — Corpus 실행 체인
##### `crates/axon-daemon/src/intelligence/corpus/corpus_executor.rs`
- `execute_shadow_campaign()`: `Ok(())` stub → 6단계 실행 체인으로 실체화
  1. `pre_hash` — 원본 SHA-256 해시 계산
  2. `shadow_copy` — tmpfs에 격리 복사본 생성
  3. `mutation` — 복사본에만 변환 적용 (원본 절대 불변)
  4. `semantic_gate` — 줄 수 드리프트(5% 이하) + 함수 선언부 보존 여부 확인
  5. `rollback` — tmpfs 정리
  6. `post_hash` — pre_hash 일치 검증 (원본 불변 증명)

##### `crates/axon-daemon/src/intelligence/corpus/mutation_campaign.rs`
- `run_campaign()`: 하드코딩 더미값 → **SAFE_SUBSET_V1 실측 루프**로 교체
  - 실제 `source_files`를 순회하며 `CorpusExecutor::execute_shadow_campaign()` 호출
  - 실측된 `determinism_rate`, `semantic_drift` 기반 `AggregatedMetrics` 반환

##### `crates/axon-daemon/src/intelligence/corpus/corpus_ingestor.rs`
- `ReproducibleCorpusSnapshot`에 `source_files: Vec<String>` 필드 추가

#### Phase 4 — Pipeline 연결
##### `crates/axon-daemon/src/pipeline.rs`
- **`[LINEAGE_GUARD]` 훅 추가** (Junior 제출 직후):
  - 제출된 코드에서 알려진 `SymptomLineage` 키워드를 탐지
  - `catastrophe_archive.jsonl`에 동일 계통 기록이 있을 경우 `⚠️ [LINEAGE_GUARD] WARN` 로그 발생
  - REJECT 없이 Senior에게 위험 컨텍스트 전달
- **`[CORPUS_GATE]` 훅 추가** (Boss 승인 후 promotion 직전):
  - sandbox 파일에 identity mutation(변환 없음) shadow campaign 실행
  - `pre_hash == post_hash` 검증으로 파일 불변성 증명
  - 위반 시 `🚨 [CORPUS_GATE] INTEGRITY VIOLATION` 에러 로그

```
