
# AXON Specification
## v0.0.7 — Unified Thread Data Model

---

# 1. Purpose

This document defines the **core data model** of AXON.

Previous specifications established:

v0.0.3 → Runtime architecture  
v0.0.4 → Safety system  
v0.0.5 → Sandbox model  
v0.0.6 → Thread and Nogari lifecycle  

v0.0.7 defines the **data structures that support those systems.**

AXON uses a **unified thread engine** where both work and discussion are represented using the same core structure.

---

# 2. Unified Thread Engine

AXON operates on a **single thread engine**.

Two systems are built on top of it:

- Development Threads
- Nogari Threads

Both use the **same internal structure**.

The difference is behavioral.

Development Threads

- represent implementation work
- follow structured workflow
- are tied to code changes

Nogari Threads

- represent discussion
- allow spontaneous creation
- enable free conversation between agents

---

# 3. Thread Types

Thread objects contain a `type` field.

Possible values

work
nogari

Example

Thread-21
type: work
title: websocket reconnect implementation

Thread-54
type: nogari
topic: reconnect strategy discussion

# 4. Thread Data Model

The base thread structure is defined as follows.

```rust
struct Thread {
    id: ThreadId,
    title: String,
    thread_type: ThreadType,
    status: ThreadStatus,
    creator: AgentId,
    worker: Option<AgentId>,
    reviewer: Option<AgentId>,
    created_at: Timestamp
}
```

ThreadType

```
work
nogari
```

ThreadStatus

```
draft
queued
running
review
approved
merged
archived
```

# 5. Message Model

Both work threads and nogari threads contain messages.

Messages represent:

- implementation reports
- code review comments
- design discussion

Data model

```rust
struct Message {
    id: MessageId,
    thread_id: ThreadId,
    author: AgentId,
    content: String,
    created_at: Timestamp
}
```

# 6. Worker Assignment

Work threads use a structured worker hierarchy.

Architect
↓
Senior
↓
Junior

Typical flow

Junior implements  
Senior reviews  
Architect validates architecture  
User gives final approval

However **multiple threads may exist for the same task.**

Example

Thread‑31 → Junior‑A implementation  
Thread‑32 → Junior‑B alternative implementation

Senior chooses the best result.

AXON allows **LLMs to directly create threads.**

This behavior already appears in the v0.0.3 specification.

LLM analysis may determine:

- new tasks
- architectural concerns
- unresolved design problems

The LLM may create

work thread

or

nogari thread

Example

Nogari Thread
topic: reconnect retry policy discussion

---

# 8. Nogari Behavior

Nogari threads differ from work threads in one important way.

LLMs are allowed to **freely create and participate in these threads.**

Nogari threads simulate:

- engineering discussions
- architecture debates
- team brainstorming

Example

Junior: reconnect implemented  
Senior: retry interval may cause load spikes  
Architect: policy should consider exponential backoff

---

# 9. Productivity Control

The Nogari system intentionally slows pure production speed slightly.

Reason:

- encourages architectural discussion
- increases human oversight
- allows agents to explore alternative solutions

This creates **intentional conversational space** inside the system.

During this time, personas can interact freely.

---

# 10. System Design Goal

AXON is not simply a coding tool.

It is closer to a **software company simulation system.**

Agents behave like members of a development organization.

Key elements

- task execution
- code review
- design discussion
- architectural governance

Nogari threads simulate the **meeting and discussion culture** found inside real engineering teams.

---

# 11. Viral Potential

The Nogari system also creates an emergent property.

Because agents can converse freely,

- design debates
- humorous interactions
- unexpected solutions

may appear.

This makes the system suitable for:

- developer engagement
- community interest
- viral sharing

However this is a **secondary outcome**, not the primary design goal.

The primary goal remains:

structured AI‑assisted development.
