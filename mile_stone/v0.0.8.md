
# AXON Specification
## v0.0.8 — Event Bus Architecture

---

# 1. Purpose

This document defines the **Event Bus architecture** used in AXON.

The Event Bus acts as the **central communication layer** of the system.

All components interact through events rather than direct calls.

This design allows AXON to scale from a single project environment to a multi‑project system without architectural changes.

---

# 2. Event‑Driven System

AXON is designed as an **event‑driven runtime**.

Core concept

State = Thread state
Change = Event

Example

ThreadStatus → state
ReviewRequested → event

All system actions are expressed as events.

---

# 3. Event Bus Role

The Event Bus serves several functions.

System communication

Agents, dispatchers, and the runtime communicate through events.

State propagation

Thread state changes are broadcast to interested components.

Agent coordination

Agents react to events and perform work.

Studio updates

The UI subscribes to events to display real‑time updates.

Scheduler triggers

Certain events trigger execution of agents.

# 6. Global Event Bus

AXON uses a **global event bus** shared by all projects.

Events are separated using a **project namespace**.

Example

project_a.ThreadCreated
project_b.ThreadCreated
project_c.MessagePosted

This approach simplifies architecture and allows the daemon to coordinate multiple projects.

---

# 7. Free vs Paid Model

The system architecture remains the same for both free and paid users.

The difference is the **number of concurrently active projects.**

Free users

Concurrent projects = 1

Paid users

Concurrent projects = multiple

The Event Bus always supports multiple projects internally.

Free mode simply enforces a runtime restriction.

---

# 8. Multi‑Project Execution

When a paid key is enabled the daemon may run multiple projects simultaneously.

Example

Event Bus

Project A events  
Project B events  
Project C events  

Agents subscribe only to the events relevant to their assigned project.

---

# 9. Agent Communication

Agents communicate indirectly using the Event Bus.

Example

Junior finishes implementation

MessagePosted

Senior receives event

ReviewRequested

Architect receives event

Architecture discussion begins

This model allows agents to collaborate without tight coupling.

---

# 10. Nogari Integration

Nogari threads also operate entirely through the Event Bus.

Example

NogariThreadCreated
AgentMessagePosted

Agents can freely participate in these discussions.

Because the event system is shared, discussions and work threads coexist naturally.

---

# 11. Scheduler Interaction

The dispatcher and scheduler subscribe to events.

Example

TaskAssigned

The scheduler activates the appropriate agent.

This ensures work execution remains asynchronous and scalable.

---

# 12. Studio Integration

AXON Studio subscribes to events for real‑time monitoring.

Example

MessagePosted
ThreadCreated
ReviewRequested

These events update the Thread Board and monitoring views.

---

# 13. Design Goal

The Event Bus acts as the **central nervous system** of AXON.

It connects

agents  
threads  
scheduler  
studio  
storage  

into a unified runtime.

Because all communication flows through events, the system remains flexible and scalable.
