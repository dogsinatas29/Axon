# AXON Specification

## v0.0.9 --- Scheduler, Dispatcher, and Organizational Control

------------------------------------------------------------------------

# 1. Purpose

This document defines the **execution control layer** of AXON.

The Scheduler and Dispatcher are responsible for executing work that has
already been defined during the **LLM parsing phase**.

Unlike traditional job schedulers, AXON **does not assign workers
dynamically**. Worker assignment is determined earlier during the
specification parsing stage and recorded in the project's architecture
document.

This design mirrors the way **real software organizations assign work**.

------------------------------------------------------------------------

# 2. Responsibility Separation

AXON separates responsibilities clearly.

LLM Parser\
→ task decomposition\
→ worker assignment

architecture document\
→ permanent record of the system design\
→ record of worker responsibilities

Scheduler\
→ decides when work should run

Dispatcher\
→ launches the appropriate agent

Agent Runtime\
→ performs the actual task

------------------------------------------------------------------------

# 3. architecture.md (Project Organization Record)

AXON maintains a project document that records system design and worker
roles.

Typical contents include:

-   architectural decisions
-   thread definitions
-   worker assignments
-   role hierarchy

Example:

Feature: WebSocket reconnect

Threads - reconnect handler - retry policy - connection monitor

Workers - Junior: reconnect handler - Senior: retry policy review -
Architect: architecture validation

This document functions similarly to a **company's technical planning
document combined with a team responsibility chart.**

------------------------------------------------------------------------

# 4. Organizational Changes During Development

AXON allows users to modify the agent organization at any time during a
project.

Possible operations include:

-   hiring a new junior agent
-   removing an agent
-   reassigning responsibilities
-   restructuring the development team

These changes are recorded by the **Architect agent** in the
architecture document.

Example:

Team update

Junior_B hired\
Senior_A reassigned to review role

The system immediately adapts to the updated structure.

------------------------------------------------------------------------

# 5. Scheduler Role

Because workers are already assigned, the Scheduler focuses on execution
timing.

Responsibilities:

-   maintain thread execution queue
-   enforce concurrency limits
-   prioritize threads
-   delay or resume work
-   manage execution order

Example flow:

ThreadCreated ↓ Scheduler queue ↓ Ready for execution ↓ TaskAssigned
event

------------------------------------------------------------------------

# 6. Dispatcher Role

The Dispatcher receives scheduling decisions and launches agents.

Responsibilities:

-   locate correct agent instance
-   start task execution
-   monitor task lifecycle
-   emit events when work completes

Example:

TaskAssigned ↓ Dispatcher ↓ Agent Runtime start

------------------------------------------------------------------------

# 7. Priority Handling

Threads may contain priority information.

Typical priorities:

critical normal low nogari

Nogari threads intentionally have lower priority so that discussion does
not block implementation work.

------------------------------------------------------------------------

# 8. Interaction with Event Bus

Scheduler and Dispatcher operate entirely through the Event Bus.

Example loop:

ThreadCreated ↓ Scheduler receives event ↓ Thread queued ↓ Dispatcher
executes agent ↓ MessagePosted ↓ Thread updated

This loop continues until the thread reaches a completed state.

------------------------------------------------------------------------

# 9. Human Control

Users maintain full control over the development organization.

At any time they may:

-   change agent roles
-   remove agents
-   introduce new agents
-   adjust team structure

These changes do not require complex system migration.

AXON simply updates the architecture document and continues operation.

------------------------------------------------------------------------

# 10. Multi‑LLM Orchestration Goal

A core design goal of AXON is enabling **multi‑LLM orchestration that
mirrors real organizational structures.**

Agents behave like members of a software company:

Architect\
Senior\
Junior

Each role participates in development and discussion threads.

This design allows AXON to simulate the workflow of a real engineering
team while coordinating multiple language models.

------------------------------------------------------------------------

# 11. System Flow

Spec ↓ LLM Parsing ↓ Architecture Document ↓ Thread Creation ↓ Event Bus
↓ Scheduler ↓ Dispatcher ↓ Agent Runtime ↓ Review ↓ Approval ↓
Architecture Update
