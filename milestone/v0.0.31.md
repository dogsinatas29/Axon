## 1. Phase 1 → Phase 2 Auto-Resume Race Condition 숙청

### 문제
Boss approve/reject/retry 핸들러에서 `pipeline_running.store(true)`를 `run_background()` 호출 전에 실행.
`run_background()` 내부에서 `running.load() == true` 체크 → 즉시 return → Phase 2가 실행되지 않음.

### 조치 파일: `crates/axon-daemon/src/server.rs`
- `approve_thread` (Phase 1→2 자동 재개, line 254-267)
- `approve_pipeline_review` (line 574-576)
- `reject_pipeline_review` (line 652)
- `retry_pipeline_review` (line 733)

### 수정 전/후
```
수정 전: Boss approve → pipeline_running.store(true) → run_background() → running.load()=true → return ❌
수정 후: Boss approve → run_background() → running.load()=false → running.store(true) → Phase 2 실행 ✅
```

## 6. WAL Flush Barrier (비동기 I/O 동기화 장벽)

### 문제
`update_thread_status`가 `save_thread`로 Thread 상태를 비동기 WAL 큐에 넣은 후 즉시 return.
WAL 큐가 플러시되기 전에 파이프라인이 Auto-resume되거나 데몬이 재시작되면,
과거 스냅샷 데이터를 읽어 Phase 2 완료 태스크를 Pending으로 오독하는 '초기화 대참사' 발생 가능.

### 조치 파일: `crates/axon-daemon/src/pipeline.rs`
- `update_thread_status` 함수 내 `save_thread` 직후 `storage.flush().await;` 장벽 주입
- "과거의 모든 디스크 쓰기 연산이 100% 완료되기 전에는 그 어떤 다음 페이즈 연산 노드도 진입하지 못한다"는 결정론적 선후 관계 보장

## 9. 주니어 에이전트 추가 영향 분석 (2026-05-26)

### 현재 구성
| 역할 | 수량 | 모델 |
|------|------|------|
| Architect | 1 | qwen2.5:7b-instruct-q4_K_M |
| Junior | 2 | qwen2.5:7b, deepseek-coder-v2:lite |
| Senior | 1 | deepseek-coder-v2:lite |

### 동시 처리 메커니즘
```rust
// pipeline.rs:586
let semaphore = Arc::new(Semaphore::new(juniors.len()));
// pipeline.rs:598
let junior = juniors[idx % juniors.len()].clone();
```
- Semaphore 제한: `juniors.len()` (현재 2)
- 작업 분배: Round-robin (`idx % juniors.len()`)

### 주니어 추가 시 영향

#### ✅ 긍정적 영향
1. **동시 처리량 증가**: 주니어 수 = 동시 행 가능 태스크 수
2. **Phase 완료 시간 단축**: 태스크 수 ÷ 주니어 수
3. **Draft 태스크 즉시 처리**: 대기 중인 태스크(`task_007`~`011`) 빠른 처리

#### ⚠️ 주의사항
1. **리소스 소비**: 주니어 1명당 약 2-4GB 메모리
2. **Diminishing Returns**: 태스크 수 < 주니어 수이면 이득 없음
3. **Senior 병목**: Junior가 빠르게 완료하면 Senior 검토 큐 쌓일 수 있음

### 권장 사항
- 현재 상태: 2명 Junior, 5개 Draft 태스크
- 권장: **3명으로 증가** (1명 추가)
- 이유: Draft 5개 처리에 2명으로는 3회차 필요, 3명이라면 2회차로 처리 가능

### 설정 방법
```bash
# 데몬 재시작 시
How many Juniors to hire? [default: 3]: 3
```
또는 `axon_config.json` 직접 수정.

            "Junior" | "junior" => {

                    return BAD_REQUEST; // 마지막 주니어 해고 불가
                }

## 20. AXON Intelligence Module Hub-and-Spoke 리팩토링 (2026-05-31)

### 문제
Synapse 의존성 그래프에서 `intelligence/` 모듈 및 하위 디렉토리(`corpus/`, `replay/`)에 속한 40여 개의 Rust 파일들이 별도의 계층적 추상화 장벽 없이 flat하게 나열되어 상호 참조선(Edges 1045개)을 가득 뿜어내며 의존성 스파게티 및 컴파일 병목을 야기하고 있었음.

### 조치 내역

#### 1단계: 공통 타입 인양 및 SSOT 확립
- `crates/axon-daemon/src/intelligence/common/types.rs` 및 `mod.rs`를 새로 생성하여, `CampaignMetrics`, `CatastropheKind`, `FailureClassification`, 그리고 두 가지 성격의 `CorpusFingerprint` 구조체를 단일 진실 공급원(SSOT)으로 인양하였습니다.
- 기존에 `corpus/`와 `replay/` 디렉토리 간에 교차 참조되던 양방향 의존성을 해결하기 위해, `replay/corpus_fingerprint.rs`에 정의된 구조체는 `CorpusFailureFingerprint`로 명칭을 변경하여 `common/types.rs`로 인양하였습니다.

#### 2단계: 플랫폼 시뮬레이터 추상화 (PlatformStrikeSim)
- `crates/axon-daemon/src/intelligence/replay/platform/mod.rs`를 신설하여 `PlatformStrikeSim` 트레이트 및 디스패칭을 정의합니다.
- `win32_topology_strike.rs`, `gtk2_gobject_hell.rs`, `gtk_topology_strike.rs`, `c_topology_strike.rs`를 `replay/platform/` 디렉토리로 이동하고 `PlatformStrikeSim`을 구현하도록 변환하여 구체적인 의존성 화살표들을 정리합니다.

#### 3단계: pub(crate) 보안 장벽 및 파사드 관문 구축
- `crates/axon-daemon/src/intelligence/mod.rs` 에 외부로 열려있던 `pub mod` 선언들을 전부 `pub(crate) mod`로 전환하여 캡슐화합니다.
- 외부 오케스트레이터 및 파이프라인에서 접근 가능한 통합 관문 구조체 `IntelligenceEngine`을 `intelligence/mod.rs`에 새로 설계하여 외부와의 의존성을 일원화합니다.
- `bin/` 및 `governance/` 하위의 바이너리들이 이용하는 최소한의 모듈(`causality`, `decision`, `semantic_debugger`, `jurisprudence`, `evolution`, `topology`, `corpus`, `replay`)만 외부로 열어두어 바이너리 컴파일 하위 호환성을 완벽하게 보장하였습니다.

#### 4단계: 테스트 및 검증
- `cargo check --all-targets`를 실행하여 `axon-daemon` 전체 크레이트 및 `bin/` 하위 바이너리들의 컴파일이 경고 외에는 오류 없이 완벽하게 성공함을 검증하였습니다.
- 단위 테스트(`proof_artifact.rs`에 존재하던 기존 LineageDelta 필드 생성 에러)의 빌드 오류를 추가로 수정하여 컴파일 무결성을 보장했습니다.
