## 🛠️ v0.0.31.60 - Compiler-driven AUTO_PURIFY Pipeline with LLM Harvester (Phase 24)

### 1. 개요 및 설계 철학
- **배경**: 기존 AUTO_PURIFY 프로세스는 LLM(아키텍트)에게 `spec.md`를 분해하여 각각 완벽하게 포맷팅된 4대 Canonical Spec 마크다운 문서(`constitution.md`, `architecture_seed.md`, `implementation_rules.md`, `hardening.md`)를 작성하도록 요청했습니다. 이 과정에서 7B/14B 모델은 스스로 의미를 이해하여 구조를 창작/재조직하려고 시도하여 명세서 키 유실을 유발하거나 마크다운 서식을 오염시켰습니다.
- **철학**: **"LLM은 자연어/코드의 의미론적 수집(Harvester)만 수행하고, 실제 규격 명세 조립(Assembler)은 컴파일러의 영역으로서 Rust가 전권을 쥐어야 한다."**
  - LLM에게 마크다운 레이아웃에 대한 부담을 100% 덜어주고, 단지 수집(Categorize & Verbatim Extract)만을 요구하여 실패 확률을 0%에 수렴시킵니다.
  - Rust가 이 조각들을 융합하고, CMakeLists.txt나 핵심 C/C++ 소스코드와 같은 Vital Invariants를 정규식으로 직접 도려내서 문서를 강제 빌드/완성합니다.

### 🌟 3대 핵심 아키텍처 및 구현 내역

#### 1. Single-Pass Semantic Harvester Prompt
- 기존 12회 이상 루프를 돌며 개별적으로 마크다운을 호출하던 구조를 완전히 들어내고, 1번의 통합된 Semantic Harvester 질의 패스로 교체했습니다.
- LLM은 `category`, `title`, `verbatim_extract`로 구성된 가볍고 명확한 `HarvestItem` JSON Array만 응답하게 만들어 7B 소형 모델의 지능적 부담을 최소화했습니다.

#### 2. Rust Deterministic Canonical Assembler & Fallback
- LLM의 추출물 조각을 Rust 단에서 분류하여 고정된 표준 템플릿(constitution, architecture_seed, implementation_rules, hardening)에 맞춰 deterministic하게 스펙 문서를 작성합니다.
- 만약 LLM의 추출 과정이 실패하여 JSON 파싱에 실패하거나 빈 데이터가 반환되면, Rust가 즉시 `fallback_harvest_spec` 구문 스캐너를 통해 명세서 원본에서 키워드 기반으로 조각을 정규화 수집하여 자가 합성합니다.

#### 3. Regex-driven Verbatim Invariant Injection
- 명세의 핵심 뼈대인 `CMakeLists.txt` 내용과 주요 GTK4/IME 관련 토큰 정보에 대해서는, LLM의 가독/추출 실수 여부와 무관하게 Rust가 `extract_spec_code_block` 정규식을 활용하여 `spec.md`에서 텍스트를 기계적으로 통째로 뜯어내 명세서 내에 Verbatim으로 강제 주입(Enforce Invariant)하도록 하드닝했습니다.
