## 🛠️ v0.0.31.70 - Non-Creative Semantic Extractor & Rust Semantic Validator (Phase 24 - Step 2)

### 1. 개요 및 설계 철학
- **배경**: 기존 AUTO_PURIFY 컴파일러 전면(Frontend) 파이프라인에서 LLM이 명세서의 마크다운 서식을 직접 생성하거나 스스로 더 좋은 구조라 판단하여 온톨로지를 자의적으로 발현(Ontology mismatch)시키는 등 생성형 자유도 문제가 지속되었습니다.
- **철학**: **"LLM을 비창작형 의미 스팬 파서(Semantic Extractor)로 격하하고, 컴파일러 구조적 결정성(Deterministic routing 및 Early-stage validator)은 Rust 단에서 완벽히 집행한다."**
  - LLM에게 마크다운 서식이나 레이아웃 작성을 전면 금지하고, 단지 원문 텍스트 조각을 타입(`constraint`, `architecture`, `rule`, `hardening`)과 함께 추출하는 기계식 `SemanticFragment` JSON Array 출력을 강제했습니다.
  - Rust 단에서 조기에 이 추출물의 완전성을 검사하는 `Semantic Validator`를 탑재하여 파이프라인을 조기 정화합니다.

---

### 🌟 3대 핵심 아키텍처 및 구현 내역

#### 1. Non-Creative Semantic Extractor & Typed IR
- LLM Harvester가 자의적으로 섹션을 만들어내거나 문장을 요약/재조직하지 못하도록 프롬프트를 전면 통제했습니다.
- 추출 결과물을 형상화한 `SemanticFragment` Typed IR 구조체를 정의하고, `tags`, `source_span`, `content` 만을 수집하게 함으로써 LLM의 창작 자유도를 AST 전위에서 제거했습니다.

#### 2. Early-Stage Semantic Validator (Rust)
- LLM Extractor의 출력이 Rust 단에 도달하면, 원본 `spec.md`에 포함되어 있던 8대 필수 기술 식별자(GtkApplicationWindow, lua_State, cairo_t 등)가 유실되지 않았는지 `validate_semantic_fragments` 유틸리티를 통해 조기에 1차 검증(Early sanitization)합니다.
- 검증 누락 발생 시, LLM에게 즉각 피드백을 전달하여 재생성/재추출을 유도합니다.

#### 3. Rust Canonical Compiler & Deterministic Routing
- 검증을 완수한 Semantic IR 조각들을 Rust가 기계적으로 해석하여 목적 버킷(`constraint` -> constitution, `architecture` -> architecture_seed, `rule` -> implementation_rules, `hardening` -> hardening)으로 **Deterministic Routing**하여 마크다운 사양 문서 파일들을 완벽한 규격 포맷으로 직접 조립/빌드합니다.
