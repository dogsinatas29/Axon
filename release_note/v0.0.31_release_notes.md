## 🏛️ Sovereign Protocol & Registry Specification
**"파일 시스템은 증거를 저장할 뿐, 연산의 진실은 데몬 런타임 메모리(SandboxRegistry)에 상주한다."**

### 1. 단일 진실 공급원 (SSOT)의 런타임 메모리 고정
- **기존 문제:** `server.rs` 및 `main.rs` 가 UI 승인 대기 상태를 검사할 때, 디스크 전체를 재귀적으로 뒤지는 DFS 폴링 스캔 방식을 취하여 속도가 느리고 상태 레이스 컨디션이 빈발하여 데드락이 발생함.
- **개선 사양:** 
  - `Daemon`에 `Arc<RwLock<HashMap<String, SandboxContext>>>` 형식의 `SandboxRegistry`를 도입.
  - 기동 시 지정된 `spec_path` 프로젝트를 즉각 샌드박스 레지스트리에 적재하여 유효한 상태(Active, Bootstrapping)로 부트스트랩 보장.
  - `/approval/pending` API 및 `/semantic/risks` API를 DFS에서 `sandbox_registry` 조회(O(1))로 변경하여 레이스 컨디션 완벽 차단. 상대 경로 혹은 단독 파일명(예: `spec.md`)으로 명세를 구동할 때의 프로젝트 ID 매핑 불일치 오작동 해결.

### 2. 결정론적 승인 상태 전이
- **SEAL (승인) 및 REJECT (반려) 처리 최적화:**
  - 보스의 의사결정 수신 시 (`submit_semantic_decision`), 레지스트리 내에 있는 대상 샌드박스의 `pending_approval` 플래그를 원천적으로 `false`로 갱신하여 메모리와 파일 시스템의 즉각적인 동기화 보장.

## 🛠️ 기술적 변경 사항
- **`crates/axon-daemon/src/lib.rs`**: `SandboxRegistry` 구조체 선언, `SandboxContext`, `SandboxState` 추가 및 초기화 로직 연동.
- **`crates/axon-daemon/src/main.rs`**: 기동 인자 `spec_path` 경로를 기반으로 즉시 레지스트리에 프로젝트 정보를 사전 적재(Pre-load)하도록 부트스트랩 고도화. 단독 파일명이나 상대 경로 인입 시 `current_dir` 과 결합해 절대경로로 온전하게 변환한 후 부모 폴더명을 프로젝트 ID로 매핑하여 중복 등록 차단.
- **`crates/axon-daemon/src/server.rs`**:
  - `get_pending_approval`: 글로벌 전역 단일 경로 의존성을 제거하고 레지스트리 Traversal을 통해 O(1) 멀티프로젝트 승인 대기 조회 구현.
  - `get_semantic_risks`: DFS 폴링을 걷어내고 `sandbox_registry`를 스캔하여 결정론적으로 프로젝트 리스크 수집.
  - `submit_semantic_decision`: 승인 통과 시 해당 프로젝트의 `pending_approval` 상태를 즉각 갱신하여 UI 및 데몬 간 타이밍 레이스 완벽 해결.

## ⚠️ 검증 및 안정성
- **Zero-Error Compilation:** `cargo check` 및 `cargo test` 가 에러나 경고 없이 통과함을 전방위 검증.
- **Determinism Verified:** 데몬 재기동 시에도 `TEST4_GTK` 샌드박스의 상태가 레지스트리에서 즉각 로딩되어 UI가 항상 최신 및 일치된 락 해제 상태를 유지함.
- **Semantic Validator Correction:** Core 컴포넌트가 `blocking_forbidden: true` 오폭으로 인해 시맨틱 위반 에러 및 `SKELETON_FAIL` 데드락을 일으키던 문제 해결.
  - `crates/axon-daemon/src/lib.rs` 내 검증 로직에 `ComponentStatus::Optional` 가드 보강.
  - `crates/axon-agent/src/lib.rs` 내 디폴트 주입 시 `Core` 상태에 맞춰 `blocking_forbidden: false` 로 자동 주입되도록 최적화.

## 🚀 v0.0.31.31 - Hardening Transactional Integrity & 2-Phase Commit
**"비동기 이벤트 루프와 비결정적 파일 I/O 간의 경계를 2단계 트랜잭션 커밋으로 원천 봉쇄한다."**

### 1. 2-Phase Commit 및 `Committing` 상태 머신
- **상태 기하학 확장:** `DaemonState`에 `Committing` 상태를 삽입하여 물리적 디스크 쓰기와 LSP 재해석 완료(Diagnostics Verification) 간의 불일치를 해결.
- **Strict Scheduling:** `can_schedule_next_task`에서 `Committing` 또는 `Validating` 상태일 때 다음 작업을 예약할 수 없도록 강제 차단하여 race condition 제거.

### 2. Single-use 승인 인가 토큰 (Generation & Token Validation)
- **Token Generation:** `.axon_approval_pending` 파일 생성 시 고유의 `uuid` 기반 단회성 권한 토큰을 함께 기록.
- **Strict Authorization:** UI에서 승인 요청(`SEAL`/`REWORK`)을 보낼 때 토큰 값을 함께 송신하고, 백엔드는 해당 파일의 토큰과 1:1 대조 후 다를 경우 트랜잭션을 거부.
- **Single-use Invalidation:** 검증 통과 후에는 파일 내의 토큰을 즉시 무효화(`null`) 처리하여 리플레이 공격 및 중복 승인 시도를 차단.
- **Generation Counter:** 커밋 시마다 `commit_generation`을 atomic하게 1 증가시켜 LSP 변경 버전 정밀 추적.

### 3. 원자적 파일 I/O 및 동기식 LSP 검증 (`semantic_gate` 동기화)
- **Atomic fsync Write:** `.axon_approval_pending` 및 `immutable_constraints.json`을 수정할 때 임시 파일(`.tmp`)을 생성한 후 `fsync` 처리를 통해 디스크 버퍼 비우기를 강제하고 `std::fs::rename`으로 덮어씀으로써 I/O 정합성 확보.
- **Synchronous Gate Verification:** 승인 직후 `LspSupervisor::semantic_gate`를 동기적으로(await) 대기하여 LSP 진단이 통과(`Verdict::Accept`)할 때만 최종 `Idle` 상태로 전이하고, 실패할 경우 `Failed` 상태로 즉시 롤백하여 UI와 모델의 세만틱 싱크가 틀어지는 것을 방지.
- **15초 Watchdog Timeout:** 전체 트랜잭션 블록에 `tokio::time::timeout` 15초 가드를 걸어 LSP 파싱 지연이나 데드락이 발생하더라도 샌드박스를 `Failed` 상태로 자동 복구하고 트랜잭션을 리젝트함.

### 4. UI 락다운 및 Debouncing (BossBoard.tsx)
- **UI Debounce:** 승인 프로세스 진행 중(`isSubmitting == true`)에는 `SEAL`, `REWORK`, `DISCARD` 버튼을 비활성화하고 로딩 상태를 명시하여 중복 API 요청 차단.

## 🚀 v0.0.31.32 - Isolated Lounge (Nogari) System & Historical Record Compilation
**"에이전트의 사회적 관계와 쉼터(Nogari) 로직을 메인 코딩 런타임으로부터 분리하여, 토큰 낭비 및 의도 오염이 없는 안전한 관제 환경을 수립한다."**

### 1. Nogari (Lounge) 시스템의 엄격한 격리 (Decoupling)
- **개념 정립:** 에이전트의 사회적 상호작용 및 감정(Vibe) 패킷이 메인 소프트웨어 공장 라인(`process_task` 및 2-phase commit pipeline)을 오염시키지 않도록 `NogariService` 및 잡담 레이어를 완전히 독립된 구조로 분리.
- **NogariPersona 모델 도입:** `NogariPersona` 구조체를 도입하여 각 에이전트 역할군(Architect 🏛️, Senior 👴, Junior 🐣)에 고유한 닉네임, 스타일, 커스텀 이모지 및 페르소나 정보를 주입할 수 있도록 아키텍처 토대를 구축.

### 2. 결정론적 Nogari.md 히스토리 컴파일러 구현
- **Atomic Compilation:** 각 에이전트가 쏟아낸 개별 잡담 로그(`lounge/*.log`) 파일들을 파싱 및 수집하여 단일한 프로젝트 역사서인 `Nogari.md` 파일로 변환하는 `compile_to_nogari_md` 컴포넌트 탑재.
- **Atomic Write & Fsync:** `Nogari.md` 컴파일 시에도 임시 파일(`.tmp`)을 생성하여 쓰기를 완료하고 `fsync`로 동기화한 뒤 원자적 `rename`을 수행하여 파일 깨짐 및 동시성 락 충돌 방지.
- **역할 기반 뷰어용 포스트 구조화:** 로그의 작성자 및 성격에 따라 🏛️, 👴, 🐣 이모지와 타임스탬프를 포함한 아름다운 마크다운 형식으로 역사를 기록하도록 시각적 사양 구축.

### 3. 승인 대기 상태 회귀 및 무한 루프 버그 해결 (Bypass Approval Gate)
- **문제 진단:** 데몬이 재시작되거나 리로드될 때 `SpecAnalysis` 단계의 루프에서 이미 승인이 완료된 `.axon_approval_pending` 파일이 있음에도 불구하고, `approved: false`로 덮어쓰고 다시 승인을 대기하여 공정이 굳어버리던 타이밍 및 상태 관리 회귀(Regression) 이슈 감지.
- **해결 방안:** `SpecAnalysis` 진입 시 이미 승인이 완료(`approved: true`)된 기존 파일이 존재할 경우, 신규 생성 및 WaitingApproval 상태 전이를 생략(Bypass)하고, 즉시 다음 `Skeleton` 단계로 통과하도록 로직 하드닝.

### 4. Robust Unit Tests
- `LoungeManager`의 `compile_to_nogari_md` 호출 및 `Nogari.md` 파일 생성/제거 기능을 검증하는 단위 테스트 `test_lounge_logging`에 통합 성공.

## 🚀 v0.0.31.33 - Instant Wake-up Governance & Resilient Semantic Parser
**"승인 처리 완료 후 데몬 파이프라인의 영구 슬립 지연을 해소하고, LLM의 비표준 단일 객체 JSON 응답에 대응하는 견고한 복원력을 구현한다."**

### 1. 지연 없는 승인 파이프라인 전이 (Instant Wake-up)
- **Notify-driven Select Loop:** `run_v3` SpecAnalysis 내의 승인 대기 폴링 루프 및 에러 중재(arbitration) 루프에 `tokio::select!` 매크로를 도입하여 `daemon.stage_notify.notified()` 이벤트를 결합. 2초의 대기 지연 없이 승인 즉시 즉각적인 파이프라인 재개를 보장.
- **Explicit Waiter Notification:** `submit_semantic_decision` axum HTTP 핸들러가 승인 검증 통과(Idle 전환), 실패(Failed 전환), 혹은 프로젝트 반려(Rejection) 처리를 마칠 때마다 `daemon.stage_notify.notify_waiters()`를 명시적으로 호출해 백그라운드 스케줄러 스레드를 즉각 Waking-up 하도록 하드닝.

### 2. 비표준 단일 JSON Object 응답 복원력 강화 (Resilient Parser)
- **JSON Object to Array Fallback:** LLM이 표준 JSON Array 대신 비표준 단일 JSON Object 형식으로 Semantic IR을 반환할 때의 파서 충돌 방지를 위해, JSON 파싱 실패 전 `serde_json::Value`를 추출하여 object 타입일 경우 단일 원소의 Vec으로 감싸 복원 파싱하도록 유연하게 개선.
- **Auto Purifier 안정성:** 저성능 모델이 유발하는 단일 Object 응답 하에서도 fallback parser로 이행하지 않고 본래의 고품질 LLM Semantic IR을 보존하도록 설계 신뢰성 극대화.

### 3. 테스트 하드닝 (Admin Intervention Test)
- `admin::tests::test_admin_intervention`의 SQLite 비동기 큐 타이밍 이슈 해결을 위해, 개별 테스트 케이스마다 독립적인 UUID 기반 `thread_id`를 사용하도록 보강하고 비동기 I/O 대기 시간을 500ms로 연장하여 병렬 환경 테스트 통과 보장.
