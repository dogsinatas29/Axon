# Release Notes: v0.0.31

## 🚀 주요 변경 사항 (Core Updates)

1. **[IR_PARSE_FAIL] 강제 인터락 완공 (Pipeline Interlock)**
   - `pipeline.rs` 내 `run_background`, `run_pipeline`, `execute_one_task` 세 구간에 `ProjectIR::from_md` 실패 시 즉시 파이프라인을 일시 정지(SUSPEND)하고 `.axon_approval_pending` 파일로 Boss 승인을 대기하는 이중/삼중 잠금 웰딩을 적용했습니다.

2. **FFI 번역 가드레일 (Translation Guardrail) 차단망 구축**
   - `axon-agent`의 시스템 프롬프트(Lua)에 `FFI TRANSLATION GUARD (MANDATORY)` 조항을 추가하여, Lua 에이전트가 `std::vector`, `std::string` 등의 C++ 정적 타입을 사용하는 환각을 원천 차단하고 네이티브 Lua 타입(table, string 등)으로 매핑하도록 강제했습니다.

3. **CMake Pruning 자동 복구 로직 (Atomic Restore)**
   - `execute_one_task` 종료 후 샌드박스의 파일이 실제 작업 디렉토리(real_path)로 승격(Promoted)될 때, 즉시 `architecture.md`를 재해석하여 `DepGraph`를 갱신하고 `CMakeLists.txt`를 다시 생성함으로써 임시 주석([PRUNED])된 타겟 목록을 자동 복구합니다.

4. **AXON Intelligence Module Hub-and-Spoke 리팩토링 (Facade & pub(crate))**
   - `intelligence/` 모듈 및 하위 디렉토리(`corpus/`, `replay/`) 내 40여 개의 Rust 파일들이 별도의 계층적 추상화 장벽 없이 flat하게 배치되어 발생하던 1045개의 의존성 스파게티 선들을 정리하기 위해, `mod.rs`를 API Gate로 삼는 파사드(Facade) 구조체 `IntelligenceEngine`을 도입하고 외부 직접 접근을 `pub(crate)`로 격리하였습니다.
   - 단, 디버그/프로파일링용 외부 바이너리(`bin/` 하위 파일들)의 컴파일 하위 호환성을 유지하기 위해 외부에서 직접 사용하는 필수 모듈들에 대해서만 `pub` 가시성을 허용하였습니다.

5. **플랫폼 시뮬레이터 추상화 및 격리 (`PlatformStrikeSim` Trait)**
   - `win32_topology_strike.rs`, `gtk2_gobject_hell.rs`, `gtk_topology_strike.rs`, `c_topology_strike.rs` 등 플랫폼별로 flat하게 노출되어 의존성을 높이던 시뮬레이터 4종을 `replay/platform/` 하위 모듈로 완전히 이동 및 격리하였습니다.
   - `PlatformStrikeSim` 추상화 트레이트를 설계하고 각 구조체에 이를 구현함으로써, 상위 엔진이 트레이트 객체(`Box<dyn PlatformStrikeSim>`)를 통해서만 검증을 수행하도록 구조를 정돈했습니다.

6. **공통 타입 수직 인양을 통한 SSOT 확립**
   - `CampaignMetrics`, `CatastropheKind`, `FailureClassification`, `CorpusFingerprint` 등 `corpus`와 `replay` 간에 순환 참조를 유발하던 기저 데이터 구조체들을 `intelligence/common/types.rs`로 인양(Lift)하여 단방향 의존성을 확립하였습니다.
   - 양방향으로 교차 참조되며 타입 충돌을 일으키던 두 `CorpusFingerprint` 중 하나는 `CorpusFailureFingerprint`로 명칭을 변경하여 모호함을 해소하였습니다.

## 🚀 추가 변경 사항 (2026-05-31) - Boss Board 뱃지 정합성 및 워크 보드 수동 인터락 승인 무반응 패치

1. **보스 보드 뱃지 카운트 배제 (Frontend)**
   - `bossPendingCount` 계산 로직에서 `threadsData.filter(t => t.status === 'BossApproval')` 개수 합산을 제거했습니다.
   - 수동 인터락 대기 건(BossApproval 스레드)은 워크 보드 상세 모달에서 처리하므로 보스 보드 알림 뱃지에서 나타나지 않게 되었습니다.

2. **수동 인터락 대기 파일명 고유화 (Backend)**
   - 기존에 고정 파일명 `.axon_approval_pending`을 공유하던 방식을 `.axon_approval_pending_{task_id}`로 변경하여 병렬 태스크 실행 시 파일이 덮어써지던 Race condition 결함을 완전 제거했습니다.

3. **워크 보드 승인 API 핸들러 수동 인터락 폴백 추가 (Backend)**
   - `approve_thread`, `reject_thread`, `retry_thread` API 핸들러가 호출되었을 때, 해당 태스크가 인메모리 `pending_reviews`에 존재하지 않는 경우 고유 승인 대기 파일(`.axon_approval_pending_{thread_id}`)을 직접 갱신하도록 폴백을 작성했습니다.
   - 수동 인터락 대기 상태에서 승인을 진행하면 백그라운드 폴링 루프가 정상적으로 해제되며, 승격(Promotion, 파일 이동) 및 다음 공정(Phase 2)으로의 이행이 안전하게 재개됩니다.
